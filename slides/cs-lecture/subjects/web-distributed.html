웹 / 분산 시스템

수백만 명이 동시에 써도 버티는 서비스는 여러 대의 컴퓨터로 일을 나눠서 만듭니다. 웹의 기본 동작부터 대규모 시스템의 설계 원리까지 살펴봅니다.

웹 동작 원리 (클라이언트-서버)

웹은 요청을 보내는 쪽(클라이언트)과 응답을 돌려주는 쪽(서버)의 대화로 이루어집니다. 브라우저에 주소를 입력하면, 브라우저(클라이언트)가 서버에 "이 페이지 주세요"라고 HTTP 요청을 보내고, 서버는 HTML 등을 응답으로 돌려줍니다.

요청 → 처리 → 응답. 웹의 모든 상호작용은 이 한 사이클의 반복이다

HTTP 응답에는 상태 코드가 함께 옵니다. 200은 성공, 404는 못 찾음, 500은 서버 오류 같은 식으로, 한눈에 결과를 알 수 있습니다.

http_message.txt
GET /page HTTP/1.1          # ← 클라이언트가 보내는 요청
Host: example.com

HTTP/1.1 200 OK            # ← 서버가 돌려주는 응답 (200 = 성공)
Content-Type: text/html

<html>...페이지 내용...</html>

REST API

API는 프로그램끼리 약속된 방식으로 데이터를 주고받는 창구입니다. REST는 그 창구를 설계하는 가장 흔한 규칙으로, "자원(데이터)을 URL로 가리키고, HTTP 메서드로 행동을 표현"합니다.

메서드의미예시
GET조회GET /users/42
POST생성POST /users
PUT수정PUT /users/42
DELETE삭제DELETE /users/42

주고받는 데이터 형식으로는 사람이 읽기 쉬운 JSON이 널리 쓰입니다.

rest_example.py
# 간단한 REST 서버 (Flask)
from flask import Flask, jsonify, request
app = Flask(__name__)

@app.get("/users/")        # 자원: 사용자
def get_user(uid):
    return jsonify({"id": uid, "name": "Kim"})   # JSON 응답

@app.post("/users")                 # 새 사용자 생성
def create_user():
    data = request.json
    return jsonify({"created": data}), 201       # 201 = Created

데이터베이스 확장 (복제 · 샤딩)

사용자가 늘면 DB 한 대로는 버겁습니다. 두 가지 확장 방법이 있습니다.

  • 복제(Replication) — 같은 데이터를 여러 대에 복사해 둡니다. 보통 쓰기는 한 대(주, primary), 읽기는 여러 대(부, replica)가 나눠 처리해 읽기 부하를 분산합니다.
  • 샤딩(Sharding) — 데이터를 쪼개서 여러 대에 나눠 담습니다. 예: 사용자 ID 기준으로 1~100만은 1번 DB, 100만~200만은 2번 DB. 한 대가 담을 양 자체를 줄입니다.
라우터가 키를 보고 어느 샤드로 보낼지 결정 — 데이터를 나눠 담는다
💡 복제 vs 샤딩

읽기가 많아 부담이면 복제, 데이터 양 자체가 너무 커지면 샤딩을 떠올리세요. 큰 서비스는 둘을 함께 씁니다.

캐싱과 로드밸런싱

캐싱(caching)은 자주 쓰는 결과를 빠른 저장소(예: 메모리, Redis)에 미리 담아 두어, 매번 DB까지 가지 않고 바로 돌려주는 기법입니다. DB 부담을 크게 줄이고 응답을 빠르게 합니다.

로드밸런서(load balancer)는 들어오는 요청을 여러 서버에 고르게 분배하는 교통경찰입니다. 한 서버에 몰리는 것을 막고, 한 대가 죽어도 나머지가 받아 주어 가용성을 높입니다.

로드밸런서가 요청을 여러 서버로 분산 — 부하 분산 + 장애 대비
cache_demo.py
import redis
cache = redis.Redis()

def get_user(uid):
    key = f"user:{uid}"
    cached = cache.get(key)
    if cached:
        return cached            # 캐시 적중: DB 안 거치고 바로 반환

    user = db.query(uid)         # 캐시 미스: DB 조회
    cache.set(key, user, ex=60)  # 60초간 캐시에 저장
    return user

마이크로서비스

하나의 거대한 프로그램(모놀리식)에 모든 기능을 담는 대신, 기능별로 작고 독립적인 서비스로 쪼개는 방식이 마이크로서비스입니다. 예: 결제 서비스, 주문 서비스, 알림 서비스가 각각 따로 돌고 API로 협력합니다.

구분모놀리식마이크로서비스
구조한 덩어리여러 작은 서비스
배포전체를 한 번에서비스별 개별 배포
장점단순함독립 확장·장애 격리
단점커지면 무거움운영·통신 복잡도 증가
⚠️ 만능이 아니다

마이크로서비스는 서비스 간 통신·배포·모니터링이 복잡해집니다. 작은 프로젝트라면 모놀리식이 더 나은 경우가 많습니다.

CAP 정리와 일관성

여러 대로 나뉜 분산 데이터베이스에는 유명한 원칙이 있습니다. CAP 정리는 다음 셋 중 동시에 셋 다는 보장할 수 없고, 둘만 택할 수 있다고 말합니다.

  • 일관성(Consistency) — 어느 서버에 물어도 같은 최신 값을 본다.
  • 가용성(Availability) — 일부가 죽어도 요청에 응답한다.
  • 분단 내성(Partition tolerance) — 서버 간 통신이 끊겨도 동작한다.
네트워크 분단(P)은 현실에서 피할 수 없어, 보통 C와 A 사이에서 선택한다

현실에서는 통신 단절(P)을 가정할 수밖에 없으므로, 보통 C(일관성)와 A(가용성) 사이에서 무엇을 우선할지 고릅니다. 은행 잔고는 일관성을, SNS 좋아요 수는 가용성을 우선하는 식입니다. 그래서 "잠깐은 값이 달라도 결국 같아진다"는 최종 일관성(eventual consistency) 개념도 많이 쓰입니다.

메시지 큐와 비동기 통신

지금까지의 통신은 "요청을 보내고 응답을 기다리는" 동기(synchronous) 방식이었습니다. 하지만 주문 후 영수증 이메일 보내기처럼 당장 끝나지 않아도 되는 일은, 따로 떼어 나중에 처리하는 게 낫습니다. 이때 메시지 큐(message queue)가 등장합니다.

메시지 큐는 생산자(producer)가 할 일을 큐에 넣어 두면, 소비자(consumer)가 자기 속도에 맞춰 하나씩 꺼내 처리하는 중간 우편함입니다. 보내는 쪽은 큐에 넣자마자 다음 일을 할 수 있어(비동기) 응답이 빨라지고, 받는 쪽이 잠시 죽어도 메시지는 큐에 남아 있어 안전합니다.

생산자는 큐에 넣고 바로 응답 → 소비자가 자기 속도로 처리(FIFO)

대표적인 메시지 큐로는 RabbitMQ, Apache Kafka, AWS SQS 등이 있습니다. 갑자기 요청이 몰려도 큐가 충격을 흡수해 주는 완충 장치(buffer) 역할도 합니다.

message_queue.py
# 생산자: 무거운 일을 직접 하지 않고 큐에 맡긴다
def place_order(order):
    save_to_db(order)                 # 빠른 일만 즉시 처리
    queue.publish("send_email", order)  # 느린 일은 큐에 넣고
    return {"status": "주문 완료"}      # 곧바로 응답 (안 기다림)

# 소비자: 큐에서 하나씩 꺼내 천천히 처리
def email_worker():
    while True:
        msg = queue.consume("send_email")  # 메시지 대기
        send_email(msg.order)              # 실제 전송 (느려도 OK)
        msg.ack()                          # 처리 완료 표시
💡 언제 쓰나

이메일·알림 발송, 영상 인코딩, 대량 데이터 처리처럼 즉시 답할 필요 없는 무거운 작업을 떼어낼 때 씁니다. 사용자는 빠른 응답을 받고, 무거운 일은 뒤에서 차근차근 처리됩니다.

이벤트 기반 아키텍처

메시지 큐를 한 걸음 더 밀고 나가면 이벤트 기반 아키텍처(event-driven architecture)가 됩니다. 서비스끼리 직접 "이거 해 줘"라고 명령하는 대신, "이런 일이 일어났다"는 이벤트만 알리고, 관심 있는 서비스들이 알아서 반응합니다.

예를 들어 "주문 완료" 이벤트 하나가 발생하면, 결제 서비스·재고 서비스·알림 서비스가 각자 독립적으로 그 이벤트를 받아 처리합니다. 주문 서비스는 누가 듣고 있는지 신경 쓸 필요가 없습니다 — 이를 느슨한 결합(loose coupling)이라 합니다.

한 이벤트를 여러 구독자(subscriber)가 각자 받아 처리 — 발행/구독(pub/sub)
구분명령 기반 (직접 호출)이벤트 기반
결합도강함 (누구를 부를지 알아야)느슨함 (알릴 뿐)
확장새 기능마다 코드 수정구독자만 추가하면 끝
흐름 추적쉬움 (직선적)어려움 (흩어짐)
⚠️ 디버깅이 어렵다

이벤트가 여기저기로 퍼지므로 "무슨 일이 어떤 순서로 일어났는지" 추적이 까다롭습니다. 그래서 뒤에서 배울 관측성(observability)이 특히 중요해집니다.

일관성 모델 상세 (강한 · 최종 일관성)

CAP 정리에서 잠깐 본 일관성을 더 깊이 봅시다. 데이터가 여러 대에 복제돼 있을 때, "방금 쓴 값을 다른 서버에서 언제 볼 수 있는가"에 따라 일관성 모델이 갈립니다.

  • 강한 일관성(strong consistency) — 쓰기가 끝나면 모든 서버가 즉시 같은 값을 보여 줍니다. 정확하지만, 모든 복제본이 동기화될 때까지 기다려야 해 느리고 가용성이 떨어집니다.
  • 최종 일관성(eventual consistency) — 잠깐은 서버마다 값이 다를 수 있지만, 시간이 지나면 결국 같아집니다. 빠르고 잘 죽지 않지만, 방금 쓴 값이 곧바로 안 보일 수 있습니다.
최종 일관성: 처음엔 일부만 v2, 시간이 지나면 모두 v2로 수렴
모델속도정확성어울리는 곳
강한 일관성느림항상 최신은행 잔고·재고 차감
최종 일관성빠름잠깐 옛값 가능SNS 좋아요·조회수
💡 일상 속 최종 일관성

유튜브 조회수가 잠시 멈춰 있다가 갑자기 확 오르는 것을 본 적 있나요? 여러 서버의 카운트를 나중에 모아 맞추는 최종 일관성의 흔적입니다. 정확한 실시간보다 빠른 응답을 택한 것이죠.

합의 알고리즘 (Raft · 리더 선출)

여러 서버가 데이터를 복제할 때, "누구 말이 진짜인가"를 정해야 합니다. 모두가 동등하게 쓰기를 받으면 값이 충돌하니까요. 그래서 한 대를 리더(leader)로 뽑고, 나머지는 팔로워(follower)로서 리더를 따라갑니다. 이렇게 여러 대가 하나의 값에 동의하는 절차를 합의(consensus)라 하고, 대표적인 알고리즘이 Raft입니다.

리더 선출(leader election)의 핵심 아이디어는 간단합니다. 팔로워는 리더로부터 주기적으로 "나 살아 있어"라는 신호(하트비트)를 받습니다. 일정 시간 신호가 없으면 "리더가 죽었나 보다" 하고 후보(candidate)가 되어 다른 서버들에게 표를 구합니다. 과반수(majority)의 표를 얻으면 새 리더가 됩니다.

리더가 죽으면 → 후보가 등장 → 과반수 득표 → 새 리더 (F=팔로워, C=후보)

과반수를 요구하는 이유는, 두 명이 동시에 리더가 되는 분열(split-brain)을 막기 위해서입니다. 표는 한 표만 줄 수 있으므로 과반수를 동시에 둘이 받을 수는 없습니다. 그래서 보통 서버 수를 홀수(3·5대)로 둡니다.

📌 어디서 쓰이나

분산 시스템의 핵심 부품인 etcd(쿠버네티스의 두뇌), Consul, ZooKeeper 등이 이런 합의 알고리즘으로 "단 하나의 진실"을 유지합니다.

분산 트랜잭션과 사가(Saga) 패턴

한 DB 안에서는 "전부 성공하거나 전부 취소"되는 트랜잭션이 쉽습니다. 하지만 주문·결제·배송이 각각 다른 서비스의 다른 DB에 있다면? 하나의 거대한 트랜잭션으로 묶기 어렵습니다. 이때 쓰는 패턴이 사가(Saga)입니다.

사가는 큰 작업을 여러 개의 작은 로컬 트랜잭션으로 쪼개 순서대로 실행합니다. 중간에 하나라도 실패하면, 이미 끝난 단계들을 거꾸로 되돌리는 보상 작업(compensating transaction)을 실행합니다. "취소는 환불로, 재고 차감은 재고 복구로" 되돌리는 식입니다.

③에서 실패하면 ②·①을 역순으로 보상(취소)해 일관성을 회복한다
saga.py
steps = [
    (create_order,  cancel_order),    # (실행, 보상)
    (charge_payment, refund_payment),
    (ship_order,    cancel_shipping),
]

done = []
try:
    for do, undo in steps:
        do()                 # 단계 실행
        done.append(undo)    # 보상 작업을 기억해 둠
except Exception:
    for undo in reversed(done):   # 실패 시 역순으로
        undo()                    # 이미 한 일을 되돌린다
⚠️ 완벽한 원자성은 아니다

사가는 보상으로 "결과적 일관성"을 맞출 뿐, 중간 단계가 잠깐 외부에 보일 수 있습니다. 그래서 보상 작업은 여러 번 실행해도 같은 결과가 되도록(멱등성, idempotent) 설계해야 안전합니다.

장애 처리와 회복탄력성 (서킷 브레이커 · 재시도)

분산 시스템에서 일부 서비스가 죽는 것은 예외가 아니라 일상입니다. 한 서비스의 장애가 도미노처럼 전체로 번지지 않게 막는 능력을 회복탄력성(resilience)이라 합니다. 대표 기법 둘을 봅시다.

  • 재시도(retry) — 일시적 오류(네트워크 깜빡임 등)는 잠시 후 다시 시도하면 성공할 때가 많습니다. 단, 간격을 점점 늘리는 지수 백오프(exponential backoff)를 써서 죽어가는 서버를 더 때리지 않도록 합니다.
  • 서킷 브레이커(circuit breaker) — 두꺼비집(누전 차단기)처럼, 실패가 일정 수준을 넘으면 회로를 끊어 더 이상 요청을 보내지 않고 즉시 실패 처리합니다. 죽은 서비스에 헛된 요청을 퍼붓는 것을 막아, 시스템 전체를 보호합니다.

서킷 브레이커는 세 가지 상태를 오갑니다. 닫힘(Closed)은 정상 통과, 열림(Open)은 차단(즉시 실패), 일정 시간 뒤 반열림(Half-Open)으로 살짝 풀어 시험 요청을 보내 봅니다. 성공하면 다시 닫고, 실패하면 다시 엽니다.

닫힘 →(실패 누적) 열림 →(시간 경과) 반열림 →(성공) 닫힘 / (실패) 열림
resilience.py
import time

# 지수 백오프 재시도: 0.5s → 1s → 2s ...
def call_with_retry(fn, attempts=4):
    for i in range(attempts):
        try:
            return fn()
        except TransientError:
            if i == attempts - 1:
                raise
            time.sleep(0.5 * (2 ** i))   # 점점 길게 대기

# 서킷 브레이커: 실패가 많으면 회로를 연다
if breaker.is_open():
    return fallback()        # 죽은 서비스에 안 보내고 즉시 대안
result = breaker.call(payment_service)   # 성공/실패를 집계
💡 함께 쓰는 친구들

재시도·서킷 브레이커 외에도 타임아웃(무한정 안 기다리기), 폴백(대체 응답), 벌크헤드(자원을 칸막이로 격리)를 함께 씁니다. 모두 "장애를 가두는" 같은 목표를 가집니다.

모니터링과 관측성 (로그 · 메트릭 · 추적)

서비스가 수십 개로 쪼개지면, "왜 느리지? 어디서 막혔지?"를 눈으로 보기 어렵습니다. 시스템 내부 상태를 밖에서 들여다볼 수 있게 하는 능력을 관측성(observability)이라 하며, 세 가지 기둥으로 이루어집니다.

기둥무엇인가답하는 질문
로그(Logs)시간순 사건 기록"무슨 일이 있었나?"
메트릭(Metrics)수치 지표(CPU·응답시간·요청수)"얼마나 건강한가?"
추적(Traces)요청이 거친 서비스 경로"어디서 느려졌나?"

특히 분산 추적(distributed tracing)은 하나의 요청에 추적 ID를 붙여, 그 요청이 게이트웨이 → 주문 → 결제 → DB를 거치며 각 단계에 얼마나 걸렸는지 한 줄로 이어 보여 줍니다. 병목을 찾는 데 결정적입니다.

한 요청의 구간별 소요 시간 — 막대가 긴 DB 조회가 병목임을 한눈에
📌 대표 도구

메트릭/대시보드는 Prometheus + Grafana, 로그는 ELK(Elasticsearch·Logstash·Kibana), 추적은 Jaeger·OpenTelemetry가 널리 쓰입니다. "모니터링"이 정해진 지표를 보는 것이라면, "관측성"은 예상 못한 문제까지 캐물을 수 있는 더 넓은 개념입니다.

컨테이너 오케스트레이션과 API 게이트웨이

마이크로서비스를 실제로 굴리려면 두 가지 도구가 거의 필수입니다. 서비스 묶음을 실행·관리하는 오케스트레이션과, 외부 요청의 단일 입구 역할을 하는 API 게이트웨이입니다.

컨테이너(container)는 앱과 그 실행 환경을 통째로 담은 가벼운 상자입니다(예: Docker). 서비스가 수십·수백 개로 늘면 이 컨테이너들을 일일이 손으로 관리할 수 없습니다. 쿠버네티스(Kubernetes) 같은 오케스트레이션 도구가 자동으로 배치하고, 죽으면 다시 띄우고, 부하에 맞춰 수를 늘리거나 줄입니다(오토스케일링).

쿠버네티스가 자동으로설명
자가 치유(self-healing)죽은 컨테이너를 감지해 다시 띄움
오토스케일링트래픽에 따라 컨테이너 수를 조절
롤아웃/롤백새 버전을 점진 배포, 문제 시 되돌림
서비스 디스커버리서비스끼리 서로를 자동으로 찾게 함

한편 API 게이트웨이는 모든 외부 요청이 가장 먼저 도착하는 정문입니다. 클라이언트는 내부에 서비스가 몇 개인지 몰라도, 게이트웨이 한 곳에만 요청하면 됩니다. 게이트웨이는 알맞은 서비스로 라우팅하고, 그 길목에서 인증·요청 제한(rate limiting)·로깅 같은 공통 작업을 한 번에 처리합니다.

클라이언트는 게이트웨이 한 곳만 알면 됨 — 정문에서 인증·라우팅을 일괄 처리
💡 로드밸런서와 뭐가 다른가

로드밸런서가 "같은 서비스의 여러 복제본"에 부하를 고르게 나누는 데 집중한다면, API 게이트웨이는 "어떤 서비스로 보낼지" 판단하고 인증·변환·제한 같은 부가 기능까지 맡는 더 똑똑한 정문입니다. 둘은 함께 쓰입니다.

🎯 핵심 정리
  • 웹은 클라이언트 요청과 서버 응답의 반복으로 동작한다.
  • REST는 자원을 URL로, 행동을 HTTP 메서드로 표현하는 API 설계 규칙이다.
  • 읽기 부하는 복제로, 데이터 양은 샤딩으로 분산한다.
  • 캐싱은 DB 부담을 줄이고, 로드밸런서는 요청을 여러 서버로 나눈다.
  • 분산 시스템은 CAP 정리에 따라 일관성과 가용성 사이에서 선택한다.
  • 메시지 큐는 무거운 일을 비동기로 떼어내 응답을 빠르게 하고 충격을 흡수한다.
  • 이벤트 기반 아키텍처는 이벤트만 알려 서비스 간 결합을 느슨하게 한다.
  • 강한 일관성은 항상 최신값, 최종 일관성은 시간이 지나면 같아진다.
  • 합의 알고리즘(Raft)은 리더를 뽑아 과반수 동의로 하나의 진실을 유지한다.
  • 사가 패턴은 분산 작업을 쪼개고, 실패하면 보상 작업으로 되돌린다.
  • 서킷 브레이커와 재시도는 장애가 전체로 번지지 않게 가둔다.
  • 관측성은 로그·메트릭·추적으로 시스템 내부를 밖에서 들여다본다.
  • 쿠버네티스는 컨테이너를 자동 배치·치유·확장하고, API 게이트웨이는 단일 입구가 된다.
클라이언트 / 서버
요청하는 쪽과 응답하는 쪽. 웹 통신의 두 주체.
REST API
자원을 URL로, 행동을 HTTP 메서드로 표현하는 API 설계 방식.
샤딩(Sharding)
데이터를 쪼개 여러 DB에 나눠 담아 한 대의 부담을 줄이는 기법.
로드밸런서
들어오는 요청을 여러 서버에 고르게 분배하는 장치.
CAP 정리
분산 시스템에서 일관성·가용성·분단내성 중 둘만 보장 가능하다는 원칙.
메시지 큐
생산자가 넣은 일을 소비자가 자기 속도로 꺼내 처리하는 비동기 우편함.
이벤트 기반 아키텍처
명령 대신 "일어난 일(이벤트)"만 알려 서비스를 느슨하게 결합하는 구조.
최종 일관성
잠깐은 값이 다를 수 있으나 시간이 지나면 모든 복제본이 같아지는 모델.
합의 알고리즘 / Raft
여러 서버가 리더를 뽑아 과반수 동의로 하나의 값에 합의하는 방법.
사가(Saga) 패턴
분산 트랜잭션을 작은 단계로 쪼개고, 실패 시 보상 작업으로 되돌리는 패턴.
서킷 브레이커
실패가 누적되면 회로를 끊어 죽은 서비스로의 요청을 막는 장애 차단기.
관측성(Observability)
로그·메트릭·추적으로 시스템 내부 상태를 밖에서 파악하는 능력.
쿠버네티스
컨테이너를 자동으로 배치·치유·확장·롤아웃하는 오케스트레이션 도구.
API 게이트웨이
외부 요청의 단일 입구로, 라우팅·인증·요청 제한을 일괄 처리하는 정문.
