소프트웨어 공학

코드를 잘 짜는 것을 넘어, 여러 사람이 함께 오래 유지되는 소프트웨어를 체계적으로 만드는 방법을 배웁니다.

소프트웨어 공학이란?

소프트웨어 공학(software engineering)은 소프트웨어를 체계적이고 규율 있게 만들고 유지하는 학문입니다. 혼자 짜는 작은 스크립트가 아니라, 여러 명이 수년간 함께 키워 가는 큰 시스템을 다룹니다.

집 짓기에 비유하면, 코딩은 벽돌을 쌓는 일이고 소프트웨어 공학은 설계도를 그리고 공정을 관리하며 품질을 검사하는 일 전체입니다. 벽돌만 잘 쌓는다고 좋은 집이 되지 않듯, 코드만 잘 짠다고 좋은 소프트웨어가 되는 건 아닙니다.

📌 이 페이지에서 배우는 것

개발 생명주기 · 요구사항 분석 · 설계와 디자인 패턴 · 버전 관리(Git) · 테스트 · 유지보수까지, 소프트웨어가 태어나 운영되는 전 과정을 다룹니다.

개발 생명주기 (SDLC)

소프트웨어는 한 번에 완성되지 않고 요구사항 → 설계 → 구현 → 테스트 → 배포 → 유지보수의 단계를 거칩니다. 이 전 과정을 SDLC(Software Development Life Cycle, 개발 생명주기)라 합니다.

SDLC: 단계를 거쳐 만들고, 유지보수로 다시 순환한다

이 단계를 진행하는 방식에는 크게 두 가지 흐름이 있습니다.

  • 폭포수(Waterfall) — 단계를 한 방향으로 차례차례. 계획이 명확할 때 좋지만, 중간에 요구가 바뀌면 대응이 어렵습니다.
  • 애자일(Agile) — 작은 단위(스프린트)로 만들고-피드백-수정을 짧게 반복. 변화에 유연하며 오늘날 가장 널리 쓰입니다.
💡 폭포수 vs 애자일

폭포수는 "지도 보고 한 번에 가기", 애자일은 "한 블록씩 가며 길 확인하기"에 가깝습니다. 요구가 자주 바뀌는 현대 제품 개발에서는 애자일이 잘 맞습니다.

요구사항 분석 (Requirements)

모든 프로젝트의 출발점입니다. "무엇을 만들 것인가"를 명확히 정의하는 단계로, 여기서 어긋나면 아무리 코드를 잘 짜도 잘못된 제품이 됩니다. 요구사항은 보통 두 가지로 나눕니다.

  • 기능 요구사항 — 시스템이 무엇을 해야 하는가. 예: "사용자는 로그인할 수 있어야 한다."
  • 비기능 요구사항 — 시스템이 얼마나 잘해야 하는가. 예: "로그인은 2초 안에 끝나야 한다." (성능·보안·사용성 등)
user_story.md
# 사용자 스토리 (User Story) 형식

As a [사용자 유형],
I want [원하는 기능]
so that [그로 인한 이점].

## 예시
As a 쇼핑몰 고객,
I want 장바구니에 상품을 담을 수 있다
so that 한 번에 여러 상품을 결제할 수 있다.

## 인수 조건 (Acceptance Criteria)
- [ ] 상품 상세 페이지에 "담기" 버튼이 있다
- [ ] 담으면 장바구니 개수가 즉시 늘어난다
- [ ] 로그아웃해도 장바구니가 유지된다
⚠️ 요구사항 오류는 가장 비싸다

요구사항 단계의 실수는 뒤로 갈수록 고치는 비용이 기하급수적으로 커집니다. 처음에 사용자와 충분히 대화해 "진짜 필요한 것"을 파악하는 게 핵심입니다.

설계와 디자인 패턴

설계는 "어떻게 만들 것인가"의 청사진입니다. 시스템을 어떤 모듈로 나누고, 각 모듈이 어떻게 협력할지 정합니다. 좋은 설계는 변경에 강하고(낮은 결합도), 한 곳에 책임을 모읍니다(높은 응집도).

오랜 경험으로 검증된 "자주 쓰는 설계 해법"을 모아 둔 것이 디자인 패턴(design pattern)입니다. 매번 바퀴를 새로 발명하지 않고, 검증된 틀을 가져다 씁니다.

패턴해결하는 문제비유
싱글턴(Singleton)인스턴스를 하나만 보장한 회사의 대표이사 한 명
팩토리(Factory)객체 생성을 한곳에 위임주문하면 알아서 만들어 주는 공장
옵저버(Observer)상태 변화를 구독자에 알림유튜브 구독 알림
전략(Strategy)알고리즘을 갈아끼움상황별로 바꾸는 결제 수단
singleton.py
# 싱글턴: 인스턴스가 단 하나만 존재하도록 보장
class Config:
    _instance = None

    def __new__(cls):
        if cls._instance is None:           # 아직 없으면
            cls._instance = super().__new__(cls)  # 한 번만 생성
        return cls._instance                 # 항상 같은 객체 반환

a = Config()
b = Config()
print(a is b)   # True — 둘은 완전히 같은 하나의 객체

버전 관리 (Git)

여러 사람이 같은 코드를 동시에 고치면 충돌이 나고 누가 무엇을 바꿨는지 알 수 없게 됩니다. 버전 관리 시스템(VCS)은 코드의 모든 변경 이력을 기록하고, 필요하면 과거로 되돌리며, 여러 사람의 작업을 안전하게 합쳐 줍니다. 가장 널리 쓰이는 것이 Git입니다.

핵심 개념은 브랜치(branch)입니다. 본 줄기(main)를 건드리지 않고 곁가지를 따로 만들어 작업한 뒤, 완성되면 본 줄기에 합칩니다(merge).

feature 브랜치에서 작업하고, 완성되면 main에 병합
git_workflow.sh
# 새 기능을 위한 브랜치 만들고 이동
git checkout -b feature/login

# 변경을 기록(커밋)
git add .
git commit -m "로그인 화면 추가"

# 본 줄기로 돌아와 작업을 합치기
git checkout main
git merge feature/login

# 원격 저장소(GitHub 등)에 올리기
git push origin main
💡 커밋은 자주, 메시지는 명확하게

작은 단위로 자주 커밋하면 문제 발생 시 되돌리기 쉽습니다. 커밋 메시지는 "무엇을 왜 바꿨는지"가 드러나게 쓰세요.

테스트 (Testing)

테스트는 코드가 기대대로 동작하는지 자동으로 검증하는 일입니다. 잘 갖춘 테스트는 새 코드가 기존 기능을 망가뜨리지 않았는지 즉시 알려 줍니다. 테스트는 보통 테스트 피라미드 모양으로 균형을 잡습니다 — 빠르고 저렴한 단위 테스트는 많이, 느리고 비싼 E2E 테스트는 적게.

아래일수록 많고 빠르게, 위일수록 적고 신중하게
  • 단위 테스트(Unit) — 함수·클래스 하나를 따로 검증. 가장 빠르고 기본.
  • 통합 테스트(Integration) — 여러 모듈이 함께 잘 동작하는지 검증.
  • E2E 테스트 — 사용자 시나리오 전체를 실제처럼 검증.
test_calc.py
def add(a, b):
    return a + b

# 단위 테스트: add 함수 하나만 검증
def test_add_positive():
    assert add(2, 3) == 5          # 기대값과 같아야 통과

def test_add_negative():
    assert add(-1, -1) == -2

# pytest 로 실행하면 모든 test_ 함수를 자동으로 돌려 준다
# $ pytest test_calc.py   →   2 passed

유지보수 (Maintenance)

소프트웨어는 배포로 끝나지 않습니다. 오히려 전체 비용의 절반 이상이 출시 후 유지보수에 들어간다고 알려져 있습니다. 유지보수는 보통 네 가지로 나눕니다.

종류목적예시
수정(Corrective)버그 고치기로그인 오류 패치
적응(Adaptive)환경 변화 대응새 OS·브라우저 지원
완전화(Perfective)기능·성능 개선속도 최적화, UI 개선
예방(Preventive)미래 문제 예방리팩터링, 기술 부채 정리

오래 유지하려면 코드가 읽기 쉬워야 합니다. 당장 돌아가지만 지저분한 코드를 쌓아 두면 나중에 갚아야 할 빚이 되는데, 이를 기술 부채(technical debt)라 합니다. 주기적인 리팩터링(refactoring) — 동작은 그대로 두고 구조만 개선하기 — 으로 부채를 갚아야 합니다.

⚠️ 기술 부채를 방치하지 말 것

"일단 돌아가니까"로 미룬 정리는 이자가 붙습니다. 새 기능을 추가할 때마다 점점 더 느려지고 위험해지므로, 작은 리팩터링을 꾸준히 하는 편이 결국 빠릅니다.

애자일과 스크럼 상세

애자일은 "철학"이고, 스크럼(Scrum)은 그 철학을 실천하는 가장 인기 있는 "틀"입니다. 스크럼의 핵심은 스프린트(sprint) — 보통 1~4주의 짧은 개발 주기입니다. 한 스프린트가 끝나면 "실제로 동작하는 결과물"이 나오고, 거기서 피드백을 받아 다음 스프린트를 더 낫게 만듭니다.

백로그 → 계획 → 스프린트 → 리뷰·회고, 그리고 다시 반복

스크럼에는 정해진 세 가지 역할이 있습니다.

역할책임비유
제품 책임자(PO)무엇을 만들지 우선순위 결정"무엇을"의 주인
스크럼 마스터팀이 잘 굴러가게 장애물 제거코치·심판
개발팀실제로 만들고 품질 책임"어떻게"의 주인

그리고 스프린트를 굴리는 네 가지 의식(ceremony)이 있습니다.

  • 스프린트 계획 — 이번 스프린트에 무엇을 할지 백로그에서 골라 담음.
  • 데일리 스탠드업 — 매일 15분, "어제 한 일·오늘 할 일·막힌 것" 공유.
  • 스프린트 리뷰 — 만든 결과물을 이해관계자에게 시연하고 피드백 받음.
  • 회고(retrospective) — 팀의 일하는 방식 자체를 돌아보고 개선점 도출.
💡 칸반과의 차이

스크럼이 "정해진 기간(스프린트)"으로 끊는다면, 칸반(Kanban)은 기간 없이 작업 흐름을 보드(To Do → 진행 → 완료)로 시각화하고 동시 진행 수를 제한합니다. 둘을 섞은 "스크럼반"도 흔합니다.

주요 디자인 패턴

앞에서 디자인 패턴의 개념을 봤으니, 실무에서 가장 자주 쓰이는 패턴 셋을 코드로 익혀 봅니다. 패턴은 "이런 상황엔 이렇게 구조를 짜라"는 검증된 처방전입니다. GoF(Gang of Four) 패턴은 보통 생성·구조·행위 세 갈래로 나뉩니다.

전략 패턴 (Strategy)

알고리즘(동작)을 갈아 끼울 수 있게 분리합니다. 거대한 if-else 대신, 동작을 객체로 만들어 상황에 맞게 주입합니다.

strategy.py
# 결제 방식(전략)을 갈아 끼운다
def pay_card(amount):  return f"카드로 {amount}원 결제"
def pay_cash(amount):  return f"현금으로 {amount}원 결제"

class Checkout:
    def __init__(self, strategy):   # 전략을 주입받음
        self.strategy = strategy

    def pay(self, amount):
        return self.strategy(amount)  # 어떤 전략이든 동일하게 호출

print(Checkout(pay_card).pay(5000))  # 카드로 5000원 결제
print(Checkout(pay_cash).pay(3000))  # 현금으로 3000원 결제

옵저버 패턴 (Observer)

한 객체의 상태가 바뀌면 구독자들에게 자동으로 알립니다. 유튜브 알림, UI 이벤트, 이벤트 버스가 모두 이 패턴입니다.

observer.py
class Channel:
    def __init__(self):
        self.subscribers = []          # 구독자 목록

    def subscribe(self, fn):
        self.subscribers.append(fn)

    def upload(self, title):           # 상태 변화 발생
        for notify in self.subscribers:
            notify(title)              # 모든 구독자에게 알림

ch = Channel()
ch.subscribe(lambda t: print("철수에게 알림:", t))
ch.subscribe(lambda t: print("영희에게 알림:", t))
ch.upload("새 영상!")     # 두 구독자 모두 자동 통지

데코레이터 패턴 (Decorator)

객체를 감싸서 기능을 덧붙입니다. 원본을 고치지 않고 행동을 층층이 추가합니다 — 커피에 우유·시럽을 얹듯이.

decorator.py
def coffee():        return ("아메리카노", 3000)

def with_milk(base):                 # 기능을 감싸서 덧붙임
    name, price = base()
    return (name + "+우유", price + 500)

def with_syrup(base):
    name, price = base()
    return (name + "+시럽", price + 300)

order = with_syrup(lambda: with_milk(coffee))  # 층층이 감싸기
print(order())   # ('아메리카노+우유+시럽', 3800)
패턴분류한 줄 요약
전략행위알고리즘을 갈아 끼움
옵저버행위변화를 구독자에 자동 통지
데코레이터구조감싸서 기능 추가
어댑터구조맞지 않는 인터페이스를 변환
⚠️ 패턴 남용 주의

패턴은 도구이지 목표가 아닙니다. 간단한 문제에 억지로 패턴을 끼우면 오히려 코드가 복잡해집니다. "이 패턴이 푸는 문제가 지금 내게 있는가?"를 먼저 물으세요.

UML 다이어그램

설계를 말로만 설명하면 오해가 생깁니다. UML(Unified Modeling Language)은 시스템의 구조와 동작을 표준 그림으로 그리는 공통 언어입니다. 가장 자주 쓰는 둘은 클래스 다이어그램(구조)과 시퀀스 다이어그램(시간 순 상호작용)입니다.

왼쪽: 클래스 간 상속 관계 / 오른쪽: 시간 순 메시지 주고받기
다이어그램보여 주는 것언제 쓰나
클래스클래스·속성·관계(정적 구조)설계 구조를 잡을 때
시퀀스객체 간 시간 순 메시지흐름·API 호출을 설명할 때
유스케이스사용자와 기능의 관계요구사항 정리 때
활동(Activity)작업 흐름·분기업무 절차를 그릴 때
💡 완벽한 UML보다 소통이 목적

UML은 표준이지만, 실무에선 화이트보드에 빠르게 그리는 약식 다이어그램이 더 흔합니다. 목적은 "같은 그림을 보며 같은 이해에 도달"하는 것이니, 형식에 너무 얽매이지 마세요.

CI/CD 파이프라인

브랜치에서 작업한 코드를 본 줄기에 합칠 때마다 사람이 일일이 테스트하고 배포하면 느리고 실수가 생깁니다. CI/CD는 이 과정을 자동화합니다.

  • CI(Continuous Integration, 지속적 통합) — 코드를 올릴 때마다 자동으로 빌드·테스트해서 망가진 곳을 즉시 찾아냅니다.
  • CD(Continuous Delivery/Deployment, 지속적 배포) — 테스트를 통과한 코드를 자동으로 배포까지 이어 갑니다.
push 한 번이 빌드→테스트→배포까지 자동으로 흐른다 (한 단계라도 실패하면 멈춤)
.github/workflows/ci.yml
# GitHub Actions CI 파이프라인 예시
name: CI
on: [push]                  # 코드를 push 할 때마다 실행

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4      # 코드 받아오기
      - name: 의존성 설치
        run: pnpm install
      - name: 테스트 실행
        run: pnpm test                 # 실패하면 여기서 빌드 멈춤
      - name: 빌드
        run: pnpm build
💡 빌드는 "항상 초록"

CI의 목표는 본 줄기(main)를 언제나 배포 가능한 상태로 유지하는 것입니다. 빨간 빌드(테스트 실패)는 다른 모든 일보다 먼저 고쳐야 합니다 — 망가진 채로 쌓이면 누가 깼는지 찾기 어려워집니다.

코드 리뷰 문화

코드 리뷰(code review)는 코드를 본 줄기에 합치기 전, 동료가 변경 사항(보통 PR, Pull Request)을 읽고 피드백하는 과정입니다. 목적은 단지 버그 잡기가 아니라 지식 공유·일관성 유지·집단 책임입니다.

PR → 리뷰 → 피드백·수정 반복 → 승인되면 병합
좋은 습관이유
PR을 작게 유지리뷰가 쉽고 빠르며 버그가 덜 새어 나감
코드가 아니라 코드를 비평"너는"이 아니라 "이 함수는"으로 — 사람이 아닌 코드 얘기
왜(why)를 설명지적만 말고 더 나은 방향과 이유를 제시
칭찬도 남기기잘한 부분을 짚으면 좋은 패턴이 퍼짐
⚠️ 리뷰는 비난이 아니다

리뷰의 목적은 사람을 평가하는 게 아니라 코드를 함께 좋게 만드는 것입니다. 받는 쪽은 방어적으로 굴지 말고, 주는 쪽은 명령조 대신 질문·제안형으로 — "이렇게 하면 어떨까요?"가 "이건 틀렸어요"보다 낫습니다.

기술 부채 관리

앞서 유지보수에서 기술 부채를 소개했습니다. 여기서는 그것을 어떻게 관리하는지 봅니다. 부채 자체가 항상 나쁜 건 아닙니다. 마감을 맞추려 "일단 빠르게" 짜는 건 의도된 부채일 수 있습니다. 문제는 갚지 않고 방치할 때입니다.

마틴 파울러의 기술 부채 사분면 — 의도되고 신중한 부채만 그나마 건강하다

관리의 핵심은 부채를 보이게 만드는 것입니다. 머릿속에만 있으면 영원히 안 갚게 됩니다.

  • 기록 — 코드에 TODO·FIXME 주석이나 이슈로 남겨 추적.
  • 예산 배정 — 스프린트마다 일정 비율(예: 20%)을 리팩터링·정리에 할당.
  • 보이스카우트 규칙 — "왔을 때보다 깨끗하게" 만지는 김에 조금씩 개선.

일정 추정

"이거 얼마나 걸려요?"는 개발에서 가장 어려운 질문입니다. 미래의 불확실성, 모르는 변수, 사람마다 다른 속도 때문입니다. 그래서 정확한 시간보다 상대적 크기로 추정하는 기법이 발전했습니다.

  • 스토리 포인트(Story Point) — "이 작업이 저것보다 몇 배 큰가"를 점수로. 절대 시간이 아닌 상대 크기라 더 안정적입니다.
  • 플래닝 포커 — 팀원이 각자 카드(보통 피보나치: 1,2,3,5,8,13)를 동시에 내 추정. 차이가 크면 토론으로 합의합니다.
  • 삼점 추정 — 낙관·비관·현실 세 값을 섞어 평균을 냄: (낙관 + 4×현실 + 비관) / 6.
간격이 벌어지는 피보나치 — 클수록 불확실성이 커짐을 반영
⚠️ 추정은 약속이 아니다

추정은 현재 정보로 본 예측이지 확정된 마감이 아닙니다. 작업을 잘게 쪼갤수록 정확해지고, 큰 덩어리(13점 이상)는 더 나눠야 한다는 신호입니다. 또 팀의 실제 처리량(속도, velocity)을 누적 관찰하면 점점 추정이 현실에 맞아 갑니다.

🎯 핵심 정리
  • 소프트웨어 공학은 코드 작성을 넘어, 만들고 유지하는 전 과정을 체계화한다.
  • SDLC는 요구사항 → 설계 → 구현 → 테스트 → 배포 → 유지보수의 순환이다.
  • 폭포수는 한 방향, 애자일은 짧은 반복 — 변화가 많으면 애자일이 유리하다.
  • Git의 브랜치로 본 줄기를 지키며 안전하게 협업하고 병합한다.
  • 테스트 피라미드는 단위 테스트를 많이, E2E를 적게 두어 균형을 잡는다.
  • 스크럼은 스프린트·세 역할(PO·SM·개발팀)·네 의식으로 애자일을 실천한다.
  • 전략·옵저버·데코레이터 같은 디자인 패턴은 검증된 설계 처방전이다.
  • UML(클래스·시퀀스)은 설계를 표준 그림으로 소통하는 공통 언어다.
  • CI/CD는 빌드·테스트·배포를 자동화해 본 줄기를 항상 배포 가능하게 한다.
  • 코드 리뷰는 버그 잡기를 넘어 지식 공유와 집단 책임을 키운다.
  • 기술 부채는 보이게 만들어 꾸준히 갚고, 일정은 상대 크기로 추정한다.
SDLC
소프트웨어 개발 생명주기. 요구사항부터 유지보수까지의 전 과정.
애자일(Agile)
짧은 반복(스프린트)으로 만들고 피드백받아 개선하는 개발 방식.
브랜치(Branch)
본 줄기와 독립적으로 작업하기 위해 따로 낸 코드의 곁가지.
디자인 패턴
자주 쓰는 설계 문제에 대한 검증된 해법의 틀.
기술 부채
당장의 편의를 위해 미룬 정리가 나중에 갚아야 할 비용으로 쌓인 것.
리팩터링
동작은 그대로 두고 코드 구조만 개선하는 작업.
스프린트
스크럼에서 동작하는 결과물을 만드는 1~4주의 짧은 개발 주기.
백로그
해야 할 일을 우선순위대로 모아 둔 목록(제품 백로그·스프린트 백로그).
전략·옵저버·데코레이터
각각 알고리즘 교체·변화 통지·기능 덧붙이기를 위한 디자인 패턴.
UML
시스템 구조·동작을 표준 그림으로 그리는 모델링 언어(클래스·시퀀스 등).
CI/CD
지속적 통합·배포. 빌드·테스트·배포를 자동화하는 파이프라인.
코드 리뷰 / PR
변경 사항(풀 리퀘스트)을 동료가 검토해 품질·지식을 공유하는 과정.
스토리 포인트
작업의 상대적 크기를 점수로 매기는 추정 단위(보통 피보나치).
속도(Velocity)
한 스프린트에 팀이 처리하는 스토리 포인트의 누적 평균.
