프로그래밍 언어론

언어들은 왜 서로 다르게 생겼을까요? 문법·실행 방식·타입·메모리 관리를 비교하며 언어를 보는 눈을 기릅니다.

프로그래밍 언어론이란?

프로그래밍 언어론(programming languages)은 언어를 만들고 비교하는 원리를 배우는 과목입니다. 특정 언어 하나를 더 잘 쓰는 법이 아니라, "왜 파이썬은 타입을 안 적어도 되고, C는 적어야 할까?" 같은 언어 설계의 선택과 결과를 다룹니다.

외국어를 여러 개 배우면 모국어가 다시 보이듯, 여러 언어의 공통 원리를 알면 새 언어도 빠르게 익히고, 상황에 맞는 언어를 고를 수 있게 됩니다.

📌 이 페이지에서 배우는 것

문법과 의미론 · 컴파일 vs 인터프리트 · 타입 시스템 · 변수 스코프와 바인딩 · 패러다임 · 메모리 관리(GC)까지, 언어를 가로지르는 핵심 개념을 다룹니다.

문법과 의미론

언어를 이해하는 두 축은 문법(syntax)과 의미론(semantics)입니다.

  • 문법 — 코드가 "올바르게 생겼는가". 괄호·세미콜론·키워드 같은 형태 규칙. 비유하면 문장의 맞춤법·어순.
  • 의미론 — 그 코드가 "무슨 뜻이고 어떻게 동작하는가". 비유하면 문장의 실제 의미.

문법만 맞고 의미가 이상할 수 있습니다. x = 1 / 0 은 문법적으로는 완벽하지만, 실행하면 의미상 오류(0으로 나눔)가 납니다.

코드는 먼저 문법(형태)으로, 그다음 의미(동작)로 해석된다
syntax.txt
# 같은 의미(1 더하기 2)를 서로 다른 문법으로 표현
# Python
result = 1 + 2

# Lisp — 연산자가 앞에 오는 전위 표기 문법
# (+ 1 2)

# 문법은 달라도 의미(덧셈)는 동일하다

컴파일 vs 인터프리트

사람이 쓴 코드를 컴퓨터가 실행하려면 기계어로 바꿔야 합니다. 그 방식이 두 갈래입니다.

  • 컴파일(compile) — 실행 전에 전체를 기계어로 번역해 실행 파일을 만든다. 번역서를 미리 출판하는 것. (C, C++, Rust, Go)
  • 인터프리트(interpret) — 실행하는 순간 한 줄씩 읽어 바로 수행한다. 동시통역사. (Python, JavaScript, Ruby)
컴파일은 미리 한 번에, 인터프리트는 실행하며 그때그때
구분컴파일인터프리트
실행 속도빠름상대적으로 느림
오류 발견실행 전(컴파일 시)실행 중에야
개발 편의빌드 단계 필요바로 실행·실험
예시 언어C, Rust, GoPython, JavaScript
💡 그 중간도 있다

Java나 C#은 바이트코드로 컴파일한 뒤 가상 머신(VM)이 실행합니다. JavaScript도 요즘은 실행 중 자주 쓰는 부분만 기계어로 바꾸는 JIT(Just-In-Time) 컴파일을 씁니다.

타입 시스템 (정적·동적)

타입(type)은 값의 종류(정수·문자열·불린 등)입니다. 타입 시스템은 "어떤 연산이 어떤 값에 허용되는가"를 정하는 규칙이고, 그 검사를 언제 하느냐로 두 갈래로 나뉩니다.

  • 정적 타입(static) — 실행 전에 검사. 타입 오류를 미리 잡지만 타입을 적어야 함. (C, Java, TypeScript)
  • 동적 타입(dynamic) — 실행 중에 검사. 자유롭고 빠르게 작성하지만 오류가 런타임에 터질 수 있음. (Python, JavaScript)
types.txt
// Java — 정적 타입: 타입을 명시, 컴파일 때 검사
int    age  = 20;
String name = "철수";
// age = "스무살";   ← 컴파일 오류! (int 자리에 String)
types.py
# Python — 동적 타입: 타입을 안 적음, 실행 중 검사
age  = 20
name = "철수"
age  = "스무살"     # 허용됨 — 변수에 다른 타입 재할당 가능

# 오류는 실행 중에야 드러난다
# 1 + "둘"  →  TypeError (런타임에 발생)
⚠️ 트레이드오프

정적 타입은 안전성(오류를 일찍 발견)을, 동적 타입은 유연성·빠른 작성을 줍니다. 어느 쪽이 "더 낫다"기보다 프로젝트 규모와 목적에 따라 고릅니다.

변수 스코프와 바인딩

스코프(scope)는 변수가 보이는 범위입니다. 함수 안에서 만든 변수는 보통 그 함수 안에서만 통하죠. 바인딩(binding)은 이름(변수명)이 실제 값·메모리와 묶이는 일을 말합니다.

대부분의 언어는 렉시컬 스코프(lexical scope)를 씁니다 — 변수가 어디서 "쓰였는가"가 아니라 어디에 "적혔는가"(코드 위치)로 범위가 정해집니다. 안쪽 블록은 바깥 변수를 볼 수 있지만, 바깥은 안쪽을 못 봅니다.

안쪽은 바깥 변수를 보지만, 바깥은 안쪽 변수를 못 본다
scope.py
g = "전역"            # 전역 스코프

def outer():
    f = "함수 안"      # outer 안에서만 보임
    print(g)          # OK — 바깥(전역)을 볼 수 있음
    print(f)          # OK

outer()
# print(f)   ← NameError! f 는 outer 밖에서는 보이지 않음

패러다임 (명령형·함수형·객체지향)

패러다임(paradigm)은 문제를 푸는 사고방식입니다. 같은 문제도 어떤 관점으로 보느냐에 따라 코드 모양이 완전히 달라집니다.

  • 명령형(imperative) — "이렇게, 저렇게 해라"라고 단계를 나열. (C의 반복문)
  • 함수형(functional) — 함수의 조합으로 표현, 상태 변경을 피함. (map/filter)
  • 객체지향(OOP) — 데이터와 동작을 객체로 묶음. (클래스)
paradigm.py
nums = [1, 2, 3, 4]

# 명령형: 단계를 직접 나열
total = 0
for n in nums:
    total += n * n

# 함수형: 함수를 조합해 "무엇을" 원하는지 표현
total = sum(map(lambda n: n * n, nums))

# 객체지향: 데이터와 동작을 객체로 묶음
class Counter:
    def __init__(self):
        self.value = 0
    def add(self, n):
        self.value += n
💡 한 언어, 여러 패러다임

요즘 언어 대부분은 멀티 패러다임입니다. 파이썬·자바스크립트는 명령형·함수형·객체지향을 모두 지원하므로, 상황에 맞는 스타일을 섞어 쓸 수 있습니다.

메모리 관리 (GC)

프로그램이 만든 데이터는 메모리를 차지합니다. 더 안 쓰는 메모리를 돌려주지 않으면 점점 쌓여(메모리 누수) 결국 프로그램이 죽습니다. 이 정리를 누가 하느냐가 언어마다 다릅니다.

  • 수동 관리 — 프로그래머가 직접 할당·해제. 빠르지만 실수하기 쉬움. (C의 malloc/free)
  • 가비지 컬렉션(GC) — 언어 런타임이 "더 이상 아무도 가리키지 않는" 데이터를 자동으로 청소. (Java, Python, Go)
GC: 루트에서 도달 못 하는 데이터를 "쓰레기"로 보고 자동 회수
memory.c
/* C — 수동 관리: 직접 해제해야 함 */
int *p = malloc(sizeof(int) * 100);   /* 할당 */
/* ... 사용 ... */
free(p);   /* 해제를 잊으면 메모리 누수! */

/* Python/Java — 자동(GC): free 가 없다.
   참조가 사라지면 런타임이 알아서 회수한다. */
⚠️ GC도 공짜는 아니다

GC는 편하지만, 청소하는 순간 프로그램이 잠깐 멈출 수 있습니다(GC 일시정지). 그래서 실시간성이 중요한 분야에선 Rust처럼 소유권(ownership)으로 GC 없이 안전하게 관리하는 방식을 쓰기도 합니다.

람다와 클로저

람다(lambda)는 이름 없는 함수입니다. 간단한 동작을 그 자리에서 만들어 값처럼 넘길 때 씁니다. 클로저(closure)는 한 걸음 더 나아가, 람다가 자신이 만들어진 자리의 바깥 변수를 기억하는 현상입니다. 함수가 "환경을 봉투에 싸서(close over)" 들고 다닌다고 생각하면 됩니다.

바깥 함수가 끝나도, 반환된 클로저는 count 변수를 살려 들고 다닌다
closure.txt
# Python — 람다(이름 없는 함수)
square = lambda x: x * x
print(square(5))            # 25

# 클로저: 안쪽 함수가 바깥 변수(count)를 기억
def make_counter():
    count = 0
    def inc():
        nonlocal count
        count += 1          # 바깥 변수를 계속 기억하고 갱신
        return count
    return inc

c = make_counter()
print(c(), c(), c())        # 1 2 3 — count 가 살아남음
closure.js
// JavaScript — 화살표 함수(람다) + 클로저
const makeCounter = () => {
  let count = 0;
  return () => ++count;     // count 를 봉투에 싸서 기억
};
const c = makeCounter();
console.log(c(), c(), c()); // 1 2 3
💡 어디에 쓰나

클로저는 상태를 숨겨 캡슐화하거나, 콜백·이벤트 핸들러가 주변 값을 기억하게 할 때 핵심입니다. 함수형 프로그래밍의 토대이기도 합니다.

고차 함수

고차 함수(higher-order function)는 함수를 인자로 받거나, 함수를 결과로 돌려주는 함수입니다. 함수를 숫자나 문자열처럼 값으로 다룰 수 있는(일급 함수) 언어에서 가능합니다. 대표 삼총사가 map·filter·reduce입니다.

  • map — 모든 원소를 같은 규칙으로 변환.
  • filter — 조건에 맞는 원소만 골라냄.
  • reduce — 전체를 하나의 값으로 접어 합침.
map은 변환, filter는 선별, reduce는 하나로 접기
higher_order.txt
# Python
nums = [1, 2, 3, 4]
print(list(map(lambda x: x * 2, nums)))        # [2, 4, 6, 8]
print(list(filter(lambda x: x % 2 == 0, nums)))# [2, 4]
from functools import reduce
print(reduce(lambda a, b: a + b, nums))        # 10
higher_order.js
// JavaScript — 메서드 체이닝으로 연결
const nums = [1, 2, 3, 4];
nums.map(x => x * 2)                  // [2,4,6,8]
    .filter(x => x > 4)              // [6,8]
    .reduce((a, b) => a + b, 0);     // 14

제네릭 (매개변수 다형성)

제네릭(generic)은 타입을 나중에 채울 빈칸으로 두고 코드를 작성하는 기법입니다. "정수 리스트", "문자열 리스트"를 따로 만들 필요 없이, "무언가의 리스트"를 한 번만 만들어 재사용합니다. 학술 용어로 매개변수 다형성(parametric polymorphism)이라 합니다.

제네릭이 없다면, 타입마다 거의 똑같은 코드를 복붙하거나(중복), 모든 타입을 받는 느슨한 타입을 써서 안전성을 잃어야 합니다. 제네릭은 재사용성과 타입 안전성을 동시에 잡습니다.

generics.txt
// Java —  가 "나중에 채울 타입" 빈칸
class Box {
    private T value;
    void set(T v) { value = v; }
    T get() { return value; }
}
Box b = new Box<>();
b.set("철수");
// b.set(123);  ← 컴파일 오류! String 자리에 int (타입 안전)
generics2.txt
// TypeScript — 함수도 제네릭으로
function first(arr: T[]): T {
  return arr[0];
}
first([1, 2, 3]);   // 반환 타입이 number 로 추론됨
first(["a", "b"]);          // 반환 타입이 string
💡 다형성의 갈래

같은 코드가 여러 타입에 작동하는 것을 다형성이라 합니다. 제네릭은 매개변수 다형성, 부모 타입으로 자식을 다루는 것은 서브타입 다형성, 같은 이름의 다른 함수는 오버로딩(애드혹 다형성)입니다.

예외 처리 모델

프로그램은 늘 예상 못 한 일을 만납니다 — 파일이 없거나, 0으로 나누거나, 네트워크가 끊기거나. 예외(exception)는 이런 비정상 상황을 정상 흐름과 분리해 다루는 장치입니다. 문제가 생긴 곳에서 예외를 던지고(throw), 처리할 수 있는 바깥에서 잡습니다(catch).

예외는 잡힐 때까지 호출 스택을 거슬러 올라간다
exception.txt
# Python — try / except / finally
try:
    x = 10 / 0
except ZeroDivisionError as e:     # 특정 예외만 잡기
    print("0으로 나눌 수 없어요:", e)
finally:
    print("정리 작업은 항상 실행")   # 성공·실패 무관
모델방식예시 언어
예외(try/catch)던지고 잡기Java, Python, C++
오류값 반환오류를 값으로 돌려줌Go (err), C
Result/Option 타입성공/실패를 타입으로Rust, Haskell
errors_compare.txt
// Go — 예외 대신 오류를 값으로 반환
f, err := os.Open("a.txt")
if err != nil {            // 매번 명시적으로 확인
    return err
}
⚠️ 검사 vs 비검사 예외

Java는 반드시 처리하도록 강제하는 검사 예외(checked)가 있지만, 대부분의 언어는 강제하지 않는 비검사 예외만 씁니다. 강제는 안전하지만 코드가 장황해지는 트레이드오프가 있습니다.

언어의 동시성 모델

여러 작업을 동시에 진행하는 방법은 언어마다 철학이 다릅니다. 데이터를 공유할 것인가 메시지로 주고받을 것인가, 무거운 스레드를 쓸 것인가 가벼운 단위를 쓸 것인가로 갈립니다.

데이터를 공유하면 락이 필요하고, 메시지로 주고받으면 충돌이 줄어든다
모델핵심 아이디어대표 언어
스레드 + 락메모리 공유, 락으로 보호Java, C++
CSP(채널)가벼운 단위가 채널로 통신Go (goroutine)
액터 모델독립 객체가 메시지 교환Erlang, Akka
async/await대기 중 다른 일 처리JS, Python, Rust
concurrency.txt
// Go — goroutine + 채널(메시지 전달)
ch := make(chan int)
go func() { ch <- 42 }()   // 가벼운 동시 작업이 결과를 채널로
fmt.Println(<-ch)          // 42 — 채널에서 받기
async.txt
// JavaScript — async/await (대기 중 멈추지 않음)
async function load() {
  const res = await fetch("/data");  // 기다리는 동안 다른 일 처리
  return res.json();
}
🚨 경쟁 조건(Race Condition)

여러 스레드가 같은 데이터를 동시에 수정하면 결과가 실행 순서에 따라 달라져 버그가 됩니다. 공유 메모리 모델의 가장 흔한 함정으로, 락이나 불변 데이터로 막습니다.

메타프로그래밍과 리플렉션

메타프로그래밍(metaprogramming)은 코드가 코드를 다루는 기법입니다. 실행 중에 새 함수를 만들거나, 클래스 구조를 들여다보거나, 코드를 변형합니다. 리플렉션(reflection)은 그중에서도 프로그램이 자기 자신의 구조(타입·메서드·속성)를 실행 중에 살펴보고 조작하는 능력입니다.

리플렉션: 프로그램이 거울을 보듯 자기 구조를 들여다본다
reflection.txt
# Python — 실행 중에 객체의 구조를 들여다보기
class User:
    def __init__(self): self.name = "철수"
    def greet(self): return "안녕"

u = User()
print(type(u).__name__)          # 'User' — 타입 이름
print(hasattr(u, "greet"))       # True — 메서드 존재 확인
print(getattr(u, "name"))        # '철수' — 이름으로 속성 읽기
m = getattr(u, "greet")          # 메서드를 이름으로 꺼내 호출
print(m())                       # '안녕'
⚠️ 강력하지만 위험하다

리플렉션은 ORM·테스트 프레임워크·직렬화 라이브러리의 마법을 가능하게 합니다. 하지만 컴파일러의 타입 검사를 우회하고 느리며 디버깅이 어렵습니다. 꼭 필요할 때만 절제해서 쓰세요.

언어 평가 기준

"어떤 언어가 가장 좋은가?"에 정답은 없습니다. 대신 언어를 여러 잣대로 평가하고, 프로젝트 목적에 맞춰 고릅니다. 한 잣대를 높이면 다른 잣대가 낮아지는 트레이드오프가 흔합니다.

  • 가독성(readability) — 코드를 읽고 이해하기 쉬운가. 유지보수 비용을 좌우.
  • 작성성(writability) — 생각을 코드로 빠르게 옮기는가.
  • 표현력(expressiveness) — 적은 코드로 많은 의미를 담는가.
  • 성능(performance) — 실행이 빠르고 자원을 적게 쓰는가.
  • 안전성(safety) — 오류를 일찍·자동으로 막는가.
  • 생태계 — 라이브러리·도구·커뮤니티가 풍부한가.
대체로 성능이 높은 언어는 작성 속도를, 표현력 높은 언어는 실행 속도를 양보한다
언어강한 점약한 점
C성능·하드웨어 제어안전성·생산성 낮음
Rust성능 + 메모리 안전학습 곡선 가파름
Python가독성·표현력·생태계실행 속도 느림
Go단순함·동시성표현력 제한적
JavaScript어디서나 실행·생태계느슨한 타입의 함정
💡 "올바른 도구"라는 사고

운영체제 커널은 C/Rust, 데이터 분석은 Python, 웹 프런트엔드는 JavaScript처럼, 문제 영역에 맞는 언어를 고르는 안목이 언어론을 배우는 진짜 목적입니다.

🎯 핵심 정리
  • 문법은 코드의 형태, 의미론은 코드의 동작 — 둘은 다른 층위다.
  • 컴파일은 미리 한 번에 번역(빠름), 인터프리트는 실행하며 한 줄씩(유연).
  • 정적 타입은 안전성을, 동적 타입은 유연성을 준다.
  • 스코프는 변수가 보이는 범위 — 안쪽은 바깥을 보지만 바깥은 안쪽을 못 본다.
  • 메모리는 수동(free)으로 정리하거나 GC가 자동으로 회수한다.
  • 람다는 이름 없는 함수, 클로저는 바깥 변수를 기억하는 함수다.
  • 고차 함수(map·filter·reduce)는 함수를 값으로 다뤄 코드를 간결하게 한다.
  • 제네릭은 타입을 빈칸으로 둬 재사용성과 타입 안전성을 함께 얻는다.
  • 예외는 비정상 상황을 정상 흐름과 분리해 던지고 잡는다(언어마다 모델이 다름).
  • 동시성은 공유 메모리+락, 메시지 전달(채널·액터), async/await 등으로 다룬다.
  • 메타프로그래밍·리플렉션은 코드가 코드(자기 구조)를 다루는 강력하지만 위험한 기법이다.
  • 언어는 가독성·표현력·성능·안전성 등 여러 잣대의 트레이드오프로 평가한다.
문법 / 의미론
코드가 올바르게 생겼는가(형태) vs 무슨 뜻으로 동작하는가(의미).
컴파일러 / 인터프리터
전체를 미리 기계어로 번역하는 도구 vs 실행 중 한 줄씩 수행하는 도구.
정적 / 동적 타입
타입 검사를 실행 전에 하느냐(정적), 실행 중에 하느냐(동적).
스코프(Scope)
변수가 유효하게 보이는 코드상의 범위.
가비지 컬렉션(GC)
더 이상 참조되지 않는 메모리를 런타임이 자동으로 회수하는 기능.
람다 / 클로저
이름 없는 함수, 그리고 자신이 만들어진 자리의 바깥 변수를 기억하는 함수.
고차 함수
함수를 인자로 받거나 함수를 반환하는 함수(map·filter·reduce).
제네릭(Generic)
타입을 빈칸으로 두고 작성해 재사용성과 타입 안전성을 함께 얻는 기법(매개변수 다형성).
예외(Exception)
비정상 상황을 정상 흐름과 분리해 던지고(throw) 잡는(catch) 장치.
경쟁 조건
여러 스레드가 공유 데이터를 동시에 수정해 결과가 실행 순서에 따라 달라지는 버그.
리플렉션(Reflection)
프로그램이 실행 중에 자기 자신의 구조(타입·메서드)를 살펴보고 조작하는 능력.
다형성(Polymorphism)
같은 코드가 여러 타입에 작동하는 성질(매개변수·서브타입·애드혹).
