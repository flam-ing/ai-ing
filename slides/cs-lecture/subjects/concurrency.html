병렬 / 동시성

여러 일을 동시에 다루는 프로그램을 짜는 법입니다. 빠르지만 까다롭습니다 — 무엇이 왜 어긋나는지, 어떻게 안전하게 묶는지 배웁니다.

동시성 vs 병렬성

둘은 비슷해 보이지만 다릅니다. 동시성(concurrency)은 여러 작업을 번갈아 가며 조금씩 진행해 "동시에 진행되는 것처럼" 보이게 하는 구조입니다. 병렬성(parallelism)은 여러 작업을 물리적으로 같은 순간에 실행하는 것입니다(여러 CPU 코어 필요).

요리사 한 명이 국을 끓이며 틈틈이 야채를 써는 게 동시성, 요리사 두 명이 각자 동시에 요리하는 게 병렬성입니다.

동시성은 "번갈아", 병렬성은 "진짜 같은 순간에"
📌 이 페이지에서 배우는 것

멀티코어의 필요성 · 프로세스와 스레드 · 경쟁 상태 · 임계구역과 락 · 세마포어 · 교착상태 · 생산자-소비자 · 원자성과 메모리 가시성 · 스레드풀과 비동기 · 액터 모델까지 다룹니다.

왜 필요한가 (멀티코어)

예전엔 CPU 속도(클럭)가 매년 빨라졌지만, 발열과 전력 한계로 2000년대 중반부터 한 코어를 무작정 빠르게 만드는 게 멈췄습니다. 대신 코어 수를 늘리는 방향으로 갔습니다. 오늘날 노트북에도 코어가 8개씩 들어 있죠.

문제는, 코어가 8개라도 프로그램이 한 줄씩 순서대로만 실행되면 코어 7개는 놀게 됩니다. 그 성능을 끌어쓰려면 작업을 여러 갈래로 나눠 동시에 돌려야 합니다 — 그래서 동시성·병렬성이 필요합니다.

💡 암달의 법칙(Amdahl's Law)

프로그램에서 병렬화할 수 없는 부분이 전체 속도 향상의 상한을 정합니다. 10%가 순차적이면 코어를 무한히 늘려도 최대 10배까지밖에 못 빨라집니다. 무작정 코어만 늘린다고 비례해서 빨라지지 않습니다.

프로세스 vs 스레드

프로세스(process)는 실행 중인 프로그램 하나로, 자기만의 독립된 메모리 공간을 가집니다. 스레드(thread)는 한 프로세스 안에서 실행되는 흐름의 갈래로, 같은 메모리를 공유합니다.

메모리를 공유한다는 점이 스레드의 장점이자 위험입니다 — 데이터를 주고받기 쉽지만, 여럿이 같은 데이터를 동시에 건드리면 사고가 납니다.

한 프로세스 안의 스레드들은 메모리를 공유 / 프로세스끼리는 격리됨
구분메모리생성 비용사고 위험
프로세스독립큼낮음(격리)
스레드공유작음높음(공유 데이터)

경쟁 상태 (Race Condition)

경쟁 상태는 둘 이상의 스레드가 같은 데이터를 동시에 바꾸려다, 실행 순서에 따라 결과가 달라지는 버그입니다. count += 1 한 줄도 사실은 "읽기 → 더하기 → 쓰기" 세 단계라, 그 사이에 다른 스레드가 끼어들 수 있습니다.

둘 다 0을 읽고 각자 1로 덮어씀 → 한 번의 증가가 사라짐(lost update)
race.py
import threading

count = 0

def worker():
    global count
    for _ in range(100_000):
        count += 1          # 읽기→더하기→쓰기: 끼어들기 가능!

threads = [threading.Thread(target=worker) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()

print(count)   # 200000 이 아니라 그보다 작게 나올 수 있음
🚫 재현이 어렵다

경쟁 상태는 타이밍에 따라 가끔만 터집니다. 테스트에선 멀쩡하다가 운영에서 터지는 악질 버그라, 처음부터 공유 데이터를 보호하는 게 답입니다.

임계구역과 상호배제

임계구역(critical section)은 공유 데이터를 건드리는 코드 영역입니다. 여기서 사고가 나죠. 상호배제(mutual exclusion)는 "임계구역에는 한 번에 한 스레드만 들어가게" 보장하는 원칙입니다.

화장실 한 칸에 비유하면 임계구역은 칸 안, 상호배제는 "한 명만 들어가고 문을 잠근다"는 규칙입니다.

📌 좋은 동기화의 조건

① 상호배제(한 명만), ② 진행(아무도 없으면 들어갈 수 있어야), ③ 한정 대기(무한정 기다리지 않아야). 세 가지를 모두 만족해야 안전합니다.

락 · 뮤텍스 (Lock / Mutex)

상호배제를 구현하는 가장 기본 도구가 락(lock)입니다. 흔히 뮤텍스(mutex, mutual exclusion)라고도 부릅니다. 임계구역에 들어가기 전 acquire()로 잠그고, 나올 때 release()로 풀어, 그동안 다른 스레드는 기다리게 합니다.

lock.py
import threading

count = 0
lock = threading.Lock()

def worker():
    global count
    for _ in range(100_000):
        with lock:          # 잠그고 ~ 블록 끝나면 자동 해제
            count += 1      # 이제 한 번에 한 스레드만 진입

threads = [threading.Thread(target=worker) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()

print(count)   # 항상 200000 — 안전!
⚠️ 락의 비용

락을 잡은 동안 다른 스레드는 멈춰 기다립니다. 임계구역을 너무 넓게 잡으면 병렬성의 이점이 사라집니다. 꼭 필요한 곳만 짧게 잠그세요. 항상 해제하는 것도 잊지 마세요(with 권장).

세마포어 (Semaphore)

락이 "한 명만"이라면, 세마포어는 "N명까지" 허용하는 신호등입니다. 내부에 카운터를 두고, 자원을 쓸 때마다 줄이고(acquire) 반납할 때 늘립니다(release). 카운터가 0이면 기다립니다.

주차장 입구의 "잔여 5대" 표시판과 같습니다. 차가 들어가면 줄고, 나오면 늘죠. 0이면 입구에서 대기합니다. (카운터가 1뿐인 세마포어가 사실상 락입니다.)

semaphore.py
import threading

# 동시에 최대 3개 스레드만 DB에 접속하도록 제한
db_slots = threading.Semaphore(3)

def query(i):
    with db_slots:          # 슬롯 하나 차지(없으면 대기)
        print(f"{i}번 작업: DB 접속 중")
        # ... 작업 ...
    # 블록을 벗어나면 슬롯 반납

for i in range(10):
    threading.Thread(target=query, args=(i,)).start()

교착상태 (Deadlock)

교착상태는 여러 스레드가 서로가 가진 자원을 기다리며 영원히 멈춰버리는 상황입니다. 좁은 다리에서 마주 온 두 차가 서로 비키길 기다리며 둘 다 못 가는 것과 같습니다.

T1은 A를 쥔 채 B를, T2는 B를 쥔 채 A를 기다림 → 순환 대기로 영원히 멈춤
🚫 교착상태의 4조건

① 상호배제(자원을 한 명만), ② 점유와 대기(쥔 채 더 기다림), ③ 비선점(강제로 뺏을 수 없음), ④ 순환 대기(서로 물고 물림). 네 가지가 모두 성립할 때만 발생하므로, 하나만 깨도 예방됩니다(예: 항상 같은 순서로 락 잡기).

생산자-소비자 문제

아주 흔한 동시성 패턴입니다. 생산자(producer)가 데이터를 만들어 공유 버퍼(큐)에 넣고, 소비자(consumer)가 꺼내 처리합니다. 버퍼가 가득 차면 생산자가 기다리고, 비면 소비자가 기다립니다 — 이 조율이 핵심입니다.

생산 속도와 소비 속도가 달라도 버퍼가 완충 — 가득/비면 서로 기다림
producer_consumer.py
import threading, queue, time

buffer = queue.Queue(maxsize=5)   # 스레드 안전한 큐가 동기화를 대신 처리

def producer():
    for i in range(10):
        buffer.put(i)              # 가득 차면 자동으로 대기
        print(f"생산: {i}")

def consumer():
    while True:
        item = buffer.get()        # 비어 있으면 자동으로 대기
        print(f"소비: {item}")
        buffer.task_done()

threading.Thread(target=consumer, daemon=True).start()
producer()
buffer.join()
💡 표준 큐를 쓰자

파이썬 queue.Queue, 자바 BlockingQueue는 락·대기 조율을 내부에서 처리합니다. 직접 락으로 짜기 전에 검증된 자료구조가 있는지 먼저 찾으세요.

원자적 연산과 메모리 가시성

원자적(atomic) 연산은 "쪼갤 수 없이 한 번에" 끝나는 연산입니다 — 중간에 다른 스레드가 끼어들 수 없죠. 앞서 본 count += 1은 원자적이지 않아서 문제였습니다.

메모리 가시성(visibility)은 "한 스레드가 바꾼 값을 다른 스레드가 제때 보는가"입니다. CPU는 성능을 위해 값을 코어별 캐시에 들고 있어, 한쪽 변경이 다른 쪽에 곧바로 안 보일 수 있습니다. 그래서 락이나 volatile(자바)/원자 타입 같은 장치가 필요합니다.

Atomic.java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger count = new AtomicInteger(0);

// 락 없이도 원자적으로 1 증가 — 하드웨어가 보장
count.incrementAndGet();

// volatile: 변경을 모든 스레드가 즉시 보도록(가시성 보장)
volatile boolean running = true;

스레드풀과 비동기

스레드를 매번 새로 만들고 없애는 건 비쌉니다. 스레드풀(thread pool)은 미리 만들어 둔 스레드들을 재사용하며 작업만 갈아끼웁니다. 카페에서 손님마다 직원을 새로 뽑지 않고, 정해진 직원들이 주문을 처리하는 것과 같죠.

비동기(async)는 "결과를 기다리지 않고 다음 일을 진행"하는 방식입니다. 특히 파일·네트워크처럼 기다리는 시간이 긴 작업(I/O)에서, 기다리는 동안 CPU가 다른 일을 하게 해 효율을 높입니다.

pool.py
from concurrent.futures import ThreadPoolExecutor

def fetch(url):
    # ... 네트워크에서 가져오기(I/O 대기) ...
    return f"{url} 완료"

# 스레드 4개를 재사용하며 작업 분배
with ThreadPoolExecutor(max_workers=4) as pool:
    results = pool.map(fetch, ["a.com", "b.com", "c.com", "d.com"])

for r in results:
    print(r)
⚠️ 파이썬의 GIL

표준 파이썬(CPython)에는 GIL(전역 인터프리터 락)이 있어, 순수 계산 작업은 스레드를 여러 개 써도 동시에 진행되지 않습니다. 계산은 multiprocessing(여러 프로세스)으로, 네트워크·파일 대기는 스레드/비동기로 — 작업 성격에 맞게 고르세요.

액터 · 메시지 패싱 모델

공유 메모리에 락을 거는 방식은 까다롭고 실수가 잦습니다. 액터(actor) 모델은 발상을 뒤집습니다 — 메모리를 공유하지 않고, 메시지를 주고받아 협력합니다. 각 액터는 자기만의 데이터를 가지고, 한 번에 메시지 하나씩 처리하므로 경쟁 상태가 원천적으로 줄어듭니다.

"데이터를 공유해 소통하지 말고, 소통해서 데이터를 나누라"는 격언이 이 철학을 요약합니다. Erlang, Akka, Go의 채널(goroutine)이 대표 예입니다.

공유 메모리 없이 메시지만 주고받음 → 락 없이도 안전
방식핵심위험
공유 메모리 + 락같은 데이터를 락으로 보호경쟁 상태·교착상태
메시지 패싱(액터)데이터 격리 + 메시지 교환락 문제 적음, 설계 전환 필요
🎯 핵심 정리
  • 동시성은 "번갈아 다루는 구조", 병렬성은 "진짜 같은 순간에 실행".
  • 스레드는 메모리를 공유해 빠르지만, 공유 데이터가 사고의 원인이다.
  • 경쟁 상태는 실행 순서에 따라 결과가 달라지는, 재현 어려운 버그다.
  • 임계구역은 락(뮤텍스)으로 한 번에 한 스레드만 들어가게 보호한다.
  • 세마포어는 "N명까지" 허용하는 일반화된 락이다.
  • 교착상태는 4조건이 모두 성립할 때만 발생 — 하나만 깨도 예방된다.
  • 락이 까다로우면 메시지 패싱(액터)으로 데이터를 아예 공유하지 않는다.
  • 작업 성격(계산 vs 대기)에 맞게 프로세스·스레드·비동기를 고르라.
동시성 vs 병렬성
번갈아 진행하는 구조(동시성)와 같은 순간에 실행(병렬성)의 차이.
프로세스 / 스레드
독립 메모리의 실행 단위(프로세스)와 그 안에서 메모리를 공유하는 갈래(스레드).
경쟁 상태(Race Condition)
여러 스레드의 실행 순서에 따라 결과가 달라지는 버그.
임계구역(Critical Section)
공유 데이터를 다루는, 한 번에 한 스레드만 들어가야 하는 코드 영역.
락 / 뮤텍스
임계구역을 한 스레드만 쓰도록 잠그는 상호배제 도구.
세마포어(Semaphore)
동시에 접근 가능한 수를 N으로 제한하는 카운터 기반 동기화 장치.
교착상태(Deadlock)
서로의 자원을 기다리며 영원히 멈추는 상태. 4조건이 모두 필요.
원자적 연산 / 가시성
쪼갤 수 없는 연산(원자성)과 변경이 즉시 보이는 성질(가시성).
액터 모델
메모리를 공유하지 않고 메시지로 협력하는 동시성 모델.
