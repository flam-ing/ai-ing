컴파일러

사람이 쓴 코드를 컴퓨터가 실행할 수 있는 형태로 번역하는 프로그램입니다. 그 번역 과정을 단계별로 따라가 봅니다.

컴파일러란?

컴파일러(compiler)는 사람이 읽기 쉬운 고급 언어(예: C, 자바)로 쓴 소스 코드를, 컴퓨터가 곧장 실행할 수 있는 저급 언어(기계어·어셈블리)로 번역하는 프로그램입니다.

외국어 통역에 비유할 수 있습니다. 다만 동시통역(인터프리터)이 한 문장씩 즉석에서 옮긴다면, 컴파일러는 책 한 권 전체를 미리 번역해 두는 번역가에 가깝습니다. 미리 다 번역해 두니 실행은 빠르지만, 처음 번역(컴파일)에는 시간이 듭니다.

📌 이 페이지에서 배우는 것

컴파일의 전체 파이프라인과 그 안의 단계 — 어휘 분석 · 구문 분석 · 의미 분석 · 중간 코드와 최적화 · 코드 생성 — 을 차례로 살펴봅니다.

컴파일 단계 개요

컴파일러는 한 번에 번역하지 않고, 여러 단계를 거쳐 점점 더 컴퓨터에 가까운 형태로 코드를 바꿔 갑니다. 각 단계는 앞 단계의 결과물을 받아 다음 단계로 넘깁니다 — 마치 공장의 컨베이어 벨트처럼요.

소스 코드가 단계를 거쳐 점점 기계어에 가까워진다

흔히 앞쪽 단계(어휘·구문·의미 분석)를 프런트엔드(front-end), 뒤쪽 단계(최적화·코드 생성)를 백엔드(back-end)라고 부릅니다. 프런트엔드는 "언어를 이해하는" 일, 백엔드는 "기계어로 옮기는" 일을 맡습니다.

어휘 분석 (Lexical Analysis)

어휘 분석은 문자들의 줄줄이인 소스 코드를 의미 있는 최소 단위인 토큰(token)으로 잘게 쪼개는 단계입니다. 토큰은 문장의 "단어"에 해당합니다 — 키워드, 식별자(변수명), 숫자, 연산자, 괄호 등이죠. 이 작업을 하는 부분을 렉서(lexer) 또는 스캐너라고 합니다.

문자열 한 줄이 종류별 토큰들로 분해된다
lexer.py
import re

# (토큰 종류, 정규식) 규칙 목록
RULES = [
    ("NUMBER",  r"\d+"),
    ("ID",      r"[a-zA-Z_]\w*"),
    ("OP",      r"[+\-*/=]"),
    ("SEMI",    r";"),
    ("SKIP",    r"\s+"),     # 공백은 버림
]

def tokenize(code):
    tokens = []
    while code:
        for kind, pattern in RULES:
            m = re.match(pattern, code)
            if m:
                if kind != "SKIP":
                    tokens.append((kind, m.group()))
                code = code[m.end():]   # 매칭된 만큼 잘라냄
                break
    return tokens

print(tokenize("x = 10 + y;"))
# [('ID','x'), ('OP','='), ('NUMBER','10'), ('OP','+'), ('ID','y'), ('SEMI',';')]
💡 토큰이란 단어다

"The cat sleeps"를 단어 단위로 끊듯, 렉서는 코드를 토큰 단위로 끊습니다. 아직 문법이 맞는지는 따지지 않고, 단지 단어로 쪼갤 뿐입니다.

구문 분석 (Syntax Analysis)

구문 분석은 토큰들의 나열이 언어의 문법 규칙에 맞는지 확인하고, 그 구조를 트리 형태로 조립하는 단계입니다. 이 트리를 파스 트리 또는 더 간결한 형태로 다듬은 추상 구문 트리(AST, Abstract Syntax Tree)라고 부릅니다. 이 일을 하는 부분이 파서(parser)입니다.

단어들이 모여 문장의 "주어 + 동사 + 목적어" 구조를 이루듯, 토큰들이 모여 "대입문", "수식" 같은 구조를 이룹니다. AST는 그 문법 구조를 나무 모양으로 표현한 것입니다.

x = 10 + y 의 AST — 루트는 대입(=), 오른쪽 가지는 덧셈(+)
parser.py
# AST 노드는 보통 (연산, 왼쪽, 오른쪽) 형태의 트리로 표현한다
ast = (
    "assign",
    ("var", "x"),
    ("add",
        ("num", 10),
        ("var", "y"),
    ),
)

# 트리를 따라가며 사람이 읽을 수 있게 출력
def show(node, depth=0):
    pad = "  " * depth
    print(pad + node[0])
    for child in node[1:]:
        if isinstance(child, tuple):
            show(child, depth + 1)
        else:
            print(pad + "  " + str(child))

show(ast)
⚠️ 구문 오류

토큰이 문법에 맞지 않으면(예: x = = 10) 파서가 구문 오류(syntax error)를 냅니다. 컴파일이 실패하며 멈추는 가장 흔한 원인이 바로 이 단계의 오류입니다.

의미 분석 (Semantic Analysis)

구문이 맞다고 해서 코드가 말이 되는 것은 아닙니다. 의미 분석은 "문법은 맞지만 뜻이 어긋나는" 문제를 잡아냅니다. 영어로 치면 "Colorless green ideas sleep furiously" — 문법은 완벽하지만 의미가 이상한 문장을 걸러내는 것과 비슷합니다.

  • 타입 검사 — 숫자와 문자열을 더하려는 것처럼 맞지 않는 타입 연산을 잡음.
  • 선언 확인 — 정의하지 않은 변수·함수를 쓰는지 확인.
  • 범위(scope) 확인 — 그 변수를 그 위치에서 쓸 수 있는지 확인.

이때 모든 이름과 타입 정보를 모아 둔 표를 심볼 테이블(symbol table)이라 하며, 의미 분석의 핵심 도구입니다.

semantic_check.py
# 심볼 테이블: 이름 -> 타입
symbols = {"x": "int", "y": "int"}

def check_use(name):
    if name not in symbols:
        raise NameError(f"선언되지 않은 이름: {name}")  # 선언 확인
    return symbols[name]

def check_add(left_type, right_type):
    if left_type != right_type:                      # 타입 검사
        raise TypeError(f"타입 불일치: {left_type} + {right_type}")
    return left_type

# x = 10 + y  검사
check_use("x")
result_type = check_add("int", check_use("y"))       # int + int → OK
print("의미 검사 통과, 결과 타입:", result_type)
📌 정적 vs 동적

이렇게 실행 전에 컴파일러가 잡는 타입 검사를 정적 타입 검사라고 합니다(C, 자바). 반대로 파이썬·자바스크립트는 실행 중에 검사하는 동적 타입입니다.

중간 코드와 최적화

의미 분석을 통과하면, 컴파일러는 AST를 특정 CPU에 얽매이지 않은 중간 표현(IR, Intermediate Representation)으로 바꿉니다. IR은 "거의 기계어지만 아직 추상적인" 코드로, 여러 종류의 CPU에 공통으로 재사용할 수 있습니다.

이 IR 위에서 최적화(optimization)가 일어납니다. 같은 일을 더 적은 연산으로, 더 빠르게 하도록 코드를 다듬는 작업입니다.

최적화 기법설명예시
상수 접기미리 계산 가능한 식은 컴파일 때 계산2 * 3 → 6
죽은 코드 제거절대 실행되지 않는 코드 삭제if False: 블록 제거
공통 부분식 제거같은 계산을 한 번만 수행a*b 두 번 → 한 번
반복문 외부 이동반복마다 같은 값이면 밖으로 빼냄루프 안 상수식 → 루프 밖
constant_folding.py
# 상수 접기: 양쪽이 모두 숫자면 미리 계산해 버린다
def fold(node):
    if node[0] == "add":
        left  = fold(node[1])
        right = fold(node[2])
        if left[0] == "num" and right[0] == "num":
            return ("num", left[1] + right[1])   # 컴파일 때 계산!
        return ("add", left, right)
    return node

before = ("add", ("num", 2), ("num", 3))
print(fold(before))   # ('num', 5) — 실행 전에 이미 계산됨

코드 생성 (Code Generation)

마지막 단계입니다. 최적화된 중간 코드를 실제 CPU가 실행할 수 있는 기계어 또는 어셈블리로 바꿉니다. 이때 변수들을 어떤 레지스터(CPU 내부의 초고속 저장 공간)에 둘지 정하는 레지스터 할당도 함께 이뤄집니다.

codegen.asm
; x = 10 + y  를 x86 어셈블리로 번역한 결과 (개념적 예시)
mov  eax, 10        ; 레지스터 eax에 10을 넣는다
add  eax, [y]       ; eax에 변수 y의 값을 더한다
mov  [x], eax       ; 그 결과를 변수 x에 저장한다

이렇게 생성된 코드는 보통 어셈블러가 진짜 0과 1로 된 기계어로 바꾸고, 링커(linker)가 여러 조각을 합쳐 하나의 실행 파일로 만듭니다.

💡 컴파일러 vs 인터프리터

컴파일러는 전체를 미리 기계어로 번역해 빠른 실행 파일을 만듭니다. 인터프리터는 번역 없이 한 줄씩 즉석 실행해 느리지만 유연합니다. 파이썬·자바는 둘을 섞은 바이트코드 + 가상 머신 방식을 씁니다.

정규표현식 → NFA → DFA

어휘 분석기는 "이 문자열이 숫자인가, 식별자인가"를 빠르게 판별해야 합니다. 그 규칙은 정규표현식(regular expression)으로 적고, 컴파일러 도구는 이를 유한 오토마타(finite automaton)라는 "상태 기계"로 바꿔 한 글자씩 읽으며 상태를 옮겨 다닙니다. 오토마타에는 두 종류가 있습니다.

  • NFA(비결정 유한 오토마타) — 한 글자에 여러 갈림길이 허용되고, 글자를 읽지 않고도 건너뛰는 ε(엡실론) 간선이 있습니다. 만들기는 쉽지만 실행이 복잡합니다.
  • DFA(결정 유한 오토마타) — 각 상태에서 한 글자당 갈 곳이 딱 하나로 정해집니다. ε 간선이 없어 실행이 단순하고 빠릅니다.

그래서 보통 정규식 → NFA(톰슨 구성) → DFA(부분집합 구성) 순서로 변환합니다. NFA는 만들기 쉽고, DFA는 돌리기 빠르니, 쉬운 걸로 만든 뒤 빠른 걸로 바꾸는 셈입니다.

같은 규칙 a(b|c)* — NFA는 ε 갈림길이 많지만, DFA는 상태 2개로 단순해진다 (초록=수락 상태)
dfa.py
# DFA 로 a(b|c)* 인식하기 — 상태 전이표(transition table)
# 상태 A=시작, B=수락. 전이가 없으면 거부.
transitions = {
    ("A", "a"): "B",
    ("B", "b"): "B",     # b 를 읽어도 B 에 머문다(반복)
    ("B", "c"): "B",     # c 를 읽어도 B 에 머문다
}
ACCEPT = {"B"}

def matches(s):
    state = "A"
    for ch in s:
        key = (state, ch)
        if key not in transitions:   # 갈 곳이 없으면 거부
            return False
        state = transitions[key]     # 한 글자당 딱 한 곳으로 이동
    return state in ACCEPT           # 끝났을 때 수락 상태면 성공

print(matches("abcbc"))   # True
print(matches("acc"))     # True
print(matches("bca"))     # False — a 로 시작하지 않음
💡 왜 둘 다 배우나

NFA는 정규식에서 기계적으로 만들기 쉽고, DFA는 글자당 분기가 하나라 실행이 빠릅니다. lex·flex 같은 렉서 생성 도구가 바로 이 변환을 자동으로 해 줍니다.

문법 표기: BNF와 문맥 자유 문법

파서가 "문법에 맞는가"를 판단하려면, 먼저 그 문법 자체를 형식적으로 적어 두어야 합니다. 프로그래밍 언어의 문법은 대부분 문맥 자유 문법(CFG, Context-Free Grammar)으로 정의하고, 이를 적는 표기법이 BNF(Backus-Naur Form)입니다.

CFG는 몇 가지 규칙(생성 규칙, production)으로 이루어집니다. 왼쪽의 이름(비단말, non-terminal)을 오른쪽의 조합으로 "바꿔 쓸 수 있다"는 뜻입니다. 실제 글자(단말, terminal)만 남을 때까지 펼치면 하나의 문장이 됩니다.

grammar.bnf
# 간단한 사칙연산 문법 (BNF 표기)
# ::= 는 "다음과 같이 정의된다", | 는 "또는"

<expr>   ::= <expr> "+" <term>     ; 식 = 식 + 항
           | <term>                  ;     또는 그냥 항

<term>   ::= <term> "*" <factor>   ; 항 = 항 * 인수
           | <factor>

<factor> ::= "(" <expr> ")"        ; 괄호로 묶인 식
           | <number>               ;     또는 숫자

<number> ::= "0" | "1" | "2" | ... | "9"

이렇게 적으면 "2 + 3 * 4"가 왜 14인지(곱셈이 먼저인지) 문법 자체에 담깁니다. *가 +보다 트리의 아래쪽(먼저 계산)에 놓이도록 규칙을 계층으로 나눴기 때문입니다. 이것을 연산자 우선순위(precedence)라고 합니다.

📌 BNF·EBNF·CFG 관계

CFG는 "개념", BNF는 그 CFG를 적는 "표기법"입니다. 반복·선택을 더 간결히 쓰도록 확장한 것이 EBNF로, { }(0회 이상 반복), [ ](선택)을 추가로 씁니다.

파싱 기법: 재귀 하강·LL·LR

문법을 정의했으면, 토큰 나열을 받아 트리로 조립하는 파싱 알고리즘이 필요합니다. 방향에 따라 크게 두 갈래로 나뉩니다.

  • 하향식(top-down) — 루트(시작 기호)에서 출발해 잎(토큰)으로 펼쳐 내려갑니다. 재귀 하강 파서와 LL 파서가 여기 속합니다. 사람이 손으로 짜기 쉽습니다.
  • 상향식(bottom-up) — 잎(토큰)에서 출발해 묶어 가며 루트로 합쳐 올라갑니다. LR 파서 계열로, 더 넓은 범위의 문법을 다루지만 보통 도구가 자동 생성합니다.
하향식은 루트에서 펼치고(LL), 상향식은 잎에서 합쳐 올라간다(LR)
recursive_descent.py
# 재귀 하강 파서: 문법 규칙 하나당 함수 하나
# 문법:  expr = term ("+" term)* ,  term = NUMBER
tokens = ["2", "+", "3", "+", "4"]
pos = 0

def peek():  return tokens[pos] if pos < len(tokens) else None
def eat(t):
    global pos
    assert peek() == t, f"기대: {t}, 실제: {peek()}"
    pos += 1

def term():                       # term = NUMBER
    val = int(peek()); eat(peek())
    return val

def expr():                       # expr = term ("+" term)*
    result = term()
    while peek() == "+":          # 더하기가 이어지는 동안 반복
        eat("+")
        result += term()
    return result

print(expr())   # 9  — 2 + 3 + 4
기법방향특징대표 도구
재귀 하강하향식손으로 짜기 쉬움, 직관적직접 구현
LL(1)하향식토큰 1개만 미리 봄(lookahead)ANTLR
LR / LALR상향식더 넓은 문법, 자동 생성yacc, Bison
⚠️ 좌재귀 주의

재귀 하강·LL 파서는 expr ::= expr "+" term 처럼 왼쪽이 자기 자신으로 시작하는 좌재귀(left recursion)를 만나면 무한 호출에 빠집니다. 그래서 위 코드처럼 반복(loop) 형태로 문법을 바꿔 씁니다. 반면 LR 파서는 좌재귀를 자연스럽게 처리합니다.

심볼 테이블과 범위(Scope)

의미 분석에서 잠깐 등장한 심볼 테이블을 더 깊이 봅니다. 심볼 테이블은 코드에 등장하는 모든 이름(변수·함수·타입)에 대해 타입·범위·메모리 위치 같은 정보를 모아 둔 표입니다. 컴파일러가 "이 이름이 뭔지" 물을 때마다 찾아보는 사전인 셈입니다.

핵심은 범위(scope)입니다. 같은 이름 x라도 함수 안과 밖에서 서로 다른 변수일 수 있습니다. 그래서 심볼 테이블은 보통 블록마다 한 층씩 쌓는 스택 구조로 만들고, 이름을 찾을 때 안쪽 범위부터 바깥으로 거슬러 올라가며 검색합니다.

안쪽 블록일수록 스택 위에 — 이름은 위에서 아래로 거슬러 찾는다
symbol_table.py
class SymbolTable:
    def __init__(self):
        self.scopes = [{}]            # 범위들의 스택, [0]=전역

    def enter_scope(self):            # 블록에 들어가면 새 층 push
        self.scopes.append({})

    def exit_scope(self):             # 블록을 나가면 층 pop
        self.scopes.pop()

    def declare(self, name, type_):   # 현재(가장 안쪽) 범위에 선언
        self.scopes[-1][name] = type_

    def lookup(self, name):           # 안쪽 → 바깥쪽으로 거슬러 검색
        for scope in reversed(self.scopes):
            if name in scope:
                return scope[name]
        raise NameError(f"선언되지 않은 이름: {name}")

st = SymbolTable()
st.declare("g", "int")        # 전역 g
st.enter_scope()              # main 진입
st.declare("a", "int")
print(st.lookup("g"))         # int — 바깥 범위에서 찾음
print(st.lookup("a"))         # int — 현재 범위에서 찾음

타입 검사 자세히 보기

타입 검사(type checking)는 의미 분석의 가장 큰 일감입니다. AST를 따라 내려가며 각 식(expression)의 타입을 계산하고, 연산이 그 타입에 맞는지 확인합니다. 이렇게 "이 식의 타입은 무엇인가"를 알아내는 과정을 타입 추론(type inference)이라 합니다.

규칙은 보통 "추론 규칙" 형태로 적습니다. 예: 왼쪽이 int이고 오른쪽도 int이면, 왼쪽 + 오른쪽의 타입은 int다. 트리의 잎(상수·변수)에서 타입을 정하고, 위로 올라가며 합성합니다.

잎의 타입(int)을 합성해 위로 올림 — 최종 식의 타입은 int
type_check.py
# AST 를 따라 내려가며 각 노드의 타입을 추론한다
def type_of(node, env):
    kind = node[0]
    if kind == "num":
        return "int"                      # 숫자 리터럴은 int
    if kind == "str":
        return "str"
    if kind == "var":
        return env[node[1]]               # 심볼 테이블에서 조회
    if kind in ("add", "mul"):
        lt = type_of(node[1], env)
        rt = type_of(node[2], env)
        if lt != rt:                      # 양쪽 타입이 다르면 오류
            raise TypeError(f"{kind}: {lt} 와 {rt} 는 못 섞음")
        return lt                         # int+int=int, str+str=str
    raise ValueError(kind)

env = {"x": "int"}
ok  = ("add", ("num", 1), ("mul", ("num", 2), ("num", 3)))
print(type_of(ok, env))    # int

bad = ("add", ("num", 1), ("str", "hi"))
# type_of(bad, env)        # TypeError: add: int 와 str 는 못 섞음
📌 타입 검사가 막아 주는 것

실행하기도 전에 "숫자에 문자열 더하기", "없는 필드 접근" 같은 실수를 잡습니다. 그래서 강한 정적 타입 언어(러스트·타입스크립트 등)는 버그를 컴파일 단계에서 미리 걸러 낼 수 있습니다.

최적화 기법 깊이 보기

앞에서 표로 훑은 최적화를, IR을 직접 변형하는 관점에서 다시 봅니다. 최적화의 황금률은 "프로그램의 결과는 절대 바꾸지 않으면서, 더 적게·더 빠르게"입니다. 대표적인 세 가지를 코드로 따라가 봅니다.

결과는 그대로, 연산 수와 코드 줄은 줄어든다
optimizations.py
# IR 을 (연산, 대상, 인자들) 명령어 목록이라 가정한다
ir = [
    ("=", "a", ("mul", 2, 3)),   # a = 2 * 3
    ("=", "b", ("add", 1, 0)),   # b = 1 + 0  (항등원)
    ("dead", None, None),        # 절대 실행 안 됨
    ("=", "c", ("add", "a", "b")),
    ("=", "d", ("add", "a", "b")),  # c 와 같은 계산
]

def constant_fold(expr):                         # 상수 폴딩
    op, x, y = expr
    if isinstance(x, int) and isinstance(y, int):
        return {"add": x + y, "mul": x * y}[op]  # 미리 계산
    if op == "add" and y == 0:                   # x + 0 → x
        return x
    return expr

seen, out = {}, []
for op, dst, expr in ir:
    if op == "dead":                             # 죽은 코드 제거
        continue
    folded = constant_fold(expr)
    key = (folded if not isinstance(folded, tuple) else folded)
    if key in seen:                              # 공통 부분식 제거
        out.append(("=", dst, seen[key]))        # 이전 결과 재사용
    else:
        seen[key] = dst
        out.append(("=", dst, folded))

for line in out: print(line)
# ('=', 'a', 6)            상수 폴딩
# ('=', 'b', 1)            x+0 단순화
# ('=', 'c', ('add','a','b'))
# ('=', 'd', 'c')          공통식 제거 → c 재사용
⚠️ 최적화의 함정

잘못된 최적화는 결과를 바꿔 버그를 만듭니다. 특히 부동소수점 연산은 순서만 바꿔도 값이 달라질 수 있어, 컴파일러는 "안전하다고 증명된 변형만" 적용합니다. 그래서 -O2 같은 최적화 단계가 여러 개로 나뉩니다.

링킹과 로딩

코드 생성으로 끝이 아닙니다. 큰 프로그램은 여러 소스 파일로 나뉘어 각각 오브젝트 파일(.o)로 컴파일됩니다. 이 조각들을 하나로 합치고, printf 같은 라이브러리 함수의 실제 위치를 채워 넣어 실행 파일을 만드는 단계가 링킹(linking)입니다.

여러 .o 와 라이브러리를 링커가 합쳐 실행 파일로, 로더가 메모리에 올린다

링킹에는 두 가지 방식이 있습니다.

방식합치는 시점장점단점
정적 링킹빌드할 때 실행 파일에 포함혼자 실행 가능, 빠름파일이 크고, 라이브러리 갱신 어려움
동적 링킹실행할 때 .so/.dll 로딩파일 작고 라이브러리 공유실행 환경에 그 라이브러리 필요

마지막으로 로더(loader)가 실행 파일을 메모리에 올리고, 코드·데이터·스택 영역을 배치한 뒤 시작 주소로 점프하면 비로소 프로그램이 실행됩니다.

💡 "undefined reference" 오류의 정체

코드는 잘 컴파일됐는데 링크에서 나는 undefined reference to 'foo' 오류는, 호출은 했지만 foo의 실제 구현(.o나 라이브러리)을 링커가 못 찾았다는 뜻입니다. 컴파일 오류가 아니라 링킹 단계의 오류입니다.

🎯 핵심 정리
  • 컴파일러는 고급 언어를 기계어로 미리 번역하는 프로그램이다.
  • 파이프라인: 어휘 분석 → 구문 분석 → 의미 분석 → 최적화 → 코드 생성.
  • 어휘 분석은 코드를 토큰(단어)으로, 구문 분석은 토큰을 AST(문법 트리)로 만든다.
  • 의미 분석은 타입·선언 등 "뜻이 맞는지"를 검사한다.
  • 최적화는 같은 일을 더 빠르게, 코드 생성은 IR을 기계어로 옮긴다.
  • 정규식은 NFA로, NFA는 다시 DFA로 바꿔 빠른 렉서를 만든다.
  • 언어 문법은 CFG로 정의하고 BNF/EBNF 표기로 적는다.
  • 파싱은 하향식(재귀 하강·LL)과 상향식(LR)으로 나뉜다.
  • 심볼 테이블은 범위를 스택으로 쌓아 이름을 안쪽부터 찾는다.
  • 타입 검사는 잎의 타입을 위로 합성하며 맞지 않는 연산을 잡는다.
  • 링킹은 여러 .o와 라이브러리를 합쳐 실행 파일을 만든다(정적·동적).
토큰(Token)
소스 코드를 쪼갠 의미 있는 최소 단위(키워드·식별자·숫자 등).
AST
추상 구문 트리. 코드의 문법 구조를 나무 모양으로 표현한 것.
심볼 테이블
변수·함수의 이름과 타입·범위 정보를 모아 둔 표.
IR(중간 표현)
특정 CPU에 얽매이지 않은, 최적화에 쓰이는 중간 단계 코드.
레지스터
CPU 내부의 가장 빠른 소량 저장 공간. 코드 생성 시 변수를 배치.
NFA / DFA
유한 오토마타(상태 기계). NFA는 ε·갈림길 허용, DFA는 글자당 갈 곳이 하나로 빠름.
CFG / BNF
문맥 자유 문법과 그것을 적는 표기법. 언어의 문법 규칙을 형식적으로 정의.
재귀 하강 파서
문법 규칙마다 함수를 둔 하향식 파서. 손으로 짜기 쉽다.
LL / LR 파싱
각각 하향식·상향식 파싱 기법. LR이 더 넓은 문법을 다룬다.
범위(Scope)
이름이 유효한 코드 영역. 심볼 테이블은 범위를 스택으로 관리한다.
타입 추론
식의 타입을 잎에서부터 합성해 계산하는 과정.
링킹(Linking)
여러 오브젝트 파일·라이브러리를 합쳐 실행 파일을 만드는 단계(정적·동적).
