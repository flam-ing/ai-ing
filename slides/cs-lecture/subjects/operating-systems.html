운영체제

컴퓨터의 보이지 않는 관리자. 한정된 CPU와 메모리를 여러 프로그램이 사이좋게 나눠 쓰도록 조율하는 법을 배웁니다.

운영체제란?

운영체제(OS, Operating System)는 하드웨어(CPU·메모리·디스크)와 응용 프로그램 사이에서 자원을 관리하고 중재하는 소프트웨어입니다. 윈도우, macOS, 리눅스, 안드로이드가 모두 운영체제입니다.

비유하면 운영체제는 큰 식당의 매니저입니다. 손님(프로그램)이 자리(메모리)와 요리사(CPU)를 직접 차지하려 다투지 않도록, 매니저가 순서를 정하고 자리를 배정합니다. 덕분에 프로그램은 하드웨어를 직접 다룰 필요 없이 매니저에게 요청만 하면 됩니다.

📌 이 페이지에서 배우는 것

프로세스와 스레드 · CPU 스케줄링 · 가상 메모리와 페이징 · 동기화와 교착상태 · 파일 시스템까지, 운영체제의 핵심 다섯 기둥을 다룹니다.

프로세스와 스레드

프로세스(process)는 "실행 중인 프로그램"입니다. 디스크에 잠자던 프로그램 파일이 메모리에 올라와 CPU를 받아 돌아가는 순간, 그것이 프로세스입니다. 각 프로세스는 자신만의 독립된 메모리 공간(코드·데이터·스택·힙)을 가집니다.

스레드(thread)는 프로세스 안에서 실제로 일하는 실행 흐름입니다. 한 프로세스 안의 여러 스레드는 같은 메모리를 공유하면서 동시에 여러 작업을 처리합니다. 비유하면 프로세스는 사무실(공간), 스레드는 그 안에서 일하는 직원들입니다.

프로세스는 메모리를 격리하고, 스레드는 프로세스 안에서 메모리를 공유
threads.py
import threading

def work(name):
    print(f"{name} 스레드가 일하는 중")

# 한 프로세스 안에서 두 스레드가 동시에 실행
t1 = threading.Thread(target=work, args=("A",))
t2 = threading.Thread(target=work, args=("B",))
t1.start(); t2.start()
t1.join();  t2.join()       # 두 스레드가 끝나길 기다림
💡 스레드가 가벼운 이유

새 프로세스를 만들면 메모리 공간을 통째로 복사해야 해 무겁습니다. 스레드는 같은 공간을 공유하므로 생성·전환 비용이 훨씬 작습니다. 대신 메모리를 공유하니 동기화 문제가 따라옵니다.

프로세스 상태

프로세스는 태어나서 사라질 때까지 여러 상태(state)를 오갑니다. CPU는 하나뿐이고 프로세스는 여럿이라, 운영체제는 프로세스를 끊임없이 "실행 → 대기 → 준비" 사이로 옮기며 돌립니다.

  • 준비(Ready) — CPU만 받으면 바로 실행할 수 있는, 줄 서서 기다리는 상태.
  • 실행(Running) — 지금 CPU를 차지하고 명령을 처리 중인 상태.
  • 대기(Waiting) — 디스크 읽기 같은 입출력이 끝나길 기다리며 멈춘 상태.
CPU를 받으면 실행, 시간을 다 쓰면 준비로, 입출력이 필요하면 대기로
📌 문맥 교환(Context Switch)

한 프로세스를 멈추고 다른 프로세스로 CPU를 넘길 때, 운영체제는 현재 프로세스의 상태(레지스터·프로그램 카운터)를 저장하고 다음 프로세스의 상태를 복원합니다. 이 과정을 문맥 교환이라 하며, 자주 일어나면 그 자체가 비용(오버헤드)이 됩니다.

CPU 스케줄링

준비 상태로 줄 서 있는 여러 프로세스 중 누구에게 CPU를 줄지 정하는 규칙이 CPU 스케줄링입니다. 어떤 규칙을 쓰느냐에 따라 응답 속도와 공정성이 달라집니다.

방식규칙특징
FCFS먼저 온 순서대로단순하지만 긴 작업이 앞서면 모두 지연
SJF실행 시간이 짧은 것부터평균 대기시간 최소, 긴 작업은 굶을 수 있음
Round Robin일정 시간씩 돌아가며공정함, 응답성이 좋아 대화형에 적합
Priority우선순위 높은 것부터중요 작업 우선, 낮은 것은 굶을 위험
각 프로세스가 정해진 시간만큼 번갈아 CPU를 사용 → 아무도 오래 굶지 않음
round_robin.py
from collections import deque

def round_robin(jobs, quantum):
    queue = deque(jobs)              # (이름, 남은 시간)
    t = 0
    while queue:
        name, remain = queue.popleft()
        run = min(quantum, remain)  # 슬라이스만큼만 실행
        t += run
        print(f"{t}ms: {name} 실행")
        if remain - run > 0:        # 남았으면 다시 줄 끝으로
            queue.append((name, remain - run))

round_robin([("P1", 6), ("P2", 5), ("P3", 4)], quantum=4)

메모리 관리: 가상 메모리와 페이징

프로그램은 메모리가 자신만을 위해 통째로 존재한다고 믿지만, 실제 물리 메모리(RAM)는 작고 여러 프로세스가 나눠 씁니다. 이 간극을 메우는 마법이 가상 메모리(virtual memory)입니다.

운영체제는 메모리를 페이지(page)라는 같은 크기의 조각으로 나눕니다. 프로그램이 보는 가상 주소는 페이지 테이블(page table)을 통해 실제 물리 주소(프레임)로 변환됩니다. 덕분에 프로그램은 흩어진 물리 메모리를 연속된 것처럼 사용할 수 있고, 자주 쓰지 않는 페이지는 디스크로 잠시 내보낼 수도 있습니다.

가상 페이지 → 페이지 테이블 → 흩어진 물리 프레임으로 변환
⚠️ 페이지 폴트와 스왑

필요한 페이지가 RAM에 없으면 페이지 폴트(page fault)가 발생해 디스크에서 가져옵니다. 메모리가 부족해 디스크와의 교체(swap)가 과도하게 일어나면 시스템이 거의 멈추는 스래싱(thrashing) 상태가 됩니다.

동기화: 임계구역과 교착상태

여러 스레드가 같은 데이터를 동시에 고치면 결과가 엉킵니다. 이런 위험한 코드 구간을 임계구역(critical section)이라 하고, 한 번에 하나만 들어가도록 막는 장치가 락(lock)·뮤텍스(mutex)입니다. 화장실 한 칸을 열쇠로 잠그는 것과 같습니다.

lock.py
import threading

balance = 0
lock = threading.Lock()

def deposit():
    global balance
    for _ in range(100000):
        with lock:              # 임계구역: 한 번에 한 스레드만
            balance += 1        # 락이 없으면 값이 엉킴(경쟁 상태)

t1 = threading.Thread(target=deposit)
t2 = threading.Thread(target=deposit)
t1.start(); t2.start(); t1.join(); t2.join()
print(balance)                  # 락 덕분에 정확히 200000

그런데 락을 잘못 쓰면 교착상태(deadlock)가 생깁니다. 서로 상대가 가진 자원을 기다리며 둘 다 영원히 멈추는 상황입니다. A는 자원1을 쥔 채 자원2를 기다리고, B는 자원2를 쥔 채 자원1을 기다리면 둘 다 못 움직입니다.

서로가 쥔 자원을 기다리며 둘 다 멈춤 — 순환 대기가 교착의 핵심
🚫 교착상태 4대 조건

상호 배제 · 점유와 대기 · 비선점 · 순환 대기가 모두 성립할 때만 교착이 일어납니다. 네 조건 중 하나라도 깨면(예: 항상 같은 순서로 락을 잡기) 교착을 예방할 수 있습니다.

파일 시스템

디스크는 그냥 0과 1을 담는 거대한 창고일 뿐입니다. 이 창고를 파일과 폴더(디렉터리)라는 사람이 이해할 수 있는 단위로 정리해 주는 것이 파일 시스템(file system)입니다. NTFS, ext4, APFS가 대표적입니다.

파일 시스템은 파일의 이름·크기·권한·위치 같은 정보(메타데이터)를 관리하고, 파일이 디스크의 어느 블록에 흩어져 있는지 추적합니다. 사용자는 /home/user/note.txt 같은 경로만 알면 되고, 실제 블록 위치는 신경 쓸 필요가 없습니다.

루트(/)에서 가지치는 디렉터리 트리 — 경로는 루트부터의 길
files.py
# open() 은 운영체제에 "이 파일 열어줘"라고 요청하는 시스템 콜
with open("/home/user/note.txt", "w") as f:
    f.write("안녕하세요")     # 운영체제가 디스크 블록에 기록

with open("/home/user/note.txt", "r") as f:
    print(f.read())            # 경로만 알면 블록 위치는 몰라도 됨
💡 시스템 콜(system call)

프로그램이 파일을 열거나 메모리를 요청할 때, 직접 하드웨어를 만지지 않고 운영체제에 부탁하는 공식 창구가 시스템 콜입니다. open, read, fork 등이 모두 시스템 콜입니다.

컨텍스트 스위칭 상세

앞에서 문맥 교환(컨텍스트 스위칭)을 짧게 소개했습니다. 여기서는 "그 순간 운영체제가 실제로 무엇을 저장하고 복원하는지"를 자세히 들여다봅니다. CPU는 한 번에 하나의 흐름만 실행할 수 있으므로, 다른 프로세스로 바꿔 끼우려면 지금 프로세스의 작업 상태를 통째로 사진 찍어 보관한 뒤 다음 프로세스의 사진을 꺼내 복원해야 합니다. 이 사진첩이 PCB(Process Control Block)입니다.

저장 대상은 프로그램 카운터(다음에 실행할 명령 주소), 레지스터 값들, 스택 포인터, 메모리 매핑 정보 등입니다. 게임을 잠시 멈추고(세이브) 다른 게임을 이어서 했다가(로드), 다시 원래 게임을 세이브 지점부터 재개하는 것과 똑같습니다.

A의 상태를 PCB에 저장 → B의 PCB를 복원. 그 사이 CPU는 실제 일을 못 한다
pcb_concept.py
# PCB는 프로세스의 "상태 사진" — 개념을 파이썬으로 흉내
class PCB:
    def __init__(self, pid):
        self.pid = pid
        self.program_counter = 0   # 다음 실행할 명령 위치
        self.registers = {}        # CPU 레지스터 값들
        self.state = "ready"

def context_switch(old, new):
    old.state = "ready"            # ① 현재 프로세스 상태 저장
    save_registers(old)            #    레지스터·PC를 PCB에 보관
    restore_registers(new)         # ② 다음 프로세스 상태 복원
    new.state = "running"          #    저장해 둔 지점부터 이어서 실행
    return new
⚠️ 스위칭은 공짜가 아니다

컨텍스트 스위칭이 일어나는 동안 CPU는 실제 사용자 작업을 전혀 못 합니다(순수 오버헤드). 또 새 프로세스로 바뀌면 캐시에 쌓아둔 데이터가 무용지물이 되어(캐시 미스) 추가 비용이 듭니다. 그래서 타임 슬라이스를 너무 짧게 잡으면 스위칭만 하다 끝납니다.

페이지 교체 알고리즘 (FIFO·LRU·최적)

물리 메모리(프레임)는 한정적인데 필요한 페이지는 더 많을 때, 새 페이지를 들이려면 기존 페이지 하나를 내보내야 합니다. "누구를 내보낼까?"를 정하는 규칙이 페이지 교체 알고리즘입니다. 잘못 고르면 방금 내보낸 페이지를 곧바로 다시 불러와(페이지 폴트) 성능이 추락합니다.

  • FIFO — 가장 먼저 들어온 페이지를 내보냄. 구현은 쉽지만, 자주 쓰는 페이지도 오래됐다는 이유로 쫓아낼 수 있음.
  • LRU(Least Recently Used) — 가장 오랫동안 안 쓴 페이지를 내보냄. "최근에 쓴 건 또 쓸 것"이라는 직관이 잘 맞아 실무에서 널리 쓰임.
  • 최적(OPT) — 앞으로 가장 늦게 쓸 페이지를 내보냄. 미래를 알아야 해서 현실엔 불가능, 다른 알고리즘의 성능을 재는 기준선으로만 사용.
빨강은 페이지 폴트(교체 발생), 초록은 적중(이미 있음) — 가장 오래된 것부터 빠져나간다
lru.py
from collections import OrderedDict

def lru_faults(refs, capacity):
    frames = OrderedDict()      # 최근 쓴 순서를 기억
    faults = 0
    for page in refs:
        if page in frames:
            frames.move_to_end(page)   # 적중 → 최근 사용으로 갱신
        else:
            faults += 1                # 폴트 발생
            if len(frames) >= capacity:
                frames.popitem(last=False)  # 가장 오래된 것 제거
            frames[page] = True
    return faults

print(lru_faults([7,0,1,2,0,3,0,4], capacity=3))  # 폴트 횟수
📌 벨라디의 역설

직관적으로는 프레임(칸)을 늘리면 폴트가 줄어야 합니다. 그런데 FIFO에서는 프레임을 늘렸더니 오히려 폴트가 늘어나는 기현상이 생길 수 있습니다(벨라디의 역설). LRU·최적 알고리즘에서는 이런 일이 없어, FIFO의 약점을 보여 줍니다.

세마포어 vs 뮤텍스 상세

앞서 락(뮤텍스)으로 임계구역을 보호하는 법을 봤습니다. 비슷해 보이는 세마포어(semaphore)와 무엇이 다른지 명확히 구분해 봅시다. 한 줄 요약: 뮤텍스는 "화장실 한 칸의 열쇠"(딱 하나, 잠근 사람만 풀 수 있음), 세마포어는 "주차장 빈자리 표지판"(여러 자리, 카운터로 관리)입니다.

구분뮤텍스(Mutex)세마포어(Semaphore)
허용 개수1개(이진)N개(카운팅 가능)
소유 개념잠근 스레드만 해제 가능소유자 없음, 누구나 신호 가능
핵심 용도상호 배제(한 명만 진입)자원 개수 제한·신호 전달
연산lock / unlockP(wait) / V(signal)
semaphore.py
import threading, time

# 동시에 최대 2명만 접속 가능한 자원 (예: DB 커넥션 2개)
sem = threading.Semaphore(2)      # 카운터 = 2

def access(name):
    with sem:                     # P 연산: 카운터 -1 (0이면 대기)
        print(f"{name} 접속 중")
        time.sleep(1)
    # with 블록을 나가며 V 연산: 카운터 +1 (대기자 깨움)

for i in range(5):                # 5명이 몰려도 동시엔 2명까지
    threading.Thread(target=access, args=(f"손님{i}",)).start()
💡 이진 세마포어 ≠ 뮤텍스

카운터가 1인 세마포어(이진 세마포어)는 뮤텍스와 비슷해 보이지만, 소유 개념이 없다는 결정적 차이가 있습니다. 뮤텍스는 잠근 스레드만 풀 수 있어 안전하지만, 세마포어는 다른 스레드가 V를 호출해 풀 수도 있습니다. 상호 배제엔 뮤텍스, 신호·자원 카운팅엔 세마포어를 쓰세요.

생산자-소비자 문제

세마포어가 빛을 발하는 대표 문제입니다. 생산자(producer)는 데이터를 만들어 한정된 버퍼(창고)에 넣고, 소비자(consumer)는 그 버퍼에서 꺼내 씁니다. 핵심 난제는 두 가지입니다. 버퍼가 가득 찼는데 생산자가 더 넣으려 하거나, 버퍼가 비었는데 소비자가 꺼내려 하면 문제가 생깁니다.

버퍼를 사이에 두고 생산자와 소비자가 다른 속도로 일한다 — 세마포어로 빈칸·찬칸을 센다
producer_consumer.py
import threading, queue, time

buffer = queue.Queue(maxsize=3)   # 큐가 빈칸/찬칸 동기화를 알아서 처리

def producer():
    for i in range(5):
        buffer.put(i)             # 가득 차면 자리가 날 때까지 자동 대기
        print(f"생산: {i}")

def consumer():
    for _ in range(5):
        item = buffer.get()       # 비어 있으면 들어올 때까지 자동 대기
        print(f"   소비: {item}")
        time.sleep(0.5)           # 소비자가 더 느린 상황

threading.Thread(target=producer).start()
threading.Thread(target=consumer).start()
⚠️ 바쁜 대기(busy waiting)를 피하라

버퍼가 찰 때까지 while full: pass로 계속 확인하면 CPU를 헛돌립니다. 세마포어나 조건 변수를 쓰면 대기하는 동안 스레드를 재워 CPU를 낭비하지 않습니다. 위 예시의 queue.Queue는 이 동기화를 내부에서 처리합니다.

디스크 스케줄링 (FCFS·SCAN)

하드 디스크(HDD)는 물리적으로 회전하는 원판 위를 헤드(바늘)가 좌우로 움직여 데이터를 읽습니다. 헤드를 멀리 움직일수록 시간이 오래 걸리므로(탐색 시간), "여러 읽기 요청을 어떤 순서로 처리할지"가 성능을 좌우합니다. 이것이 디스크 스케줄링입니다.

  • FCFS — 요청이 온 순서대로. 공정하지만 헤드가 멀리 왔다 갔다 하며 비효율적일 수 있음.
  • SCAN(엘리베이터) — 한 방향으로 끝까지 가며 처리하고, 끝에 닿으면 방향을 바꿔 돌아옴. 엘리베이터가 위로 가며 층마다 서고 꼭대기에서 내려오는 것과 같음.
FCFS는 헤드가 양끝을 오가지만, SCAN은 한 방향으로 쓸며 이동해 탐색 거리를 줄인다
disk_scan.py
def scan(requests, head, disk_size=200):
    moves = sorted(requests)
    right = [r for r in moves if r >= head]   # 오른쪽 요청들
    left  = [r for r in moves if r < head][::-1]  # 왼쪽(역순)
    order = right + [disk_size - 1] + left     # 끝까지 갔다 되돌아옴
    distance, cur = 0, head
    for pos in order:
        distance += abs(pos - cur)             # 이동 거리 누적
        cur = pos
    return distance

print(scan([98, 183, 41, 122, 14, 124], head=53))  # 총 이동 거리
💡 SSD에는 해당 없음

이 알고리즘들은 물리적으로 헤드가 움직이는 HDD를 위한 것입니다. SSD는 움직이는 부품이 없어 어느 위치든 접근 시간이 거의 같으므로, 디스크 스케줄링의 중요성이 크게 줄어듭니다.

시스템 콜과 커널 모드

파일 시스템 절에서 시스템 콜을 잠깐 언급했습니다. 여기서는 "왜 시스템 콜이 필요한가"를 모드(mode) 개념으로 설명합니다. CPU는 두 가지 권한 수준에서 동작합니다. 사용자 모드(user mode)는 권한이 제한된 일반 상태, 커널 모드(kernel mode)는 하드웨어를 직접 만질 수 있는 전권 상태입니다.

일반 프로그램은 안전을 위해 사용자 모드에서만 돕니다. 디스크 읽기처럼 하드웨어가 필요한 일은 시스템 콜로 운영체제에 요청하면, CPU가 커널 모드로 전환되어 운영체제가 대신 처리하고, 끝나면 다시 사용자 모드로 돌아옵니다. 이 경계가 있어 한 프로그램의 실수가 시스템 전체를 망가뜨리지 못합니다.

시스템 콜이 일어나면 잠시 커널 모드로 넘어가 OS가 일하고, 끝나면 사용자 모드로 돌아온다
📌 모드 전환도 비용이다

사용자 모드 ↔ 커널 모드 전환에도 시간이 듭니다. 그래서 작은 데이터를 수없이 시스템 콜로 읽기보다, 한 번에 큰 덩어리로 읽는(버퍼링) 편이 빠릅니다. 파일을 한 줄씩 1만 번 읽는 것보다 통째로 읽어 메모리에서 처리하는 게 효율적인 이유입니다.

가상화와 컨테이너 입문

한 대의 물리 서버를 여러 개의 독립된 컴퓨터처럼 나눠 쓰는 기술이 가상화(virtualization)입니다. 크게 두 갈래가 있습니다. 가상 머신(VM)은 하드웨어 위에 하이퍼바이저를 두고 그 위에 운영체제를 통째로 여러 개 올립니다. 컨테이너(container)는 운영체제 커널을 공유하면서 실행 환경만 격리해, 훨씬 가볍고 빠릅니다.

VM은 게스트 OS를 통째로 올려 무겁고, 컨테이너는 커널을 공유해 가볍고 빠르게 뜬다
구분가상 머신(VM)컨테이너
격리 수준OS까지 완전 격리(강함)프로세스 수준 격리
무게/속도무겁다 · 부팅 수십 초가볍다 · 수 초 내 실행
OS각자 게스트 OS 보유호스트 커널 공유

💡 "제 컴퓨터에선 됐는데요"의 해결책

컨테이너는 앱과 그 실행 환경(라이브러리·설정)을 하나로 묶어 배포합니다. 그래서 개발자의 노트북, 테스트 서버, 운영 서버 어디서든 똑같이 동작합니다. 이것이 Docker가 현대 배포의 표준이 된 이유입니다. 컨테이너는 운영체제의 격리 기능(네임스페이스·cgroups) 위에서 동작합니다.

🎯 핵심 정리
  • 운영체제는 CPU·메모리·디스크를 여러 프로그램이 나눠 쓰도록 중재하는 관리자다.
  • 프로세스는 독립된 메모리 공간, 스레드는 그 안에서 메모리를 공유하는 실행 흐름이다.
  • CPU 스케줄링은 누구에게 CPU를 줄지 정하는 규칙이며, 라운드 로빈은 공정성이 강점이다.
  • 가상 메모리와 페이징은 작은 물리 메모리를 크고 연속된 것처럼 보이게 한다.
  • 임계구역은 락으로 보호하고, 순환 대기를 막으면 교착상태를 예방할 수 있다.
  • 컨텍스트 스위칭은 PCB에 상태를 저장·복원하는 과정이며 그 자체가 오버헤드다.
  • 페이지 교체는 LRU가 실무 표준, 최적(OPT)은 비교 기준선, FIFO는 단순하지만 약점이 있다.
  • 뮤텍스는 상호 배제(1개·소유 있음), 세마포어는 자원 카운팅·신호(N개·소유 없음)다.
  • 생산자-소비자 문제는 한정 버퍼를 세마포어/큐로 동기화해 푼다.
  • 디스크 스케줄링(SCAN 등)은 헤드 이동 거리를 줄이지만 SSD엔 영향이 작다.
  • 시스템 콜은 사용자 모드에서 커널 모드로 안전하게 넘어가 OS에 일을 맡기는 통로다.
  • 컨테이너는 커널을 공유해 VM보다 가볍고, 어디서든 똑같이 동작하는 배포를 만든다.

문맥 교환

페이징
메모리를 같은 크기 페이지로 나눠 가상 주소를 물리 주소로 변환하는 기법.
임계구역
한 번에 하나의 스레드만 들어가야 하는 공유 자원 접근 구간.

서로 상대의 자원을 기다리며 모두 멈추는 상황(순환 대기).

페이지 교체
프레임이 부족할 때 내보낼 페이지를 고르는 규칙(FIFO·LRU·OPT).
세마포어
카운터로 자원 개수를 제한하고 신호를 주고받는 동기화 도구(P/V 연산).
생산자-소비자

디스크 스케줄링

커널 모드
하드웨어를 직접 제어할 수 있는 CPU의 전권 상태(반대는 사용자 모드).
컨테이너
호스트 커널을 공유하며 실행 환경만 격리하는 가벼운 가상화 단위.
