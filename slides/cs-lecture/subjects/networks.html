컴퓨터 네트워크

멀리 떨어진 컴퓨터들이 어떻게 서로 대화하는가. 메시지 한 통이 지구 반대편 서버까지 닿는 여정을 따라갑니다.

네트워크란?

컴퓨터 네트워크(network)는 여러 컴퓨터가 데이터를 주고받도록 연결된 시스템입니다. 집 안의 와이파이부터 전 세계를 잇는 인터넷까지 모두 네트워크입니다.

비유하면 네트워크는 우편 시스템입니다. 편지(데이터)를 봉투에 담아(패킷) 주소(IP)를 적으면, 여러 우체국(라우터)을 거쳐 받는 사람에게 배달됩니다. 도중에 길이 막히면 다른 경로로 돌아가기도 합니다.

📌 이 페이지에서 배우는 것

OSI·TCP/IP 계층 모델 · IP 주소와 라우팅 · TCP와 UDP(3-way handshake) · HTTP와 DNS · 소켓 통신까지, 데이터가 흐르는 전 과정을 다룹니다.

계층 모델: OSI와 TCP/IP

네트워크는 복잡한 일을 계층(layer)으로 나눠 처리합니다. 각 층은 자기 일만 하고 위아래 층과 약속된 방식으로만 소통합니다. 덕분에 한 층을 바꿔도(예: 와이파이 → 이더넷) 다른 층은 그대로 둘 수 있습니다.

학문적 표준은 7계층 OSI 모델이고, 실제 인터넷이 쓰는 모델은 더 단순한 4계층 TCP/IP 모델입니다. 데이터는 보낼 때 위에서 아래로 내려가며 헤더가 붙고(캡슐화), 받을 때 아래에서 위로 올라가며 벗겨집니다.

IP 주소와 라우팅

IP 주소(IP address)는 네트워크상의 컴퓨터를 식별하는 주소입니다. 우편의 집 주소와 같습니다. 지금 널리 쓰는 IPv4는 192.168.0.1처럼 점으로 구분한 네 개의 숫자(각 0~255)로 이뤄집니다.

라우팅(routing)은 출발지에서 목적지까지 패킷이 갈 경로를 찾는 과정입니다. 라우터(router)는 패킷의 목적지 IP를 보고 "다음으로 어느 우체국에 넘길지"를 결정하며, 여러 라우터를 거쳐(hop) 목적지에 도달합니다.

하나의 직선이 아니라 라우터들을 건너뛰며 목적지에 도달
💡 IPv4와 IPv6

IPv4 주소는 약 43억 개뿐이라 고갈되고 있습니다. 그래서 사실상 무한대인 IPv6(2001:db8::1)가 도입되었고, 공인 IP 하나를 여러 기기가 나눠 쓰는 NAT 기술도 함께 쓰입니다.

TCP vs UDP

전송 계층에는 성격이 다른 두 프로토콜이 있습니다. TCP는 신뢰성을, UDP는 속도를 우선합니다.

  • TCP(Transmission Control Protocol) — 연결을 먼저 맺고, 순서를 보장하며, 빠진 데이터는 다시 보냅니다. 등기 우편처럼 확실하지만 느립니다. 웹·메일·파일 전송에 사용.
  • UDP(User Datagram Protocol) — 연결 없이 일단 보냅니다. 일부가 사라져도 신경 쓰지 않아 빠릅니다. 실시간 영상·게임·음성 통화에 사용.

TCP는 데이터를 주고받기 전에 3-way handshake로 연결을 엽니다. 전화 걸기 전 "여보세요? — 네 들려요 — 좋아요 시작합시다" 하고 확인하는 것과 같습니다.

HTTP와 DNS

HTTP(HyperText Transfer Protocol)는 웹 브라우저와 서버가 대화하는 약속입니다. 클라이언트가 요청(request)을 보내면 서버가 응답(response)을 돌려주는 단순한 구조입니다.

그런데 우리는 142.250.x.x 같은 IP 대신 google.com 같은 이름을 씁니다. 이 이름을 IP로 바꿔주는 인터넷의 전화번호부가 DNS(Domain Name System)입니다.

브라우저가 요청을 보내면 서버가 상태 코드와 함께 응답
http_request.py
import urllib.request

# 이름(URL)을 주면 내부적으로 DNS가 IP로 변환 → HTTP 요청
with urllib.request.urlopen("https://example.com") as res:
    print(res.status)          # 200  — 상태 코드
    print(res.headers["Content-Type"])  # text/html
    body = res.read()          # 응답 본문(HTML)
⚠️ HTTP 상태 코드

응답 첫 줄의 숫자는 결과를 알려줍니다. 2xx(성공), 3xx(리다이렉트), 4xx(클라이언트 잘못, 예: 404 없음), 5xx(서버 잘못, 예: 500 오류). 디버깅의 첫 단서입니다.

소켓 통신

소켓(socket)은 프로그램이 네트워크로 데이터를 주고받기 위해 여는 통신 창구입니다. IP 주소(어느 컴퓨터)와 포트 번호(그 안의 어느 프로그램)를 합쳐 통신 상대를 특정합니다. 아파트로 치면 IP는 건물 주소, 포트(port)는 호수입니다.

웹 서버, 채팅, 게임 서버 등 모든 네트워크 프로그램은 결국 소켓 위에서 동작합니다. 서버는 특정 포트에서 연결을 기다리고(listen), 클라이언트가 접속(connect)하면 데이터를 주고받습니다.

tcp_server.py
import socket

# TCP 소켓 생성 → 포트 9000에서 연결 대기
server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(("0.0.0.0", 9000))   # IP + 포트에 묶음
server.listen()                  # 연결 요청을 기다림

conn, addr = server.accept()     # 클라이언트 접속 수락
data = conn.recv(1024)           # 데이터 수신
conn.sendall(b"received: " + data)  # 응답 전송
conn.close()
tcp_client.py
import socket

client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client.connect(("127.0.0.1", 9000))  # 서버 IP + 포트로 접속
client.sendall(b"hello")             # 데이터 전송
print(client.recv(1024))             # b"received: hello"
client.close()
💡 잘 알려진 포트

포트 번호 중 일부는 용도가 약속돼 있습니다. 80(HTTP), 443(HTTPS), 22(SSH), 53(DNS). 브라우저에서 주소만 쳐도 자동으로 80/443 포트로 접속합니다.

서브넷과 CIDR

IP 주소는 사실 두 부분으로 나뉩니다. 네트워크 부분(어느 동네인가)과 호스트 부분(그 동네의 몇 번째 집인가)입니다. 큰 IP 덩어리를 작은 단위로 쪼개 관리하는 것이 서브넷팅(subnetting)이고, 어디까지가 네트워크 부분인지 알려주는 것이 서브넷 마스크입니다.

CIDR 표기법은 192.168.10.0/24처럼 슬래시 뒤 숫자로 "앞에서부터 몇 비트가 네트워크 부분인지"를 적습니다. /24는 앞 24비트가 네트워크, 나머지 8비트가 호스트라는 뜻이라, 한 네트워크에 약 254개의 주소를 쓸 수 있습니다.

/24 = 앞 3칸이 네트워크, 마지막 칸이 호스트. 같은 네트워크끼리는 라우터 없이 직접 통신
CIDR서브넷 마스크사용 가능 호스트 수
/24255.255.255.0약 254개
/25255.255.255.128약 126개
/26255.255.255.192약 62개
/30255.255.255.2522개(라우터 연결용)
subnet.py
import ipaddress

net = ipaddress.ip_network("192.168.10.0/24")
print(net.netmask)              # 255.255.255.0
print(net.num_addresses)        # 256 (호스트는 254개 사용 가능)

# 두 IP가 같은 네트워크에 속하는지 확인
a = ipaddress.ip_address("192.168.10.42")
b = ipaddress.ip_address("192.168.10.200")
print(a in net and b in net)    # True — 같은 서브넷
💡 비트 수가 작을수록 더 크다

/24보다 /16이 더 큰 네트워크입니다. 네트워크 부분 비트가 적을수록 호스트에 줄 비트가 많아지기 때문입니다. /16은 약 6만 5천 개의 주소를 담습니다.

NAT (네트워크 주소 변환)

IPv4 주소는 부족한데 집·회사에는 기기가 수십 개입니다. NAT(Network Address Translation)는 이 문제를 해결합니다. 내부 기기들은 사설 IP(192.168.x.x 등)를 쓰고, 인터넷으로 나갈 때만 공유기(라우터)가 공인 IP 하나로 바꿔치기합니다. 회사 대표 전화번호 하나로 내선 여러 개를 운영하는 것과 같습니다.

집안 기기들의 사설 IP를 공유기가 공인 IP 하나로 변환해 인터넷에 내보낸다
📌 포트로 구분한다

여러 기기가 같은 공인 IP를 쓰면, 돌아온 응답을 누구에게 줄지 어떻게 알까요? NAT는 각 연결마다 포트 번호를 다르게 매핑해 표(NAT 테이블)에 적어 둡니다. 응답이 오면 그 포트를 보고 원래 기기로 정확히 되돌려 줍니다.

⚠️ 외부에서 먼저 접속하기 어렵다

NAT 안의 기기는 밖으로 나가긴 쉽지만, 외부에서 안으로 먼저 들어오긴 어렵습니다(사설 IP라 직접 지정 불가). 그래서 집에서 게임 서버를 열려면 포트 포워딩 설정이 필요합니다.

TCP 흐름 제어와 혼잡 제어

TCP는 신뢰성만 챙기는 게 아니라 속도도 영리하게 조절합니다. 두 가지 제어를 구분해야 합니다. 흐름 제어(flow control)는 받는 쪽이 감당할 만큼만 보내는 것(수신자 보호)이고, 혼잡 제어(congestion control)는 네트워크 길이 막히지 않게 조절하는 것(네트워크 보호)입니다.

흐름 제어는 수신자가 "내 버퍼에 이만큼 여유 있어요"라고 알리는 윈도 크기(window size)로 이뤄집니다. 혼잡 제어는 처음엔 조금씩 보내다 문제없으면 점점 빠르게(느린 시작), 손실이 감지되면 확 줄이는 방식으로 동작합니다.

처음엔 빠르게 늘리다가, 패킷 손실을 감지하면 전송량을 크게 줄여 길을 비운다
구분흐름 제어혼잡 제어
보호 대상느린 수신자혼잡한 네트워크
판단 근거수신자 버퍼 여유(윈도)패킷 손실·지연
대표 기법슬라이딩 윈도느린 시작·혼잡 회피
💡 둘 다 "윈도"를 쓴다

실제 TCP는 매 순간 보낼 양을 "수신 윈도(흐름 제어)"와 "혼잡 윈도(혼잡 제어)" 중 더 작은 값으로 정합니다. 수신자도 여유 있고 네트워크도 한가할 때만 빠르게 보냅니다.

쿠키와 세션

HTTP는 본래 상태가 없습니다(stateless). 요청 하나가 끝나면 서버는 방금 누가 왔는지 잊어버립니다. 그런데 로그인 상태는 유지돼야 하죠. 이 모순을 쿠키(cookie)와 세션(session)이 해결합니다.

쿠키는 서버가 발급해 브라우저에 저장하는 작은 메모지로, 이후 요청마다 자동으로 함께 보내집니다. 세션은 실제 사용자 정보를 서버에 보관하고, 쿠키에는 그 보관함을 여는 열쇠(세션 ID)만 담는 방식입니다.

로그인하면 서버가 세션 ID를 쿠키로 내려주고, 이후 요청마다 그 쿠키로 사용자를 식별한다
구분쿠키세션
저장 위치브라우저(클라이언트)서버
보안탈취·변조 위험상대적으로 안전(ID만 노출)
서버 부담없음사용자 수만큼 메모리 사용
⚠️ 쿠키 보안 속성

중요한 쿠키에는 HttpOnly(자바스크립트 접근 차단 → XSS 방어), Secure(HTTPS에서만 전송), SameSite(다른 사이트 요청에 쿠키 첨부 제한 → CSRF 방어) 속성을 붙여야 합니다.

HTTPS와 TLS 상세

HTTPS는 HTTP에 TLS(Transport Layer Security)라는 암호화 계층을 입힌 것입니다. 덕분에 중간에 누가 패킷을 가로채도 내용을 읽지 못하고(기밀성), 내용이 변조되지 않았으며(무결성), 상대가 진짜 그 사이트가 맞는지(인증) 보장됩니다.

핵심 아이디어는 두 암호화의 결합입니다. 먼저 비대칭 키(공개키/개인키)로 안전하게 "대화용 비밀 열쇠"를 교환하고, 그 다음부터는 빠른 대칭 키로 실제 데이터를 암호화합니다. 느린 방식으로 열쇠만 안전하게 건네고, 빠른 방식으로 대화하는 셈입니다.

인증서로 서버를 검증하고 비밀 열쇠를 안전하게 교환한 뒤, 빠른 대칭키로 본 통신을 한다
💡 인증서와 CA

서버의 공개키가 진짜인지는 CA(인증 기관)가 서명한 인증서로 보증합니다. 브라우저는 신뢰하는 CA 목록을 미리 갖고 있어, 인증서 서명을 확인해 가짜 사이트를 걸러냅니다. 주소창의 자물쇠 아이콘이 이 검증의 통과를 뜻합니다.

웹소켓 (WebSocket)

기본 HTTP는 클라이언트가 물어야만 서버가 답하는 단방향 구조입니다. 그래서 채팅처럼 서버가 먼저 말을 걸어야 하는 실시간 서비스엔 불편합니다. 웹소켓(WebSocket)은 한 번 연결을 맺으면 양방향으로 자유롭게 메시지를 주고받는 지속 연결을 제공합니다.

폴링은 매번 새로 묻지만, 웹소켓은 한 번 연 통로로 양쪽이 자유롭게 메시지를 보낸다
websocket_client.js
// 브라우저에서 웹소켓 연결 열기
const ws = new WebSocket("wss://example.com/chat");

ws.onopen = () => ws.send("안녕하세요!");   // 연결되면 메시지 전송
ws.onmessage = (e) => {                      // 서버가 보낸 메시지 수신
  console.log("받음:", e.data);             // 서버가 먼저 말 걸 수 있음
};
ws.onclose = () => console.log("연결 종료");
💡 ws:// 와 wss://

웹소켓 주소는 ws://(암호화 안 함)와 wss://(TLS 암호화)로 시작합니다. HTTP·HTTPS의 관계와 같으며, 실서비스에선 반드시 wss://를 씁니다. 처음 연결은 HTTP 요청에 Upgrade 헤더를 붙여 웹소켓으로 "승격"하는 방식으로 시작합니다.

CDN과 캐싱

서울 사용자가 미국 서버의 이미지를 받으면 멀어서 느립니다. CDN(Content Delivery Network)은 콘텐츠 복사본을 전 세계 곳곳의 서버(엣지)에 미리 뿌려 두고, 사용자에게 가장 가까운 서버가 응답하게 합니다. 동네마다 편의점을 둬서 멀리 갈 필요를 없애는 것과 같습니다.

그 바탕에는 캐싱(caching)이 있습니다. 한 번 받은 데이터를 가까운 곳에 잠시 저장해 두고, 다음 요청은 원본까지 가지 않고 사본으로 빠르게 응답하는 기법입니다. 브라우저 캐시, CDN 캐시 등 여러 층에서 동시에 일어납니다.

원본은 한 곳이지만 사본을 세계 곳곳 엣지에 두어, 사용자는 가장 가까운 서버에서 받는다
📌 캐시 적중과 만료

요청한 데이터가 캐시에 있으면 캐시 적중(hit), 없어서 원본까지 가면 캐시 미스(miss)입니다. 캐시는 Cache-Control, ETag 같은 HTTP 헤더로 "얼마나 오래 보관할지", "원본이 바뀌었는지"를 판단합니다.

⚠️ 오래된 캐시 문제

캐싱의 가장 큰 골칫거리는 "원본은 바뀌었는데 사용자는 옛날 사본을 본다"입니다. 그래서 파일 이름에 버전(app.v2.js)이나 해시를 붙여, 내용이 바뀌면 주소도 바뀌게 만들어 새 파일을 받도록 유도합니다.

🎯 핵심 정리
  • 네트워크는 데이터를 패킷에 담아 라우터를 거쳐 전달하는 우편 시스템과 같다.
  • 계층 모델은 복잡한 통신을 층으로 나눠 한 층만 바꿔도 나머지는 그대로 둔다.
  • IP 주소는 컴퓨터의 위치이고, 라우팅은 그곳까지의 경로를 찾는 과정이다.
  • TCP는 신뢰성(3-way handshake), UDP는 속도가 강점이다.
  • HTTP는 요청-응답 구조, DNS는 이름을 IP로 바꾸는 전화번호부, 소켓은 통신 창구다.
  • CIDR(/24 등)은 IP를 네트워크 부분과 호스트 부분으로 나눠 서브넷을 관리한다.
  • NAT는 사설 IP 여러 개를 공인 IP 하나로 변환해 주소 부족을 해결한다.
  • TCP의 흐름 제어는 느린 수신자를, 혼잡 제어는 막힌 네트워크를 보호한다.
  • 쿠키는 브라우저에, 세션은 서버에 저장하며 둘이 함께 로그인 상태를 유지한다.
  • HTTPS는 TLS로 비대칭키로 열쇠를 교환한 뒤 빠른 대칭키로 암호화 통신한다.
  • 웹소켓은 한 번 연 통로로 서버·클라이언트가 양방향 실시간 통신을 한다.
  • CDN과 캐싱은 사본을 가까운 곳에 두어 응답 속도를 높인다.
패킷(Packet)
네트워크로 전송되는 데이터 조각. 헤더(주소 등)와 본문으로 구성.
IP 주소 / 포트
각각 컴퓨터의 위치와 그 안의 특정 프로그램을 가리키는 번호.
TCP / UDP
신뢰성을 보장하는 연결형 / 속도를 우선하는 비연결형 전송 프로토콜.
3-way handshake
SYN·SYN+ACK·ACK 세 단계로 TCP 연결을 맺는 절차.
DNS
도메인 이름을 IP 주소로 변환하는 인터넷의 전화번호부.
소켓(Socket)
프로그램이 네트워크 통신을 위해 여는 IP+포트 기반 창구.
서브넷 / CIDR
IP를 네트워크·호스트 부분으로 나눠 관리하는 기법과 /숫자 표기법.
NAT
사설 IP를 공인 IP로 변환해 여러 기기가 주소 하나를 공유하게 하는 기술.
흐름 제어 / 혼잡 제어
각각 느린 수신자와 막힌 네트워크를 보호하기 위한 TCP의 속도 조절.
쿠키 / 세션
상태 없는 HTTP에서 로그인 등을 유지하는, 브라우저 저장 / 서버 저장 방식.
TLS
HTTP를 암호화해 HTTPS로 만드는 보안 계층(비대칭키로 열쇠 교환).
웹소켓
한 번 연결하면 양방향으로 실시간 통신하는 지속 연결 프로토콜.
CDN / 캐싱
콘텐츠 사본을 가까운 엣지 서버에 두어 빠르게 응답하는 분산·저장 기법.
