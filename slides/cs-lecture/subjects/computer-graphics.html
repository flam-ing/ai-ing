컴퓨터 그래픽스

3D 세계를 화면 위 픽셀로 그려내는 수학과 알고리즘. 게임·영화·CAD의 화면 뒤에서 일어나는 일을 들여다봅니다.

컴퓨터 그래픽스란?

컴퓨터 그래픽스(computer graphics)는 숫자로 표현된 도형·모델을 화면의 픽셀로 변환하는 기술입니다. 게임의 3D 캐릭터, 영화의 CG, 지도 앱의 건물, CAD 설계도 모두 같은 원리 위에 있습니다.

비유하자면 화가가 머릿속 장면을 캔버스에 옮기는 것과 같습니다. 다만 컴퓨터는 붓 대신 좌표·행렬·삼각형을 쓰고, 캔버스 대신 픽셀 격자를 씁니다. 이 페이지는 그 "디지털 회화"의 도구들을 차례로 배웁니다.

📌 이 페이지에서 배우는 것

픽셀과 색 → 2D/3D 좌표 변환과 행렬 → 투영 → 래스터화 → 렌더링 파이프라인 → Z-버퍼 → 조명·셰이딩·텍스처 → 셰이더와 GPU까지, 한 픽셀이 칠해지기까지의 여정을 따라갑니다.

픽셀과 래스터 vs 벡터

화면은 작은 점들의 격자입니다. 이 점 하나를 픽셀(pixel, picture element)이라 합니다. 이미지를 표현하는 방식은 크게 둘로 나뉩니다.

  • 래스터(raster) — 픽셀 격자에 색을 직접 채움. 사진, PNG/JPG. 확대하면 계단처럼 깨짐.
  • 벡터(vector) — "여기서 저기로 선, 반지름 r 원" 같은 수식으로 도형을 정의. SVG, 폰트. 아무리 확대해도 매끈함.
래스터는 픽셀에 칠해 계단이 생기고, 벡터는 수식이라 무한 확대 가능
💡 안티에일리어싱

래스터의 계단 현상(에일리어싱)은 경계 픽셀을 중간 색으로 부드럽게 섞어 눈속임합니다. 이를 안티에일리어싱(anti-aliasing)이라 하며, 글꼴과 게임 화면이 매끄러워 보이는 비결입니다.

색 표현 (RGB)

화면의 모든 색은 빨강(R)·초록(G)·파랑(B) 세 빛을 섞어 만듭니다. 빛은 섞을수록 밝아지는 가산 혼합이라, R+G+B를 최대로 켜면 흰색이 됩니다. 각 채널은 보통 0~255(8비트) 값을 가집니다.

빛의 삼원색 RGB — 셋을 모두 켜면(255,255,255) 흰색
color.js
// 색은 (R, G, B) 세 숫자. 0~255
const red    = [255,   0,   0];   // 순수 빨강
const yellow = [255, 255,   0];   // 빨강 + 초록 = 노랑
const white  = [255, 255, 255];   // 셋 다 최대 = 흰색

// 16진수 표기 (#RRGGBB): 웹에서 흔한 방식
const accent = "#58a6ff";          // R=0x58, G=0xa6, B=0xff

// 알파(투명도)를 더하면 RGBA
const glass = "rgba(88, 166, 255, 0.5)";  // 반투명 파랑

2D 좌표계와 변환

도형을 움직이고 키우고 돌리는 것을 변환(transformation)이라 합니다. 기본 변환은 셋입니다.

  • 이동(translation) — 좌표에 값을 더함. x' = x + dx
  • 크기(scale) — 좌표에 값을 곱함. x' = x · sx
  • 회전(rotation) — 각도 θ만큼 원점 둘레로 돌림(사인·코사인 사용).
점선은 원래 위치, 실선은 변환 후 — 이동·크기·회전
transform2d.js
// 점 (x, y)에 적용하는 기본 2D 변환
function translate(x, y, dx, dy) { return [x + dx, y + dy]; }
function scale(x, y, sx, sy)     { return [x * sx, y * sy]; }

function rotate(x, y, deg) {
  const t = deg * Math.PI / 180;        // 각도 → 라디안
  return [
    x * Math.cos(t) - y * Math.sin(t),  // 회전 공식
    x * Math.sin(t) + y * Math.cos(t),
  ];
}

변환 행렬과 동차좌표

변환을 매번 따로 계산하면 번거롭습니다. 그래서 그래픽스는 모든 변환을 행렬 곱셈으로 통일합니다. 점을 벡터로, 변환을 행렬로 두면, 변환 적용은 그냥 "행렬 × 벡터"가 됩니다. 여러 변환을 미리 곱해 하나의 행렬로 합칠 수도 있어 매우 효율적입니다.

문제는 이동입니다. 이동은 "더하기"라서 곱셈인 행렬로 표현이 안 됩니다. 이를 해결하는 트릭이 동차좌표(homogeneous coordinates)입니다. 2D 점 (x, y)에 가짜 차원 1을 붙여 (x, y, 1)로 다루면, 이동까지도 3×3 행렬 하나로 표현됩니다.

3×3 행렬 한 번의 곱으로 이동 (x+dx, y+dy) 완성
matrix.js
// 동차좌표: 점은 [x, y, 1]
// 여러 변환 행렬을 곱해 '하나의 행렬'로 합치면 한 번에 적용 가능
function multiply(M, [x, y, w]) {
  return [
    M[0]*x + M[1]*y + M[2]*w,
    M[3]*x + M[4]*y + M[5]*w,
    M[6]*x + M[7]*y + M[8]*w,
  ];
}

const translateM = [1, 0, 30,   0, 1, 50,   0, 0, 1];  // dx=30, dy=50
const p = multiply(translateM, [10, 20, 1]);            // → [40, 70, 1]
📌 변환은 순서가 중요

행렬 곱은 교환법칙이 성립하지 않습니다. "회전 후 이동"과 "이동 후 회전"의 결과가 다릅니다. 합성 행렬을 만들 때 적용할 순서의 역순으로 곱한다는 점에 주의하세요.

3D 공간과 투영

3D에서는 좌표가 (x, y, z)로 늘고, 동차좌표는 (x, y, z, 1), 변환 행렬은 4×4가 됩니다. 그런데 화면은 2D입니다. 3D 점을 2D 화면으로 납작하게 만드는 과정이 투영(projection)입니다.

  • 원근 투영(perspective) — 멀리 있는 것은 작게. 사람 눈·카메라처럼 현실감 있음.
  • 직교 투영(orthographic) — 거리와 무관하게 크기 유지. CAD·설계도에 적합.
원근 투영: 눈에서 뻗은 선이 화면을 지나며, 멀수록 작게 맺힌다
원근 투영직교 투영
거리에 따른 크기멀수록 작아짐거리와 무관(일정)
현실감높음(눈·카메라와 유사)없음(왜곡 없음)
평행선한 점(소실점)으로 모임그대로 평행 유지
주 사용처게임·영화·1인칭 시점CAD·설계도·아이소메트릭
💡 MVP 행렬

실전에서는 모델 변환(Model)·시점 변환(View)·투영(Projection) 세 행렬을 곱해 MVP 행렬 하나로 만듭니다. 정점마다 이 행렬 한 번 곱하면 3D 월드 좌표가 화면 좌표로 직행합니다.

선·삼각형 래스터화

투영으로 2D 화면 좌표가 나오면, 이제 그 도형을 실제 픽셀로 채워야 합니다. 이것이 래스터화(rasterization)입니다. 3D 모델은 결국 수많은 삼각형으로 쪼개지므로, "삼각형 하나를 픽셀로 채우는 일"이 핵심입니다.

삼각형 래스터화의 기본 아이디어는 "각 픽셀의 중심이 삼각형 안에 있는가?"를 검사하는 것입니다. 안에 있는 픽셀만 칠합니다.

삼각형(초록 윤곽) 내부에 중심이 들어간 픽셀(파랑)만 칠한다
rasterize.txt
# 삼각형 래스터화 의사코드 (경계 박스 + 내부 판정)
삼각형 (A, B, C) 에 대해:
  bbox = (A,B,C 를 모두 감싸는 최소 사각형)
  for 각 픽셀 p in bbox:
    if p 의 중심이 삼각형 ABC 안에 있으면:   # 무게중심 좌표로 판정
       p 의 색을 칠한다
       (세 꼭짓점의 색/깊이를 보간해 p 값 결정)

렌더링 파이프라인

지금까지의 단계들이 일렬로 이어진 것이 렌더링 파이프라인(rendering pipeline)입니다. 정점 데이터가 들어가 화면 픽셀이 나오는 컨베이어 벨트라고 보면 됩니다. GPU는 이 단계들을 하드웨어로 초고속 처리합니다.

정점 → 정점 셰이더(변환) → 래스터화 → 프래그먼트 셰이더(색) → 출력 합성
단계입력 → 출력하는 일
정점 셰이더정점 → 정점MVP 행렬로 좌표 변환·투영
래스터화삼각형 → 프래그먼트도형을 픽셀 후보로 분해
프래그먼트 셰이더프래그먼트 → 색조명·텍스처로 최종 색 계산
출력 합성색 → 화면Z-버퍼 깊이 검사·블렌딩

은면 제거 (Z-버퍼)

3D 장면에서 앞 물체는 뒤 물체를 가립니다. 가려진 면(은면)을 그리지 않는 것이 은면 제거(hidden surface removal)입니다. 가장 널리 쓰는 방법이 Z-버퍼(z-buffer / depth buffer)입니다.

아이디어는 간단합니다. 화면 픽셀마다 "지금까지 그려진 것 중 가장 가까운 깊이(z)"를 따로 저장해 둡니다. 새 픽셀을 그릴 때, 그 깊이가 저장된 값보다 더 가까울 때만 색을 덮어씁니다.

픽셀마다 가장 가까운 z만 남긴다 → 더 가까운 초록이 노랑을 가린다
zbuffer.js
// 픽셀 (x,y)를 깊이 z, 색 color 로 그리려 할 때
function drawPixel(x, y, z, color) {
  const i = y * width + x;
  if (z < depthBuffer[i]) {     // 기존보다 더 가까우면
    depthBuffer[i] = z;          //   깊이 갱신
    colorBuffer[i] = color;      //   색 덮어쓰기
  }
  // 더 멀면 무시 → 뒤 물체가 앞 물체를 가리지 못함
}

조명과 셰이딩

색만 칠하면 평면처럼 밋밋합니다. 빛(light)을 흉내 내야 입체감이 생깁니다. 고전적인 퐁(Phong) 조명 모델은 빛을 세 성분으로 나눕니다.

  • 주변광(ambient) — 사방에서 오는 약한 기본 밝기. 완전한 그림자를 막음.
  • 난반사(diffuse) — 표면이 빛을 향한 정도에 따른 밝기. 입체감의 핵심.
  • 정반사(specular) — 매끈한 표면의 반짝이는 하이라이트.

난반사의 핵심은 표면의 법선 벡터(normal)와 빛 방향의 각도입니다. 정면으로 받으면 밝고, 비스듬하면 어둡습니다(람베르트 법칙: 밝기 ∝ 코사인).

퐁 모델: 주변광 + 난반사 + 정반사를 합쳐 입체감 있는 구를 만든다
💡 셰이딩 방식

조명 계산을 어디서 하느냐로 나뉩니다. 플랫(면마다 하나), 고러드(꼭짓점에서 계산 후 보간), 퐁(픽셀마다 법선 보간 후 계산). 아래로 갈수록 부드럽지만 비용이 큽니다.

텍스처 매핑

모든 색을 일일이 계산하는 대신, 이미지를 도형 표면에 붙이는 것이 텍스처 매핑(texture mapping)입니다. 벽돌·나무·얼굴 무늬를 단 한 장의 사진으로 표현합니다.

핵심은 UV 좌표입니다. 텍스처 이미지의 위치를 (u, v) (둘 다 0~1)로 표시하고, 3D 모델의 각 정점에 "이 정점은 텍스처의 어느 점에 대응한다"는 UV를 붙여 둡니다. 래스터화 때 이 UV를 보간해 픽셀마다 텍스처에서 색을 읽어옵니다.

UV 좌표로 텍스처의 어느 부분이 3D 면의 어디에 붙는지 정한다
⚠️ 텍스처 필터링

텍스처를 늘리거나 줄이면 픽셀이 깨지거나 어른거립니다. 밉맵(mipmap)(축소판을 미리 만들어 둠)과 이중 선형 필터링으로 이를 완화합니다. 멀리 있는 텍스처가 지글거리지 않게 하는 필수 기법입니다.

셰이더와 GPU

화면에는 수백만 픽셀이 있고, 그 각각을 매 프레임(보통 초당 60번) 계산해야 합니다. CPU 혼자서는 벅찹니다. 그래서 GPU(그래픽 처리 장치)가 등장합니다. GPU는 단순한 연산기를 수천 개 두고 모든 픽셀·정점을 동시에 병렬 처리합니다.

이 GPU 위에서 도는 작은 프로그램이 셰이더(shader)입니다. GLSL 같은 언어로 작성하며, "정점 하나를 어떻게 변환할지"(정점 셰이더), "픽셀 하나를 무슨 색으로 칠할지"(프래그먼트 셰이더)를 정의합니다.

fragment.glsl
// 프래그먼트 셰이더: 픽셀 하나의 최종 색을 정한다 (GPU에서 병렬 실행)
precision mediump float;

varying vec3 vNormal;       // 보간된 법선 벡터
uniform vec3 uLightDir;     // 빛의 방향

void main() {
  // 람베르트 난반사: 법선과 빛 방향의 코사인(내적)
  float diff = max(dot(normalize(vNormal), normalize(uLightDir)), 0.0);
  vec3 base = vec3(0.34, 0.65, 1.0);          // 기본 파랑
  gl_FragColor = vec4(base * (0.2 + diff), 1.0);  // 주변광 0.2 + 난반사
}
GPU는 작은 코어를 대량으로 두고 픽셀·정점을 병렬 처리한다
📌 한 픽셀의 여정 요약

정점 데이터 → (정점 셰이더) 변환·투영 → 래스터화로 픽셀 분해 → (프래그먼트 셰이더) 조명·텍스처로 색 결정 → Z-버퍼로 가림 판정 → 화면. 이 전체를 GPU가 초당 수십 번 반복하며 움직이는 화면을 만들어 냅니다.

🎯 핵심 정리
  • 화면의 색은 RGB 빛의 가산 혼합이고, 이미지는 래스터(픽셀)와 벡터(수식)로 나뉜다.
  • 이동·회전·크기 등 모든 변환은 행렬 곱셈으로 통일된다.
  • 동차좌표(가짜 차원 1)를 쓰면 '이동'까지 행렬 하나로 표현된다.
  • 3D를 2D 화면으로 납작하게 만드는 과정이 투영(원근/직교)이다.
  • 3D 모델은 삼각형으로 쪼개지고, 래스터화가 이를 픽셀로 채운다.
  • Z-버퍼는 픽셀마다 가장 가까운 깊이만 남겨 앞 물체가 뒤를 가리게 한다.
  • 퐁 조명(주변광+난반사+정반사)과 텍스처 매핑이 사실감을 만든다.
  • 셰이더는 GPU에서 병렬 실행되는 작은 프로그램으로, 정점과 픽셀을 처리한다.

이미지를 픽셀 격자로 / 수식으로 표현하는 두 방식.
동차좌표
이동까지 행렬 곱으로 다루려고 차원 1을 덧붙인 좌표 표기.
투영(Projection)
3D 점을 2D 화면 좌표로 사상하는 변환(원근·직교).
래스터화
도형(삼각형)을 어떤 픽셀로 채울지 결정하는 과정.
Z-버퍼
픽셀별 깊이를 저장해 가려진 면을 제거하는 버퍼.
법선 벡터(Normal)
표면이 향하는 방향 벡터. 조명 밝기 계산의 기준.
UV 좌표
텍스처 이미지 위 위치를 0~1로 나타낸 좌표.
셰이더(Shader)
GPU에서 정점·픽셀을 처리하는 작은 프로그램.
