계산이론

"무엇을 계산할 수 있고, 무엇은 끝내 풀 수 없는가?" 컴퓨터의 능력과 한계를 수학으로 따져 봅니다.

계산이론이란?

계산이론(theory of computation)은 "문제를 컴퓨터로 풀 수 있는가, 풀 수 있다면 얼마나 효율적으로 풀 수 있는가"를 연구하는 분야입니다. 실제 CPU가 아니라 이상화된 계산 모델(추상 기계)을 가지고 따집니다. 덕분에 특정 언어나 하드웨어에 얽매이지 않고, 계산 그 자체의 본질을 다룰 수 있습니다.

비유하자면, 자동차의 성능을 특정 도로가 아니라 물리 법칙으로 분석하는 것과 같습니다. "이 엔진으로 도달할 수 있는 최고 속도"처럼, "어떤 기계로도 절대 풀 수 없는 문제"가 존재한다는 사실을 증명하는 것이 이 분야의 백미입니다.

크게 세 갈래로 나뉩니다.

  • 오토마타 이론(Automata) — 단순한 추상 기계들로 어떤 언어를 인식할 수 있는지.
  • 계산 가능성(Computability) — 애초에 풀 수 있는 문제와 풀 수 없는 문제의 경계.
  • 복잡도 이론(Complexity) — 풀 수 있는 문제 중 "현실적인 시간 안에" 풀리는 것의 경계.
📌 이 페이지에서 배우는 것

형식 언어 · 정규표현식 · 유한 오토마타(DFA/NFA) · 펌핑 보조정리 · 문맥자유문법 · 푸시다운 오토마타 · 튜링 머신 · 정지문제 · P와 NP까지, 추상 기계의 사다리를 한 칸씩 올라갑니다.

형식 언어와 알파벳

계산이론에서 "언어(language)"는 한국어·영어 같은 자연어가 아니라, 정해진 규칙으로 만든 문자열의 집합을 뜻합니다. 출발점은 세 가지입니다.

  • 알파벳(Σ, 시그마) — 사용할 수 있는 기호들의 유한 집합. 예: Σ = {0, 1}.
  • 문자열(string) — 알파벳의 기호를 늘어놓은 것. 예: 0110. 아무것도 없는 빈 문자열은 ε(엡실론)으로 씁니다.
  • 언어(language) — 그런 문자열들 중 우리가 관심 있는 것만 모은 집합. 예: "0이 짝수 개인 문자열들".

그래서 "문제를 푼다"는 것은 곧 "어떤 문자열이 이 언어에 속하는지 판별한다"로 바꿔 말할 수 있습니다. 이 추상화가 계산이론 전체를 떠받칩니다.

언어 L은 "만들 수 있는 모든 문자열(Σ*)" 중 규칙에 맞는 부분집합
💡 촘스키 위계

언어는 만드는 규칙의 강력함에 따라 정규 → 문맥자유 → 문맥의존 → 재귀적 가산 4단계로 나뉩니다(촘스키 위계). 위로 갈수록 더 복잡한 언어를 표현하지만, 이를 인식하는 기계도 더 강력해야 합니다. 이 페이지는 이 사다리를 따라 올라갑니다.

정규언어와 정규표현식

가장 단순한 언어가 정규언어(regular language)입니다. "패턴 매칭"으로 표현할 수 있는 언어로, 우리가 코드에서 자주 쓰는 정규표현식(regular expression)이 바로 이것입니다.

정규표현식은 세 가지 연산만으로 만들어집니다.

  • 이어붙이기(concatenation) — ab : a 다음에 b.
  • 선택(union, |) — a|b : a 또는 b.
  • 반복(Kleene star, *) — a* : a를 0번 이상.
regex.py
import re

# "0으로 시작해 1로 끝나는 이진 문자열"
pattern = re.compile(r"0[01]*1")

print(bool(pattern.fullmatch("001")))    # True
print(bool(pattern.fullmatch("0111")))   # True
print(bool(pattern.fullmatch("10")))     # False — 0으로 시작하지 않음
print(bool(pattern.fullmatch("0")))      # False — 1로 끝나지 않음

핵심은 이것입니다. 정규언어로 표현할 수 있는 것 = 유한 오토마타로 인식할 수 있는 것 = 정규표현식으로 쓸 수 있는 것. 이 셋은 완전히 같은 표현력을 가집니다(클레이니 정리). 다음 절에서 그 "기계"를 만나봅니다.

⚠️ 정규표현식의 한계

정규언어는 "기억(메모리)"이 없습니다. 그래서 여는 괄호와 닫는 괄호의 개수가 정확히 같은지 같은 균형 검사는 정규표현식으로 절대 불가능합니다. 이유는 뒤의 '펌핑 보조정리'에서 증명합니다.

유한 오토마타 (DFA · NFA)

유한 오토마타(finite automaton)는 가장 단순한 추상 기계입니다. 상태(state)가 유한 개이고, 입력 기호를 하나씩 읽으며 상태를 옮겨갑니다. 입력을 다 읽었을 때 "수락 상태(accept state)"에 있으면 그 문자열을 받아들입니다. 메모리는 오직 "지금 어느 상태에 있는가" 뿐입니다.

  • DFA(결정적 유한 오토마타) — 각 상태에서 각 입력마다 갈 곳이 딱 하나로 정해짐.
  • NFA(비결정적 유한 오토마타) — 갈 곳이 여러 개일 수 있고, ε(입력 없이) 이동도 허용. 단, NFA와 DFA의 표현력은 똑같습니다(모든 NFA는 동등한 DFA로 변환 가능).
"1이 짝수 개"인지 판별하는 DFA — 이중 원이 수락 상태. 1을 읽을 때마다 짝/홀이 뒤집힘
dfa.py
# 위 다이어그램의 DFA를 그대로 코드로
transition = {
    ("q0", "0"): "q0", ("q0", "1"): "q1",   # q0: 1을 짝수 개 본 상태
    ("q1", "0"): "q1", ("q1", "1"): "q0",   # q1: 1을 홀수 개 본 상태
}
accept = {"q0"}                              # 짝수면 수락

def run(s):
    state = "q0"                             # 시작 상태
    for ch in s:
        state = transition[(state, ch)]      # 상태 전이
    return state in accept

print(run("1010"))   # True  — 1이 2개(짝수)
print(run("111"))    # False — 1이 3개(홀수)

펌핑 보조정리 (개념)

"이 언어는 정규언어가 아니다"를 증명하려면 어떻게 할까요? 유한 오토마타는 상태가 유한 개뿐이라, 충분히 긴 문자열을 처리하면 같은 상태를 반드시 두 번 이상 지나갑니다(비둘기집 원리). 그 사이 구간은 "고리(loop)"이므로 몇 번이고 반복(펌핑)해도 여전히 수락되어야 합니다.

정규언어라면 가운데 y를 몇 번 반복해도 여전히 언어에 속해야 한다

이를 이용해 모순을 만들면 됩니다. 예를 들어 언어 L = { 0ⁿ1ⁿ | n ≥ 0 }("0이 n개, 1이 n개")가 정규언어라고 가정해 봅시다. 충분히 긴 0ⁿ1ⁿ의 펌핑 구간 y는 반드시 0들 안에만 들어갑니다. y를 한 번 더 반복하면 0의 개수가 늘어 0과 1의 수가 어긋나므로 언어를 벗어납니다. 모순! 따라서 0ⁿ1ⁿ은 정규언어가 아닙니다.

📌 직관

유한 오토마타는 "지금까지 0을 몇 개 봤는지" 같은 무한히 커질 수 있는 수를 셀 수 없습니다. 셈을 하려면 메모리가 필요하고, 그 메모리(스택)를 더한 것이 다음에 나오는 푸시다운 오토마타입니다.

문맥자유문법 (CFG)

정규언어로는 부족한 "균형 잡힌 구조"(괄호 짝, 중첩 문장)를 표현하려면 한 단계 위인 문맥자유문법(context-free grammar, CFG)이 필요합니다. CFG는 치환 규칙(production)들의 모음입니다.

  • 비단말 기호(대문자, 예: S) — 더 펼쳐질 "변수".
  • 단말 기호(소문자/기호) — 최종 출력 문자.
  • 규칙 — S → aSb 처럼 비단말을 다른 것으로 바꾸는 법.
grammar.txt
# 0ⁿ1ⁿ 을 생성하는 문맥자유문법 (정규언어로는 불가능했던 것!)
S → 0 S 1     # 양 끝에 0과 1을 동시에 추가
S → ε         # 빈 문자열로 종료

# 유도(derivation) 예시:
# S → 0S1 → 00S11 → 000S111 → 000111   (마지막에 S → ε)

이런 규칙으로 문자열이 만들어지는 과정을 나무 모양으로 그린 것이 파스 트리(parse tree)입니다. 프로그래밍 언어의 컴파일러가 소스 코드를 분석하는 "구문 분석(parsing)" 단계가 바로 CFG 위에서 돌아갑니다.

"0011"의 파스 트리 — S가 규칙을 따라 펼쳐지는 과정

푸시다운 오토마타 (PDA)

문맥자유언어를 인식하는 기계가 푸시다운 오토마타(pushdown automaton, PDA)입니다. 유한 오토마타에 스택(stack) 하나를 붙인 것입니다. 스택은 "마지막에 넣은 것을 먼저 꺼내는(LIFO)" 메모리라, 중첩 구조를 처리하기에 안성맞춤입니다.

PDA = 유한 오토마타 + 스택. 0을 만나면 스택에 push, 1을 만나면 pop → 끝에 비면 짝이 맞음

0ⁿ1ⁿ 검사 전략은 단순합니다. 0을 읽을 때마다 스택에 표시를 쌓고, 1을 읽을 때마다 하나씩 꺼냅니다. 입력이 끝났을 때 스택이 정확히 비어 있으면 0과 1의 개수가 같았다는 뜻입니다. 유한 오토마타가 셀 수 없던 일을, 스택이라는 추가 메모리가 해결합니다.

💡 사다리 정리

정규언어 ⊂ 문맥자유언어. 유한 오토마타에 스택을 더하면 PDA, 거기서 메모리를 "무한 테이프"로 키우면 다음의 튜링 머신이 됩니다. 메모리를 키울수록 풀 수 있는 문제가 넓어집니다.

튜링 머신 (Turing Machine)

튜링 머신(Turing machine)은 계산의 궁극적 모델입니다. 무한히 긴 테이프(메모리)와, 그 위를 좌우로 움직이며 기호를 읽고 쓰는 헤드로 이루어집니다. 각 단계에서 현재 상태와 헤드 밑의 기호를 보고 ① 기호를 쓰고 ② 헤드를 왼쪽/오른쪽으로 옮기고 ③ 상태를 바꿉니다.

헤드가 가리키는 칸을 읽고, 규칙에 따라 다시 쓰고, 한 칸 이동한다 — 이 단순함이 모든 계산을 표현한다
turing.py
# 튜링 머신 한 스텝: (상태, 읽은 기호) → (쓸 기호, 이동 방향, 다음 상태)
rules = {
    ("q0", "1"): ("0", "R", "q0"),   # 1을 0으로 바꾸고 오른쪽으로
    ("q0", "_"): ("_", "S", "halt"), # 빈칸을 만나면 정지
}

def step(state, tape, head):
    sym = tape[head]
    write, move, nxt = rules[(state, sym)]
    tape[head] = write                       # 테이프에 쓰기
    head += 1 if move == "R" else -1 if move == "L" else 0
    return nxt, tape, head                    # 다음 상태로
📌 처치–튜링 명제

"직관적으로 '계산 가능한' 모든 것은 튜링 머신으로 계산할 수 있다." 이것이 처치–튜링 명제입니다. 파이썬·C·양자컴퓨터 무엇이든, 계산 능력의 상한은 결국 튜링 머신과 같습니다(이를 '튜링 완전'이라 합니다). 그래서 튜링 머신으로 못 푸는 문제는 어떤 컴퓨터로도 못 풉니다.

계산 가능성과 정지문제

이제 충격적인 사실입니다. 어떤 컴퓨터로도 결코 풀 수 없는 문제가 존재합니다. 가장 유명한 예가 정지문제(halting problem)입니다.

"임의의 프로그램 P와 입력 I가 주어졌을 때, P(I)가 언젠가 멈추는지 아니면 영원히 도는지를 항상 올바르게 판정하는 프로그램은 만들 수 있는가?" 답은 불가능입니다(튜링, 1936). 증명은 "자기 자신을 모순되게 사용하는" 대각선 논법으로 합니다.

halting_paradox.py
# 만약 '정지 여부를 판정하는' halts(prog, inp) 가 존재한다고 가정하자.
def halts(prog, inp):    # (가정) prog(inp)가 멈추면 True, 아니면 False
    ...

def paradox(prog):
    if halts(prog, prog):   # prog가 자기 자신에 대해 멈춘다면…
        while True:         #   …일부러 무한 루프 (멈추지 않음!)
            pass
    else:                   # 멈추지 않는다면…
        return             #   …즉시 멈춤!

# paradox(paradox) 는?  멈추면 안 멈추고, 안 멈추면 멈춘다 → 모순!
# 따라서 halts() 는 애초에 존재할 수 없다.

정지문제처럼 알고리즘으로 풀 수 없는 문제를 결정 불가능(undecidable) 문제라 합니다. 이는 단순한 호기심이 아닙니다. "이 프로그램에 버그(무한 루프)가 없음을 자동으로 완벽히 검증하는 도구"가 원리적으로 불가능하다는 뜻이기도 합니다.

🚫 결정 불가능의 파급

라이스 정리(Rice's theorem)에 따르면, "프로그램의 출력에 관한 사소하지 않은 모든 성질"은 일반적으로 결정 불가능합니다. 완벽한 바이러스 탐지기·완벽한 정적 분석기가 존재할 수 없는 이론적 근거가 여기에 있습니다.

시간복잡도 클래스 (P · NP)

풀 수 있는(결정 가능한) 문제 중에서도, "현실적인 시간 안에 풀리는가"가 또 다른 경계입니다. 여기서 P와 NP가 등장합니다.

  • P — 입력 크기 n에 대해 다항 시간(n², n³ 등)에 풀 수 있는 문제. "효율적으로 풀린다"의 기준선.
  • NP — 답이 주어졌을 때 그 답이 맞는지 다항 시간에 검증할 수 있는 문제. (푸는 건 어려워도 채점은 쉬운 문제.)

예: 스도쿠는 푸는 데는 오래 걸려도, 완성된 답이 맞는지 확인하는 건 순식간입니다 → NP. 모든 P 문제는 당연히 NP에도 속합니다(풀 수 있으면 검증도 할 수 있으니까). 그래서 P ⊆ NP는 확실합니다.

P는 NP 안에 들어간다. NP-완전 문제는 NP 중 가장 어려운 부류 — P와 겹치는지가 핵심 미해결 문제
📌 100만 달러짜리 질문: P = NP?

"검증이 쉬운 문제는 푸는 것도 쉬운가?" 즉 P = NP인가는 컴퓨터과학 최대 난제이며, 풀면 클레이 연구소에서 100만 달러를 받습니다. 대부분의 학자는 P ≠ NP(검증은 쉬워도 푸는 건 본질적으로 어렵다)라고 믿지만, 아직 누구도 증명하지 못했습니다.

NP-완전과 환원

NP-완전(NP-complete) 문제는 NP 안에서 "가장 어려운" 부류입니다. 정의가 멋집니다. NP의 모든 문제를 다항 시간에 이 문제로 바꿔 풀 수 있다. 그래서 NP-완전 문제 단 하나라도 다항 시간 알고리즘을 찾으면, NP 전체가 P로 무너집니다(P = NP).

핵심 도구는 환원(reduction)입니다. "문제 A를 문제 B로 환원한다"는 것은, A의 입력을 B의 입력으로 빠르게 변환해 B의 답으로 A를 풀 수 있다는 뜻입니다. 즉 "B가 A만큼은 어렵다"는 증거가 됩니다.

결정 문제 vs 최적화 문제

계산이론은 보통 답이 예/아니오인 결정 문제(decision problem)로 정리합니다. "최단 경로의 길이를 구하라"(최적화)가 아니라, "길이 K 이하인 경로가 존재하는가?"(결정)로 바꾸는 식입니다. 이렇게 하면 모든 문제를 "언어 소속 판별"이라는 하나의 틀로 다룰 수 있습니다.

  • 결정 문제 — 답이 예/아니오. (이론 분석에 적합)
  • 탐색 문제 — 조건을 만족하는 해 자체를 찾기.
  • 최적화 문제 — 가능한 해 중 최선(최소/최대)을 찾기.

다행히 셋은 보통 서로 다항 시간에 오갈 수 있어, 결정 문제로 P/NP를 분석한 결과가 실제 최적화 문제에도 그대로 적용됩니다. "이론은 결정 문제로, 현실은 최적화 문제로" 정도로 기억하면 됩니다.

💡 현실에서는?

NP-완전이라고 포기하는 게 아닙니다. 근사 알고리즘(approximation), 휴리스틱, 작은 입력에 한정한 정확 해법 등으로 충분히 쓸 만한 답을 얻습니다. "이론적으로 어렵다 ≠ 실무에서 못 쓴다"입니다.

🎯 핵심 정리
  • 계산이론은 추상 기계로 "무엇을 계산할 수 있는가"와 "얼마나 빨리 풀 수 있는가"를 따진다.
  • 표현력의 사다리: 정규언어(유한 오토마타) ⊂ 문맥자유언어(PDA) ⊂ 튜링 인식 언어(튜링 머신).
  • 유한 오토마타는 메모리가 상태뿐이라 셈을 못 한다(펌핑 보조정리로 증명).
  • 스택을 더하면 PDA, 무한 테이프를 더하면 튜링 머신 — 메모리를 키울수록 더 강력하다.
  • 처치–튜링 명제: 어떤 컴퓨터의 계산 능력도 결국 튜링 머신과 같다.
  • 정지문제는 결정 불가능 — 어떤 알고리즘으로도 절대 풀 수 없다.
  • P는 다항 시간에 풀리는 문제, NP는 다항 시간에 검증되는 문제. P=NP는 미해결 난제.
  • NP-완전 문제 하나라도 다항 시간에 풀면 NP 전체가 P가 된다.
