데이터베이스

수많은 데이터를 잃지 않고, 빠르게, 정확하게 다루는 기술입니다. 표(테이블)로 정리하고 SQL로 묻는 법을 배웁니다.

데이터베이스란?

데이터베이스(database)는 여러 사람이 함께 쓰는, 잘 정리된 데이터 창고입니다. 엑셀 파일 하나로도 데이터를 담을 수 있지만, 수백만 건을 동시에 여러 명이 읽고 쓰며, 절대 사라지지 않게 하려면 전문 도구가 필요합니다. 그 도구를 DBMS(DataBase Management System, 데이터베이스 관리 시스템)라 부릅니다.

도서관에 비유하면 책은 데이터, 책장의 분류 규칙과 사서는 DBMS입니다. 아무 데나 책을 쌓아두면 찾을 수 없듯, 규칙 없이 데이터를 모으면 쓸모가 없습니다.

📌 이 페이지에서 배우는 것

관계형 모델(테이블·키) · SQL 기초(SELECT·JOIN) · 정규화 · 인덱스 · 트랜잭션과 ACID 까지, 실무에서 가장 많이 쓰는 관계형 데이터베이스를 다룹니다.

관계형 모델 (테이블·키)

관계형 모델(relational model)은 데이터를 행과 열로 이루어진 표(테이블)에 담습니다. 한 줄(행, row)이 하나의 데이터(예: 학생 한 명), 한 칸의 종류(열, column)가 속성(예: 이름, 학번)입니다.

표끼리 흩어진 정보를 연결하는 열쇠가 키(key)입니다. 각 행을 유일하게 구별하는 열을 기본 키(Primary Key), 다른 표의 기본 키를 가리켜 두 표를 잇는 열을 외래 키(Foreign Key)라 합니다.

student 한 명이 여러 enrollment 를 가짐(1:N) — FK 가 두 표를 연결
용어뜻비유
테이블행·열로 된 데이터의 표엑셀 시트 한 장
기본 키(PK)각 행을 유일하게 구별주민등록번호
외래 키(FK)다른 표의 PK를 가리킴참고 도서 청구기호
schema.sql
CREATE TABLE student (
    id     INT PRIMARY KEY,          -- 기본 키: 학생을 유일하게 구별
    name   VARCHAR(20) NOT NULL,
    major  VARCHAR(30)
);

CREATE TABLE enrollment (
    id         INT PRIMARY KEY,
    student_id INT,
    course     VARCHAR(30),
    FOREIGN KEY (student_id) REFERENCES student(id)  -- 외래 키: 두 표를 연결
);

SQL 기초: SELECT

SQL(Structured Query Language)은 데이터베이스에게 "무엇을 원하는지"를 묻는 언어입니다. 가장 많이 쓰는 명령이 SELECT — "이런 조건의 데이터를 보여줘"라는 뜻입니다.

핵심은 절(clause)의 순서입니다. SELECT(어떤 열) → FROM(어떤 표) → WHERE(어떤 조건) → ORDER BY(정렬) 순서로 읽으면 자연스럽습니다.

select.sql
-- 컴퓨터공학과 학생의 이름을, 이름 순으로
SELECT name, major
FROM student
WHERE major = '컴퓨터공학'
ORDER BY name ASC;

-- 학과별 학생 수 세기 (그룹 묶기)
SELECT major, COUNT(*) AS 인원
FROM student
GROUP BY major;
💡 팁

SQL은 "어떻게 찾을지"가 아니라 "무엇을 원하는지"만 적는 선언형 언어입니다. 실제로 어떤 순서로 디스크를 뒤질지는 DBMS가 알아서 최적화합니다.

SQL 기초: JOIN

데이터는 여러 표에 나뉘어 있습니다(중복을 줄이려고). 흩어진 표를 키로 이어 붙여 한 결과로 합치는 것이 JOIN입니다. 가장 흔한 INNER JOIN은 양쪽 표에 모두 짝이 있는 행만 남깁니다.

두 표를 student.id = enrollment.student_id 로 연결
join.sql
-- 학생 이름과 그 학생이 수강한 과목을 함께 보기
SELECT s.name, e.course
FROM student AS s
INNER JOIN enrollment AS e
    ON s.id = e.student_id          -- PK = FK 로 연결
ORDER BY s.name;

-- 수강 이력이 없는 학생도 보고 싶다면 LEFT JOIN
SELECT s.name, e.course
FROM student AS s
LEFT JOIN enrollment AS e ON s.id = e.student_id;
⚠️ 주의

ON 조건을 빠뜨리면 두 표의 모든 행이 곱해지는 카티전 곱(cartesian product)이 일어나 결과가 폭발합니다. JOIN에는 거의 항상 연결 조건이 필요합니다.

정규화 (Normalization)

정규화는 중복 데이터를 없애 표를 잘게 나누는 설계 기법입니다. 한 표에 모든 걸 욱여넣으면, 같은 정보가 여러 번 반복되고(낭비), 한 곳만 고치면 나머지가 어긋나는 이상 현상(anomaly)이 생깁니다.

예를 들어 학생마다 학과명을 통째로 적어두면, 학과 이름이 바뀔 때 모든 행을 고쳐야 합니다. 학과를 별도 표로 빼고 FK로 연결하면, 한 군데만 고치면 됩니다.

단계핵심 규칙(쉽게)
1정규형(1NF)한 칸에는 값 하나만 (목록 금지)
2정규형(2NF)기본 키 전체에 의존하지 않는 열 분리
3정규형(3NF)키가 아닌 열에 딸린 열(이행 의존) 분리
normalize.sql
-- ❌ 정규화 전: 학과명이 학생마다 중복됨
-- student(id, name, dept_name, dept_office)

-- ✅ 정규화 후: 학과 정보를 별도 표로 분리
CREATE TABLE department (
    id     INT PRIMARY KEY,
    name   VARCHAR(30),
    office VARCHAR(30)
);

CREATE TABLE student (
    id      INT PRIMARY KEY,
    name    VARCHAR(20),
    dept_id INT,
    FOREIGN KEY (dept_id) REFERENCES department(id)  -- 중복 제거
);

인덱스 (Index)

인덱스는 책 뒤의 "찾아보기"와 같습니다. 특정 값을 찾을 때 표 전체를 처음부터 훑으면(풀 스캔, O(n)) 느립니다. 인덱스를 만들어두면 정렬된 구조를 따라 곧장 위치를 찾아 O(log n)으로 빨라집니다.

대부분의 DBMS는 인덱스를 B-tree(균형 잡힌 다진 트리)로 구현합니다. 위에서 아래로 내려가며 범위를 좁히기 때문에, 데이터가 수백만 건이어도 몇 단계 만에 도달합니다.

B-tree: 위에서 범위를 좁혀 내려가며 O(log n) 에 탐색
index.sql
-- name 으로 자주 검색한다면 인덱스 생성
CREATE INDEX idx_student_name ON student(name);

-- 이제 이 조회는 풀 스캔(O(n)) 대신 인덱스 탐색(O(log n))
SELECT * FROM student WHERE name = '김철수';
⚠️ 인덱스의 대가

인덱스는 조회를 빠르게 하지만 저장 공간을 더 쓰고, 삽입·수정·삭제를 느리게 합니다(인덱스도 함께 갱신해야 하므로). 자주 검색하는 열에만 선택적으로 거세요.

트랜잭션과 ACID

트랜잭션(transaction)은 "하나로 묶여 전부 성공하거나 전부 실패해야 하는 작업 묶음"입니다. 계좌 이체를 떠올리세요: A에서 출금하고 B에 입금하는 두 단계는 따로 떨어질 수 없습니다. 중간에 멈추면 돈이 사라지니까요.

DBMS는 이를 ACID라는 네 가지 약속으로 보장합니다.

  • A — 원자성(Atomicity): 전부 되거나, 전부 안 되거나(중간은 없음).
  • C — 일관성(Consistency): 끝난 뒤에도 규칙(잔액≥0 등)이 깨지지 않음.
  • I — 격리성(Isolation): 동시에 실행돼도 서로 간섭하지 않음.
  • D — 지속성(Durability): 완료된 결과는 정전이 나도 사라지지 않음.
출금과 입금은 하나의 트랜잭션 — 둘 다 성공해야 COMMIT
transaction.sql
BEGIN;                                          -- 트랜잭션 시작

UPDATE account SET balance = balance - 10000     -- A 계좌 출금
    WHERE id = 'A';
UPDATE account SET balance = balance + 10000     -- B 계좌 입금
    WHERE id = 'B';

COMMIT;     -- 둘 다 성공 → 확정 저장
-- 중간에 오류가 나면 ROLLBACK; 으로 전부 되돌림 (원자성)

고급 SQL: GROUP BY · HAVING · 서브쿼리

기초 SELECT를 넘어, 데이터를 묶어서 요약하고(집계) 질문 안에 질문을 넣는(서브쿼리) 방법을 배웁니다. 실무 분석 쿼리의 대부분은 이 도구들로 만들어집니다.

GROUP BY는 같은 값을 가진 행들을 한 묶음으로 모은 뒤, 각 묶음에 COUNT·SUM·AVG 같은 집계 함수를 적용합니다. 그리고 WHERE가 "행 하나하나"를 거르는 것과 달리, HAVING은 묶고 난 뒤의 그룹을 거릅니다.

행을 그룹으로 묶고(GROUP BY) → 집계한 뒤 → 그룹을 거른다(HAVING)
aggregate.sql
-- 학과별 평균 점수가 80점을 넘는 학과만
SELECT major, AVG(score) AS avg_score, COUNT(*) AS cnt
FROM exam
GROUP BY major                 -- 같은 학과끼리 묶기
HAVING AVG(score) > 80         -- 묶은 뒤의 그룹을 거름
ORDER BY avg_score DESC;

-- 실행 순서(중요): FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

서브쿼리(subquery)는 쿼리 안에 들어간 또 다른 쿼리입니다. "평균보다 높은 점수를 받은 학생"처럼, 먼저 한 값을 구한 뒤 그 결과로 다시 거를 때 씁니다.

subquery.sql
-- 전체 평균보다 높은 점수를 받은 학생
SELECT name, score
FROM exam
WHERE score > (SELECT AVG(score) FROM exam);   -- 괄호 안이 먼저 실행

-- 한 번이라도 수강 이력이 있는 학생 (EXISTS)
SELECT name FROM student s
WHERE EXISTS (SELECT 1 FROM enrollment e WHERE e.student_id = s.id);

-- IN 으로 목록을 만들어 거르기
SELECT name FROM student
WHERE id IN (SELECT student_id FROM enrollment WHERE course = 'DB');
💡 WHERE vs HAVING

한 줄 요약: WHERE는 묶기 전 개별 행을, HAVING은 묶은 뒤 그룹을 거릅니다. 그래서 집계 함수(AVG, COUNT) 조건은 반드시 HAVING에 와야 합니다.

뷰와 저장 프로시저

복잡한 쿼리를 매번 다시 쓰는 대신, 이름을 붙여 재사용할 수 있습니다. 두 가지 방법이 있습니다.

  • 뷰(View) — 자주 쓰는 SELECT에 이름을 붙인 가상 테이블. 실제 데이터를 저장하진 않고, 조회할 때마다 정의된 쿼리가 실행됩니다.
  • 저장 프로시저(Stored Procedure) — 여러 SQL 문을 묶어 DB 안에 저장해 둔 함수. 매개변수를 받아 한 번의 호출로 실행합니다.
구분뷰저장 프로시저
역할가상 테이블(읽기 중심)로직 묶음(읽기·쓰기)
호출SELECT ... FROM 뷰CALL 프로시저(...)
매개변수없음받을 수 있음
장점복잡한 조인 단순화·권한 통제반복 로직·트랜잭션 캡슐화
view_proc.sql
-- 뷰: 학생 이름 + 수강 과목을 미리 조인해 이름 붙이기
CREATE VIEW student_courses AS
SELECT s.name, e.course
FROM student s JOIN enrollment e ON s.id = e.student_id;

SELECT * FROM student_courses WHERE name = '김철수';  -- 뷰를 표처럼 사용

-- 저장 프로시저: 계좌 이체 로직을 묶어 저장 (MySQL 문법)
DELIMITER //
CREATE PROCEDURE transfer(IN from_id CHAR(1), IN to_id CHAR(1), IN amt INT)
BEGIN
    UPDATE account SET balance = balance - amt WHERE id = from_id;
    UPDATE account SET balance = balance + amt WHERE id = to_id;
END //
DELIMITER ;

CALL transfer('A', 'B', 10000);   -- 한 번의 호출로 실행
💡 뷰의 또 다른 쓸모: 보안

민감한 열(예: 주민번호)을 뺀 뷰만 특정 사용자에게 공개하면, 원본 테이블을 직접 못 보게 하면서 필요한 정보만 노출할 수 있습니다.

동시성 제어: 락과 MVCC

여러 사용자가 동시에 같은 데이터를 읽고 쓰면 충돌이 납니다. 두 사람이 같은 좌석을 동시에 예매하면 안 되겠죠. DBMS는 이를 두 가지 큰 전략으로 막습니다.

  • 락(Lock, 잠금) — 한 트랜잭션이 데이터를 만지는 동안 자물쇠를 걸어 다른 트랜잭션을 기다리게 합니다. 공유 락(읽기)은 여러 개가 동시에 가능하지만, 배타 락(쓰기)은 혼자만 가질 수 있습니다.
  • MVCC(다중 버전 동시성 제어) — 데이터를 덮어쓰지 않고 버전(스냅샷)을 여러 개 유지합니다. 읽는 사람은 자기 시점의 버전을 보므로, 읽기와 쓰기가 서로를 막지 않습니다. PostgreSQL·MySQL(InnoDB)·Oracle이 사용합니다.
락은 읽기가 쓰기를 기다리지만, MVCC는 옛 버전을 읽어 기다리지 않는다
🚨 교착 상태(Deadlock)

T1이 A를 잠그고 B를 기다리는데, T2는 B를 잠그고 A를 기다리면 서로 영원히 대기합니다. DBMS는 이를 감지해 한쪽 트랜잭션을 강제로 ROLLBACK시켜 풉니다. 락 순서를 항상 같게 잡으면 예방할 수 있습니다.

트랜잭션 격리 수준

ACID의 'I'(격리성)는 완벽히 지키면 안전하지만 느립니다. 그래서 SQL 표준은 안전성과 성능을 맞바꾸는 4단계 격리 수준을 제공합니다. 단계가 낮을수록 빠르지만, 동시성 문제(이상 현상)가 더 많이 허용됩니다.

대표적인 동시성 이상 현상 세 가지입니다.

  • 더티 리드(Dirty Read) — 아직 COMMIT 안 된(취소될 수도 있는) 남의 변경을 읽음.
  • 반복 불가능 읽기(Non-repeatable Read) — 같은 행을 두 번 읽었는데 그 사이 남이 바꿔 값이 달라짐.
  • 유령 읽기(Phantom Read) — 같은 조건으로 두 번 조회했는데 그 사이 남이 행을 추가/삭제해 행 수가 달라짐.
격리 수준더티 리드반복 불가능유령 읽기
READ UNCOMMITTED가능 ⚠️가능 ⚠️가능 ⚠️
READ COMMITTED막음 ✅가능 ⚠️가능 ⚠️
REPEATABLE READ막음 ✅막음 ✅가능* ⚠️
SERIALIZABLE막음 ✅막음 ✅막음 ✅

* MySQL InnoDB는 REPEATABLE READ에서도 유령 읽기를 대부분 막습니다(구현 차이).

오른쪽으로 갈수록 안전하지만 동시 처리량은 줄어든다
isolation.sql
-- 이 세션의 격리 수준 설정
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

BEGIN;
SELECT balance FROM account WHERE id = 'A';   -- 시점 1
-- (다른 트랜잭션이 A의 balance 를 바꾸고 COMMIT)
SELECT balance FROM account WHERE id = 'A';   -- 시점 2: 값이 달라질 수 있음
COMMIT;
-- REPEATABLE READ 였다면 두 SELECT 결과가 같았을 것
💡 기본값은?

대부분의 시스템은 균형점인 READ COMMITTED(PostgreSQL, Oracle 기본) 또는 REPEATABLE READ(MySQL 기본)를 씁니다. 정확한 정산이 필요한 금융 로직에서만 SERIALIZABLE을 고려합니다.

NoSQL 개요: 문서·키값·그래프

관계형 DB는 강력하지만, 모든 상황에 최적은 아닙니다. 정해진 표 구조가 부담스럽거나, 어마어마한 양을 여러 서버로 나눠 다뤄야 할 때 NoSQL("Not Only SQL")이 대안이 됩니다. 정해진 스키마 없이 유연하게 저장하고, 여러 서버로 잘 퍼지는(수평 확장) 것이 강점입니다.

백업과 복구

데이터베이스의 가장 큰 임무는 데이터를 절대 잃지 않는 것입니다. 디스크가 고장 나거나 실수로 테이블을 지워도 되돌릴 수 있어야 합니다. 두 가지 축으로 대비합니다.

  • 백업(Backup) — 데이터를 주기적으로 복사해 둠. 통째로 복사하는 전체 백업과, 직전 이후 바뀐 것만 복사하는 증분 백업이 있습니다.
  • 로그 기반 복구(WAL) — 변경을 데이터에 적용하기 전에 먼저 로그에 기록합니다(Write-Ahead Logging). 사고가 나도 로그를 다시 재생(replay)해 마지막 시점까지 되살립니다.
전체 백업 + 그 뒤의 로그를 합치면 원하는 시점으로 복구(PITR)
지표뜻쉽게
RPO복구 시점 목표"얼마나 옛 데이터까지 잃어도 되나"
RTO복구 시간 목표"얼마나 빨리 되살려야 하나"
🚨 백업은 복구 테스트까지가 백업

복사본만 쌓아두고 실제로 되살려 본 적이 없다면, 정작 필요할 때 복구가 안 될 수 있습니다. 정기적으로 복구 훈련을 해야 진짜 백업입니다.

쿼리 실행 계획과 최적화

SQL은 "무엇을 원하는지"만 적고, 어떻게 찾을지는 DBMS의 옵티마이저(optimizer)가 정합니다. 옵티마이저가 세운 그 작전이 실행 계획(execution plan)입니다. 같은 결과라도 어떤 계획을 고르느냐에 따라 속도가 수백 배 차이 납니다.

EXPLAIN을 붙이면 DBMS가 어떤 계획을 쓸지 미리 보여줍니다. 여기서 가장 중요한 신호는 풀 테이블 스캔(Full Scan, 표 전체 훑기)이 일어나는지입니다.

인덱스가 없으면 풀 스캔, 있으면 인덱스 스캔으로 단번에 좁힌다
explain.sql
-- 실행 계획 미리보기
EXPLAIN SELECT * FROM student WHERE name = '김철수';
-- type: ALL  → 풀 스캔(느림). 인덱스가 없다는 신호
-- type: ref  → 인덱스 사용(빠름)

CREATE INDEX idx_student_name ON student(name);  -- 인덱스 추가 후 다시 EXPLAIN

-- ❌ 인덱스를 무력화하는 안티패턴
SELECT * FROM student WHERE YEAR(birth) = 2000;  -- 열을 함수로 감싸면 인덱스 못 씀
-- ✅ 범위로 바꿔 인덱스 살리기
SELECT * FROM student WHERE birth >= '2000-01-01' AND birth < '2001-01-01';
💡 최적화 사고의 순서

① EXPLAIN으로 병목 찾기 → ② 자주 거르는 열에 인덱스 → ③ 필요한 열만 SELECT(SELECT * 지양) → ④ 함수로 열을 감싸지 않기. 대부분의 느린 쿼리는 이 네 가지로 해결됩니다.

🎯 핵심 정리
  • 관계형 DB는 데이터를 테이블에 담고, 기본 키·외래 키로 표를 연결한다.
  • SELECT로 묻고 JOIN으로 흩어진 표를 키로 이어 붙인다.
  • 정규화는 중복을 없애 이상 현상을 막는 설계 기법이다.
  • 인덱스(B-tree)는 조회를 O(log n)으로 빠르게 하지만 쓰기는 느려진다.
  • 트랜잭션은 ACID로 "전부 성공 아니면 전부 취소"를 보장한다.
  • GROUP BY는 행을 묶어 집계하고, HAVING은 묶은 뒤의 그룹을 거른다.
  • 뷰는 쿼리에 이름을 붙인 가상 테이블, 저장 프로시저는 로직 묶음이다.
  • 동시성은 락(기다림) 또는 MVCC(버전 읽기)로 제어한다.
  • 격리 수준은 안전성과 성능을 맞바꾸는 4단계(UNCOMMITTED~SERIALIZABLE)다.
  • NoSQL은 문서·키값·컬럼·그래프로, 유연성과 수평 확장이 강점이다.
  • 백업은 복구 테스트까지 해야 진짜 백업이며, WAL로 시점 복구한다.
  • EXPLAIN으로 실행 계획을 보고 풀 스캔을 인덱스 스캔으로 바꾼다.
DBMS
데이터베이스를 만들고 관리하는 소프트웨어(MySQL, PostgreSQL 등).
기본 키 / 외래 키
각 행을 유일하게 구별하는 열(PK), 다른 표의 PK를 가리켜 연결하는 열(FK).
JOIN
여러 테이블을 키 조건으로 이어 붙여 하나의 결과로 합치는 연산.
인덱스(Index)
조회 속도를 높이려고 만든 정렬된 보조 구조. 보통 B-tree로 구현.
트랜잭션 / ACID
나눌 수 없는 작업 묶음과, 그 신뢰성을 보장하는 네 가지 성질(원자성·일관성·격리성·지속성).
GROUP BY / HAVING
행을 그룹으로 묶어 집계하는 절, 묶은 뒤의 그룹을 거르는 절.
서브쿼리(Subquery)
쿼리 안에 들어간 또 다른 쿼리. 먼저 한 값을 구해 그 결과로 다시 거를 때 쓴다.
뷰(View)
자주 쓰는 SELECT에 이름을 붙인 가상 테이블. 실제 데이터를 저장하진 않는다.
저장 프로시저
여러 SQL을 묶어 DB에 저장해 둔 함수. 매개변수를 받아 호출한다.
락 / MVCC
데이터에 자물쇠를 거는 방식 vs 버전(스냅샷)을 유지해 읽기와 쓰기가 안 막히게 하는 방식.
교착 상태(Deadlock)
두 트랜잭션이 서로의 자원을 기다려 영원히 멈춘 상태.
격리 수준
동시성 이상 현상을 얼마나 허용할지 정하는 4단계(READ UNCOMMITTED~SERIALIZABLE).
NoSQL
정해진 스키마 없이 유연하게 저장하고 수평 확장에 강한 비관계형 DB(문서·키값·컬럼·그래프).
WAL
변경을 데이터에 적용하기 전에 먼저 로그에 기록해 복구를 보장하는 기법(Write-Ahead Logging).
실행 계획 / EXPLAIN
옵티마이저가 세운 데이터 검색 작전과, 그것을 미리 보여주는 명령.
