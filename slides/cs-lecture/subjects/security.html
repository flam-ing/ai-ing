대칭 · 비대칭 암호화

암호화는 데이터를 열쇠(키)로 잠가, 키 없이는 읽을 수 없게 만드는 일입니다. 키를 쓰는 방식에 따라 두 종류가 있습니다.

  • 대칭 암호화 — 잠그는 키와 푸는 키가 같습니다. 빠르지만, 그 키를 상대에게 안전하게 전달하는 게 어렵습니다. (예: AES)
  • 비대칭(공개키) 암호화 — 키가 한 쌍입니다. 공개키로 잠그면 짝꿍인 개인키로만 풀 수 있습니다. 공개키는 누구에게나 줘도 안전합니다. (예: RSA)
대칭은 빠르고, 공개키는 키 전달이 안전 — 실제로는 둘을 섞어 쓴다

실무에서는 둘을 조합합니다. 공개키로 "대칭 키"를 안전하게 주고받은 뒤, 본 데이터는 빠른 대칭 암호로 주고받습니다. HTTPS가 바로 이 방식입니다.

symmetric_demo.py
# 검증된 라이브러리(cryptography)를 쓰는 대칭 암호화 예시
from cryptography.fernet import Fernet

key = Fernet.generate_key()      # 키는 안전한 곳에 보관
cipher = Fernet(key)

token = cipher.encrypt(b"secret message")   # 잠그기
plain = cipher.decrypt(token)               # 같은 키로 풀기
print(plain)                                # b'secret message'
💡 직접 만들지 말 것

암호 알고리즘은 절대 직접 구현하지 말고, 검증된 표준(AES, RSA)과 라이브러리를 쓰세요. 자체 암호는 거의 항상 취약합니다.

해시와 비밀번호 저장

해시 함수는 입력을 고정 길이의 값으로 바꾸며, 한 방향으로만 작동합니다(결과로 원본을 되돌릴 수 없음). 그래서 비밀번호를 저장할 때 원문 그대로 저장하지 않고 해시값을 저장합니다. 서버가 털려도 원래 비밀번호를 바로 알 수 없게 하기 위함입니다.

비밀번호 + 솔트 → 해시 → 저장. 로그인 시엔 같은 과정을 거쳐 값만 비교한다

솔트(salt)는 사용자마다 다른 무작위 값으로, 같은 비밀번호라도 해시값이 달라지게 합니다. 또 비밀번호는 일반 해시(SHA-256)가 아니라 일부러 느린 전용 알고리즘(bcrypt, Argon2)으로 처리해, 대량 추측 공격을 어렵게 만듭니다.

password_store.py
import bcrypt

# 저장: 솔트는 bcrypt가 자동 생성·포함
hashed = bcrypt.hashpw(b"my-password", bcrypt.gensalt())
# DB 에는 hashed 만 저장 (원문은 저장하지 않음)

# 검증: 입력값을 같은 방식으로 처리해 비교
ok = bcrypt.checkpw(b"my-password", hashed)
print(ok)   # True
⚠️ 하지 말아야 할 것

비밀번호를 평문이나 단순 해시로 저장하지 마세요. 반드시 솔트 + bcrypt/Argon2 같은 느린 해시를 쓰세요.

인증과 권한 (JWT 등)

두 개념을 구분하는 게 출발점입니다.

  • 인증(Authentication) — "너 누구야?"를 확인. (로그인)
  • 권한/인가(Authorization) — "너 이거 해도 돼?"를 확인. (접근 제어)

로그인 후 매 요청마다 비밀번호를 다시 보낼 수는 없으니, 서버는 토큰이나 세션으로 "이미 검증된 사용자"임을 기억합니다. 대표적인 게 JWT(JSON Web Token)입니다. 사용자 정보가 담긴 토큰에 서버가 서명을 해서, 위조를 막습니다.

방식저장 위치특징
세션서버(상태 보관)서버가 로그인 상태를 직접 관리
JWT클라이언트서명으로 검증, 서버 부담 적음
jwt_demo.py
import jwt   # PyJWT
SECRET = "서버만 아는 비밀키"   # 환경변수 등 안전한 곳에 보관

# 발급: 서명된 토큰 생성
token = jwt.encode({"user_id": 42, "role": "user"}, SECRET, algorithm="HS256")

# 검증: 서명이 맞아야 통과 (위조 시 예외)
data = jwt.decode(token, SECRET, algorithms=["HS256"])
print(data["user_id"])   # 42
💡 최소 권한 원칙

각 사용자·서비스에는 꼭 필요한 권한만 부여하세요. 관리자 권한을 기본값으로 주는 것이 사고의 흔한 원인입니다.

흔한 공격과 방어 (SQLi · XSS)

웹에서 가장 흔한 두 가지 취약점과, 그것을 막는 방법을 봅니다.

SQL 인젝션 (SQLi)

사용자 입력을 SQL 쿼리 문자열에 그대로 이어 붙이면, 입력이 쿼리의 일부로 해석되어 데이터가 노출·변조될 수 있습니다. 방어: 매개변수 바인딩(Prepared Statement)을 써서 입력을 "값"으로만 다루게 합니다.

safe_query.py
# ❌ 위험: 입력을 문자열로 직접 합침
# cursor.execute("SELECT * FROM users WHERE name = '" + name + "'")

# ✅ 안전: 매개변수 바인딩 — 입력은 항상 '값'으로만 처리됨
cursor.execute("SELECT * FROM users WHERE name = ?", (name,))

크로스 사이트 스크립팅 (XSS)

사용자가 입력한 글에 스크립트가 섞여 있는데 그대로 화면에 출력하면, 다른 사용자의 브라우저에서 그 스크립트가 실행될 수 있습니다. 방어: 출력 시 이스케이프(escape)로 코드를 평범한 텍스트로 바꾸고, 콘텐츠 보안 정책(CSP)을 함께 적용합니다.

escape_demo.py
import html

user_input = ''
safe = html.escape(user_input)
# <script>...</script>  → 코드가 아니라 글자로 표시됨
# 대부분의 템플릿 엔진은 기본으로 자동 이스케이프한다
⚠️ 핵심 원칙

"모든 외부 입력은 믿지 않는다." 입력은 검증하고, 출력은 맥락에 맞게 이스케이프하세요. 이 한 줄이 대부분의 인젝션 계열 공격을 막습니다.

HTTPS · TLS

HTTPS는 HTTP에 TLS라는 암호화 계층을 입힌 것입니다. 브라우저와 서버 사이의 통신을 암호화해, 중간에서 누가 엿보거나 위조하지 못하게 합니다. 통신을 시작할 때 TLS 핸드셰이크로 안전한 채널을 만듭니다.

공개키로 안전하게 세션키를 합의한 뒤, 본 통신은 빠른 대칭 암호로 진행

인증서(certificate)는 "이 서버가 진짜 그 도메인의 주인이 맞다"를 신뢰받는 기관(CA)이 보증해 주는 문서입니다. 그래서 가짜 서버가 중간에 끼어드는 것을 막을 수 있습니다.

📌 왜 항상 HTTPS인가

로그인 정보·결제 정보 같은 민감 데이터는 물론, 평범한 페이지도 변조·추적을 막기 위해 오늘날엔 모든 통신에 HTTPS를 쓰는 것이 기본입니다.

위협 모델링

보안을 코드에 바로 적용하기 전에, "무엇을, 누구로부터, 어떻게 지킬 것인가"를 먼저 정리하는 일이 위협 모델링(threat modeling)입니다. 집을 짓기 전에 "현관·창문·지하실 중 어디가 약한가"를 점검하는 것과 같습니다. 방어자가 공격자보다 먼저 약점을 찾아 메우기 위한 사고 도구입니다.

널리 쓰이는 분류법으로 STRIDE가 있습니다. 여섯 가지 위협 유형의 머리글자입니다.

유형위협방어(예)
Spoofing신원 위장강한 인증·MFA
Tampering데이터 변조무결성 검증·서명
Repudiation행위 부인로그·감사 기록
Information Disclosure정보 유출암호화·접근 제어
Denial of Service가용성 마비속도 제한·이중화
Elevation of Privilege권한 상승최소 권한·검증

실무에서는 시스템을 그림으로 그려 데이터가 흐르는 경계(신뢰 경계)를 표시하고, 각 경계에서 위 여섯 유형을 하나씩 점검합니다. "여기서 입력이 들어오는데 위조될 수 있나? 가로채일 수 있나?"를 묻는 식입니다.

💡 설계 단계에서 시작하라

위협 모델링은 코드를 다 짠 뒤가 아니라 설계 초기에 할수록 비용이 적게 듭니다. 완성된 집의 토대를 고치는 것보다 도면을 고치는 게 쉬운 것과 같습니다.

접근제어 모델 (RBAC · 최소 권한)

인가(authorization)를 체계적으로 관리하는 방법이 접근제어 모델입니다. 가장 널리 쓰이는 것이 RBAC(Role-Based Access Control, 역할 기반 접근제어)입니다. 사용자에게 권한을 하나하나 붙이는 대신, 역할(role)에 권한을 묶고 사용자에게 역할을 부여합니다.

사용자 → 역할 → 권한. 역할만 바꾸면 권한이 한 번에 정리된다

이 위에 반드시 함께 적용하는 원칙이 최소 권한(Principle of Least Privilege)입니다. 각 사용자·프로그램에는 업무에 꼭 필요한 만큼만 권한을 줍니다. 기본값은 "아무 권한 없음"이고, 필요한 것만 명시적으로 허용합니다(기본 거부, default deny).

rbac_check.py
ROLE_PERMISSIONS = {
    "admin":  {"read", "write", "delete"},
    "editor": {"read", "write"},
    "viewer": {"read"},
}

def can(user, action):
    perms = ROLE_PERMISSIONS.get(user.role, set())   # 모르면 빈 집합
    return action in perms                            # 기본은 거부

# 서버는 민감 작업 전에 항상 권한을 확인한다
if not can(current_user, "delete"):
    raise PermissionError("권한 없음")   # 안전: 명시적 차단
⚠️ 화면만 숨기는 건 보안이 아니다

버튼을 안 보이게 하는 것은 편의일 뿐입니다. 실제 권한 확인은 반드시 서버에서 해야 합니다. 클라이언트의 검증은 우회될 수 있습니다.

디지털 서명과 PKI

암호화가 "내용을 숨기는" 것이라면, 디지털 서명(digital signature)은 "누가 보냈는지(인증)와 내용이 안 바뀌었는지(무결성)를 증명"하는 것입니다. 서명은 공개키 암호를 거꾸로 씁니다. 보내는 사람이 개인키로 서명하면, 누구나 그의 공개키로 검증할 수 있습니다.

개인키로 서명 → 공개키로 검증. 해시가 일치하면 위조·변조가 없었다는 증거

그런데 "이 공개키가 정말 그 사람 것인가?"는 어떻게 믿을까요? 이를 보증하는 체계가 PKI(공개키 기반구조)입니다. 신뢰받는 인증기관(CA)이 "이 공개키는 이 주체의 것"이라고 서명한 문서가 인증서(certificate)이며, 브라우저의 HTTPS 자물쇠가 바로 이 구조 위에서 동작합니다.

verify_signature.py
# 검증된 라이브러리로 '서명 검증'만 수행 (방어 측 사용)
from cryptography.hazmat.primitives.asymmetric import padding
from cryptography.hazmat.primitives import hashes
from cryptography.exceptions import InvalidSignature

def verify(public_key, message, signature):
    try:
        public_key.verify(signature, message,
                          padding.PKCS1v15(), hashes.SHA256())
        return True                # 서명 일치 → 진본
    except InvalidSignature:
        return False               # 위조·변조 의심 → 거부
💡 암호화와 서명은 짝꿍이지만 다르다

암호화는 기밀성(못 읽게), 서명은 무결성+인증(못 바꾸게·진짜인지)을 담당합니다. 소프트웨어 업데이트가 진짜인지 확인할 때도 서명을 씁니다.

OWASP Top 10은 "무엇부터 막을지" 정하는 데 좋은 출발점입니다. 새 서비스를 만들 때 이 목록을 체크리스트로 한 번 훑어보면 큰 구멍을 미리 막을 수 있습니다.

CSRF와 방어

CSRF(Cross-Site Request Forgery, 사이트 간 요청 위조)는 로그인된 사용자가 모르는 사이에, 악의적인 다른 사이트가 그 사용자 명의로 요청을 보내게 만드는 공격입니다. 브라우저가 쿠키를 자동으로 함께 보내는 성질을 노립니다. 방어 관점에서 막는 방법을 봅니다.

  • CSRF 토큰 — 서버가 폼마다 예측 불가능한 임의 토큰을 심고, 요청이 올 때 그 토큰이 함께 왔는지 확인합니다. 외부 사이트는 이 토큰을 알 수 없어 위조가 막힙니다.
  • SameSite 쿠키 — 쿠키에 SameSite=Lax 또는 Strict를 설정하면, 다른 사이트에서 시작된 요청에는 쿠키가 따라가지 않습니다.
  • 중요 작업은 재인증 — 비밀번호 변경·결제 같은 민감 작업에는 비밀번호 재확인을 둡니다.
csrf_defense.py
import secrets

# 폼을 줄 때: 안전한 임의 토큰을 만들어 세션에 보관
def issue_token(session):
    token = secrets.token_urlsafe(32)
    session["csrf_token"] = token
    return token        # 폼의 hidden 필드로 함께 전달

# 요청을 받을 때: 토큰이 일치해야만 처리 (불일치 시 거부)
def check_token(session, form_token):
    expected = session.get("csrf_token")
    return expected is not None and secrets.compare_digest(expected, form_token)

# 쿠키 설정 예: SameSite + Secure + HttpOnly
# Set-Cookie: session=...; SameSite=Lax; Secure; HttpOnly
💡 대부분 프레임워크가 내장

Django·Spring·Rails 등 주요 프레임워크는 CSRF 토큰 기능을 기본 제공합니다. 직접 구현하기보다 프레임워크의 기능을 켜서 쓰는 것이 안전합니다.

보안 헤더와 CSP

서버가 응답에 붙이는 보안 헤더(security headers)는 브라우저에게 "이렇게 안전하게 동작해 줘"라고 지시하는 한 줄짜리 방어막입니다. 코드 수정 없이도 큰 보호 효과를 줍니다.

헤더역할(방어)
Content-Security-Policy허용한 출처의 스크립트만 실행 → XSS 완화
Strict-Transport-Security항상 HTTPS로만 접속하도록 강제
X-Content-Type-OptionsMIME 추측 차단(nosniff)
X-Frame-Optionsiframe 삽입 차단 → 클릭재킹 방어
Referrer-Policy참조 주소 노출 최소화

가장 강력한 것은 CSP(Content Security Policy)입니다. "이 사이트의 스크립트·이미지·스타일은 어떤 출처에서만 불러올 수 있다"를 선언해, 공격자가 끼워 넣은 스크립트가 실행되는 것을 브라우저가 막게 합니다. XSS의 강력한 2차 방어선입니다.

headers.py
# 응답에 보안 헤더를 일괄 부여 (방어 설정)
SECURITY_HEADERS = {
    "Content-Security-Policy": "default-src 'self'",   # 자기 출처만 허용
    "Strict-Transport-Security": "max-age=31536000",   # 항상 HTTPS
    "X-Content-Type-Options": "nosniff",
    "X-Frame-Options": "DENY",
    "Referrer-Policy": "no-referrer",
}

def add_security_headers(response):
    for name, value in SECURITY_HEADERS.items():
        response.headers[name] = value
    return response
⚠️ CSP는 점진적으로

엄격한 CSP는 정상 스크립트까지 막을 수 있습니다. 처음엔 Content-Security-Policy-Report-Only로 위반만 보고받아 점검한 뒤, 안전을 확인하고 강제 모드로 전환하세요.

방화벽과 네트워크 보안

방화벽(firewall)은 네트워크의 문지기입니다. 들어오고 나가는 트래픽을 정해진 규칙에 따라 허용하거나 차단합니다. 핵심 원칙은 접근제어와 같습니다. 기본은 모두 차단(default deny)하고, 꼭 필요한 포트·주소만 명시적으로 허용합니다.

허용한 포트(예: 443)만 통과, 나머지는 막는다 — 기본 거부 원칙

방화벽 외에도 네트워크 보안에는 여러 겹의 방어가 있습니다. 이를 심층 방어(defense in depth)라 합니다. 한 겹이 뚫려도 다음 겹이 막도록 여러 층으로 보호합니다.

  • 네트워크 분리 — 외부에 노출되는 서버와 내부 DB를 다른 영역(예: DMZ)으로 나눕니다.
  • VPN — 원격 접속을 암호화된 터널로만 허용합니다.
  • IDS/IPS — 침입 시도를 탐지(IDS)하거나 자동 차단(IPS)합니다.
  • 속도 제한(rate limiting) — 과도한 요청을 제한해 무차별 시도와 DoS를 완화합니다.
💡 한 겹에 의존하지 말 것

"방화벽이 있으니 안심"은 위험합니다. 방화벽·암호화·접근제어·로그를 여러 겹으로 쌓아야, 한 곳이 실패해도 전체가 무너지지 않습니다.

사회공학

아무리 기술적 방어가 튼튼해도, 사람이 속으면 무너집니다. 사회공학(social engineering)은 기술이 아니라 사람의 심리를 노려 비밀번호·정보를 빼내거나 위험한 행동을 유도하는 수법입니다. 방어자 입장에서 "이런 수법이 있으니 조심하고 절차로 막자"는 관점에서 봅니다.

  • 피싱(phishing) — 진짜처럼 꾸민 메일·문자로 가짜 로그인 페이지에 유도. 방어: 발신자·링크 주소 확인, 의심되면 직접 공식 사이트로 접속.
  • 빙자(pretexting) — "IT 부서입니다"처럼 신분을 사칭해 정보를 요구. 방어: 정해진 절차로만 본인 확인, 전화로 비밀번호 요구에 응하지 않기.
  • 미끼(baiting) — 호기심을 자극하는 USB·링크. 방어: 출처 불명 매체·링크 금지.
⚠️ 사람도 방어선이다

가장 효과적인 방어는 교육과 절차입니다. "비밀번호는 누구에게도 알려주지 않는다", "민감 요청은 두 번째 경로로 확인한다(전화 콜백)", "다단계 인증(MFA)을 켠다" 같은 규칙이 사회공학을 크게 줄입니다.

💡 의심을 일상으로

"급하게, 비밀스럽게, 평소와 다른 경로로" 무언가를 요구하면 일단 멈추고 확인하세요. 긴박함을 조성하는 것이 사회공학의 흔한 신호입니다.

🎯 핵심 정리
  • 보안의 목표는 기밀성·무결성·가용성(CIA) 세 가지를 지키는 것이다.
  • 대칭 암호는 빠르고, 공개키 암호는 키 전달이 안전 — 실무는 둘을 조합한다.
  • 비밀번호는 평문이 아니라 솔트 + bcrypt/Argon2 해시로 저장한다.
  • 인증은 "누구냐", 인가는 "해도 되냐" — 최소 권한 원칙을 지킨다.
  • 모든 외부 입력은 검증하고 출력은 이스케이프 — SQLi·XSS의 기본 방어다.
  • 위협 모델링으로 설계 단계에서 약점을 먼저 찾는다(STRIDE).
  • RBAC로 역할에 권한을 묶고, 기본 거부 + 최소 권한을 적용한다.
  • 디지털 서명은 무결성과 인증을, PKI·인증서는 공개키의 신뢰를 보장한다.
  • OWASP Top 10은 흔한 취약점의 방어 체크리스트로 활용한다.
  • CSRF는 토큰과 SameSite 쿠키로, XSS는 CSP 등 보안 헤더로 한 겹 더 막는다.
  • 방화벽은 기본 거부 원칙으로 운영하고, 심층 방어로 여러 겹을 쌓는다.
  • 기술 방어와 함께 사람을 노리는 사회공학을 교육·절차·MFA로 막는다.
CIA 3요소
기밀성·무결성·가용성. 정보보안이 지켜야 할 세 가지 목표.
공개키 / 개인키
한 쌍의 키. 공개키로 잠그고 짝인 개인키로만 푼다.
솔트(Salt)
사용자마다 다른 무작위 값. 같은 비밀번호도 다른 해시가 되게 한다.
JWT
서명된 토큰으로 사용자 인증 상태를 전달하는 방식.
TLS
통신을 암호화해 엿보기·위조를 막는 계층. HTTPS의 'S'.
위협 모델링
무엇을·누구로부터·어떻게 지킬지 설계 단계에서 점검하는 사고 도구(STRIDE 등).
RBAC
역할에 권한을 묶고 사용자에게 역할을 주는 접근제어 모델.
최소 권한
꼭 필요한 만큼만 권한을 주는 원칙. 기본은 거부(default deny).
디지털 서명
개인키로 서명하고 공개키로 검증해 무결성과 발신자를 증명.
PKI / 인증서
CA가 공개키 소유자를 보증하는 체계와 그 보증 문서.
OWASP Top 10
가장 흔한 웹 취약점 10가지를 정리한 방어용 표준 목록.
CSRF
로그인된 사용자 명의로 위조 요청을 보내는 공격. 토큰·SameSite로 방어.
CSP
허용한 출처의 리소스만 실행하게 해 XSS를 완화하는 보안 헤더.
방화벽
규칙에 따라 네트워크 트래픽을 허용·차단하는 문지기. 기본 거부 운영.
심층 방어
한 겹이 뚫려도 다음 겹이 막도록 여러 층으로 보호하는 전략.
사회공학
기술이 아니라 사람의 심리를 노려 정보를 빼내는 수법. 교육·절차·MFA로 방어.
