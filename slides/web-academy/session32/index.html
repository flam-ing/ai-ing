try-catch 메커니즘 —
더 깊게 이해하기

try-catch가 에러를 처리하는 방식은 단순히 "에러를 잡는다"가 아니다.

1
코드 실행 (try 블록)
정상 흐름의 코드가 실행된다. 에러가 없으면 catch 블록은 건너뛴다.
2
에러 발생
에러 객체가 생성되고 실행이 즉시 중단된다. 에러 타입(TypeError, NetworkError 등)이 담긴다.
3
catch 블록 진입
에러 객체를 받아 처리한다. 로그를 남기고, 사용자에게 친절한 메시지를 보여줄 수 있다.
4
복구 또는 기본값 반환
서비스가 멈추지 않도록 안전한 기본값을 반환하거나 대안 동작을 수행한다.

🛡️

try/catch — 에러 안전망

예상 가능한 오류를 잡아 서비스가 멈추지 않게 보호한다.

async function getUserData(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    const data = await response.json();
    logger.info(`사용자 조회 성공: ${userId}`);
    return data;
  } catch (error) {
    // 오류를 잡아 서비스 중단 방지
    logger.error(`사용자 조회 실패: ${error.message}`);
    return null; // 안전한 기본값 반환
  }
}

catch 블록에서 로그를 남기고, 사용자에게는 친절한 에러 메시지를 보여준다.

실생활에서
이렇게 쓰여요

우리가 매일 사용하는 앱과 서비스 속에 로깅과 에러 처리가 녹아 있다.

🛒
쿠팡 결제 오류 메시지
"잠시 후 다시 시도해주세요" — try/catch로 결제 에러를 잡아 서비스를 멈추지 않고 친절하게 안내한다.
💛
카카오 장애 공지
ERROR 레벨 로그가 급증하면 팀에 알림이 가고, 장애를 인지한 즉시 공지를 낼 수 있다.
🔭
Sentry 에러 알림
배포 직후 에러 급증을 Sentry가 감지해 슬랙으로 알림을 보내 빠른 롤백을 가능하게 한다.

💡 이 개념이 없다면 서비스가 에러로 멈춰도 개발자가 한참 뒤에야 인지하게 된다.

자주 하는
오해 바로잡기

이렇게 생각하기 쉽지만, 실제로는 조금 다릅니다.

❌ 흔한 오해
  • 에러는 모두 개발자의 버그다
  • 로그는 개발자만 보는 것이다
  • 에러를 숨기면 사용자가 편하다
✓ 실제로는
  • 네트워크 오류, 외부 API 실패 등 예상 가능한 에러도 존재한다
  • 운영팀·고객센터도 로그를 분석해 장애 원인을 파악한다
  • 에러를 숨기면 나중에 디버깅이 훨씬 어려워진다
